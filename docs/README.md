# POMS 文档总览

`docs` 目录用于沉淀 `POMS` 的业务依据、架构决策、系统设计和模块详细设计。

当前 `POMS` 已有两份制度类业务源文档：

- `销售规范流程制度（试行）.md`
- `销售提成方案制度（试行）.md`

这两份文档应继续作为业务规则的源头依据，不直接替代系统设计文档。

## 建议目录结构

```text
docs/
  README.md                         # 文档总览与阅读顺序
  销售规范流程制度（试行）.md         # 业务制度源文档
  销售提成方案制度（试行）.md         # 业务制度源文档
  adr/                              # 架构决策记录
  design/                           # 跨模块系统设计
    poms-requirements-spec.md       # 制度到系统的需求与规则映射
    poms-hld.md                     # 系统高层设计
    poms-design-progress.md         # 设计进度跟踪与治理看板
  modules/                          # 模块详细设计
  reference/                        # 参考资料、术语、字典、模板
```

## 各目录职责

- `docs/` 根目录：
  - 存放业务制度源文档和总览索引
  - 不承载研发细节设计

- `docs/adr/`：
  - 存放高影响、难回退的架构决策
  - 例如权限模型、组织单元模型、导航可信源、状态机策略

- `docs/design/`：
  - 存放跨模块设计文档
  - 用于承接“制度规则”到“系统设计”的翻译层
  - 重点包括系统需求说明、HLD、跨域流程设计

- `docs/modules/`：
  - 存放具体模块的详细设计
  - 例如用户管理、角色与权限、组织单元、导航菜单、销售线索、提成结算

- `docs/reference/`：
  - 存放术语表、状态字典、角色字典、模板、设计约定等辅助材料

## 推荐阅读顺序

1. 业务制度源文档
2. `docs/design/poms-requirements-spec.md`
3. `docs/adr/`
4. `docs/design/poms-hld.md`
5. `docs/design/poms-design-progress.md`
6. `docs/modules/`

## 推荐编写顺序

1. 先维护业务制度源文档
2. 再补系统需求说明，把制度翻译成系统规则
3. 对关键争议点补 ADR
4. 再输出 HLD，稳定系统蓝图
5. 最后拆到各模块详细设计
