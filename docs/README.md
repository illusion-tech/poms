# POMS 文档总览

`docs` 目录用于沉淀 `POMS` 的业务依据、架构决策、系统设计、评审治理材料和模块详细设计。

当前 `POMS` 已有两份制度类业务源文档：

- `销售规范流程制度（试行）.md`
- `销售提成方案制度（试行）.md`

这两份文档应继续作为业务规则的源头依据，不直接替代系统设计文档。

## 建议目录结构

```text
docs/
  README.md                         # 文档总览与阅读顺序
  销售规范流程制度（试行）.md         # 业务制度源文档
  销售提成方案制度（试行）.md         # 业务制度源文档
  adr/                              # 架构决策记录
  design/                           # 系统设计与设计治理入口
    poms-requirements-spec.md       # 制度到系统的需求与规则映射
    poms-hld.md                     # 系统高层设计
    poms-design-progress.md         # 设计进度跟踪与治理看板
  modules/                          # 模块详细设计
  reference/                        # 参考资料、术语、字典、模板
```

## 文档治理原则

`POMS` 当前采用两套不同的文档治理方式：

- `docs/adr/` 采用编号治理，用于承载高影响、难回退、需要保留决策演进历史的架构决策
- `docs/design/` 采用分类治理，用于承载持续演进的系统设计、业务域设计、治理设计和评审过程文档

因此：

- `ADR` 继续使用 `ADR-001` 这一类顺序编号
- 设计文档默认不采用全局顺序编号，而是按主题命名、按分类维护
- 若某一类设计文档后续数量显著增加，再考虑在该分类内部增加子目录或局部编号

## 各目录职责

- `docs/` 根目录：
  - 存放业务制度源文档和总览索引
  - 不承载研发细节设计

- `docs/adr/`：
  - 存放高影响、难回退的架构决策
  - 例如权限模型、组织单元模型、导航可信源、状态机策略

- `docs/design/`：
  - 存放系统设计与设计治理文档
  - 用于承接“制度规则”到“系统设计”的翻译层
  - 当前主要包含四类资产：基线设计、业务域设计、治理与横切设计、评审与治理文档
  - 详细分类与命名规则以 `docs/design/README.md` 为准

- `docs/modules/`：
  - 预留给后续更细粒度的模块实现设计或独立模块设计沉淀
  - 当某一业务域或平台治理子设计继续细化到实现层时，可逐步从 `docs/design/` 下沉到此目录

- `docs/reference/`：
  - 存放术语表、状态字典、角色字典、模板、设计约定等辅助材料

## 推荐阅读顺序

1. 业务制度源文档
2. `docs/design/poms-requirements-spec.md`
3. `docs/adr/`
4. `docs/design/poms-hld.md`
5. `docs/design/poms-design-progress.md`
6. `docs/design/README.md` 与对应业务域或治理子目录入口
7. `docs/modules/`

## 推荐编写顺序

1. 先维护业务制度源文档
2. 再补系统需求说明，把制度翻译成系统规则
3. 对关键争议点补 ADR
4. 再输出 HLD，稳定系统蓝图和领域边界
5. 再补业务域设计、治理设计和评审治理文档
6. 最后视复杂度把实现层细化内容拆到 `docs/modules/`

## 当前推荐入口

- `docs/design/README.md`：设计资产分类、命名规则、状态规则和阅读路径
- `docs/design/poms-design-progress.md`：当前设计成熟度、文档状态和下一步治理动作
- `docs/adr/README.md`：已接受 ADR 索引和 ADR 使用约定
