## 1. 文档目标

本文档用于沉淀 `POMS` 第一阶段跨文档反复出现的核心术语，降低需求说明、HLD、业务域设计、治理设计和评审清单之间的命名漂移。

本文档只收录已经在设计文档中基本收敛的术语，不替代各领域详细设计中的完整对象定义。

---

## 2. 命名使用原则

- 优先使用已经在设计文档中固定下来的正式对象名，不使用临时描述性别名
- 同一个术语在不同文档中应尽量保持同一英文标识和同一中文语义
- 若术语含义发生重大变化，应优先通过 ADR 或上游设计文档收敛，再回写本表

## 3. 核心业务术语

| 术语                         | 中文口径       | 当前统一含义                                                                       | 备注                                   |
| ---------------------------- | -------------- | ---------------------------------------------------------------------------------- | -------------------------------------- |
| `Project`                    | 项目           | 第一阶段销售流程域主对象，在线索转入正式推进后创建，签约并完成移交后进入正式执行态 | 不再把“项目机会”作为独立正式主对象命名 |
| `Lead`                       | 线索           | 销售前置线索对象，用于承接最初商机信息，并作为创建 `Project` 的前置依据            | 未形成有效线索不得创建 `Project`       |
| `BidProcess`                 | 招投标子流程   | 当 `Project.commercialMode = bidding` 时启用的第一类受控子流程对象                 | 不单独占据 `Project.stage`             |
| `ProjectAssessment`          | 立项评估单     | 用于完成立项判断与放行的受控对象                                                   | 结论影响 `Project` 是否继续推进        |
| `QuotationReview`            | 报价与毛利评审 | 商务收口阶段中的关键受控动作                                                       | 不是独立主阶段                         |
| `ExecutiveEscalationRequest` | 高层介入申请   | 毛利红线以下、重大例外或战略项目场景下的高层放行对象                               | 不替代正常对象权限                     |
| `ProjectHandover`            | 项目移交记录   | 用于完成签约后责任交接、资料冻结和执行前切换的对象                                 | 未完成移交不得进入正式执行态           |
| `Contract`                   | 合同           | 合同资金域中的合同台账主对象                                                       | 合同资金域不接管销售主状态机           |
| `AcceptanceRecord`           | 验收记录       | 用于承接阶段验收或最终验收结论的对象                                               | 无有效验收记录不得进入完成态           |

## 4. 平台治理术语

| 术语                       | 中文口径     | 当前统一含义                              | 备注                               |
| -------------------------- | ------------ | ----------------------------------------- | ---------------------------------- |
| `User`                     | 用户         | 平台访问主体                              | 账户启停由 `isActive` 表达         |
| `Role`                     | 角色         | 权限集合的承载体                          | 不是组织范围或数据范围的承载体     |
| `Permission`               | 权限项       | 稳定权限字典项                            | 权限 key 一经发布不得原地改名      |
| `OrgUnit`                  | 组织单元     | 平台治理域正式组织管理实体                | 第一阶段采用树结构                 |
| `NavigationItem`           | 导航项       | 后端导航单一事实源中的节点对象            | 前端忠实渲染，不自行发明导航事实   |
| `RolePermissionAssignment` | 角色权限关系 | 表达角色与权限绑定关系的正式关系实体      | 推荐状态为 `active` / `revoked`    |
| `UserRoleAssignment`       | 用户角色关系 | 表达用户获得平台角色的正式关系实体        | 正式事实源，不建议退化成数组字段   |
| `UserOrgMembership`        | 用户组织关系 | 表达用户与主责/附属组织关系的正式关系实体 | 每个用户必须且仅有一个有效主责组织 |

## 5. 提成与结算术语

| 术语                       | 中文口径     | 当前统一含义                                       | 备注                                     |
| -------------------------- | ------------ | -------------------------------------------------- | ---------------------------------------- |
| `CommissionRoleAssignment` | 提成角色分配 | 用于表达项目中参与提成分配的角色版本关系           | 冻结后不得直接改写原版本                 |
| `CommissionCalculation`    | 提成计算结果 | 用于表达提成池计算、复核、生效和重算替代链路的对象 | 已生效结果不得直接覆盖                   |
| `CommissionPayout`         | 提成发放记录 | 第一阶段业务发放记录对象                           | 不与真实财务付款动作强联动               |
| `CommissionAdjustment`     | 提成调整单   | 用于承接异常调整、补差、扣回等动作的对象           | 已执行调整不得直接删除                   |
| `CommissionRuleVersion`    | 提成规则版本 | 提成规则的正式版本对象                             | 已生效规则不得静默改写，应通过新版本替代 |

## 7. 易混概念澄清

### 7.1 `Project` 与 `BidProcess`

- `Project` 是销售流程域主对象
- `BidProcess` 是仅在招投标成交路径下出现的第一类受控子流程
- 不应把投标过程状态直接折叠进 `Project.stage`

### 7.2 平台权限与业务对象动作授权

- `platform:*` 与 `nav:*` 主要回答“能否进入平台、看到菜单、访问平台接口”
- 业务对象动作授权回答“能否操作某个业务对象上的某个动作”
- 两者共享权限事实源，但语义层不可混用

### 7.3 组织轻量视图与正式组织实体

- `OrgUnit` 是正式组织管理实体
- 当前用户资料中的 `orgUnits[]` 是用户上下文下的轻量组织视图输出
- 轻量输出不应反向定义正式持久化模型
