## 1. 文档目标

本文档用于统一 `POMS` 第一阶段权限 key 的命名方式、前缀分层和使用边界，避免平台访问控制、导航可见性和业务对象动作授权在不同文档中继续混用。

本文档当前不是完整权限清单，而是第一阶段权限目录和命名规则基线。

---

## 2. 权限分层原则

第一阶段权限继续采用前缀分层：

- `platform:*`：平台管理能力与平台接口访问边界
- `nav:*`：导航可见性控制
- `project:*`：销售流程域相关业务动作权限
- `contract:*`：合同资金域相关业务动作权限
- `commission:*`：提成治理域相关业务动作权限

核心原则：

- 平台权限不替代业务对象动作授权
- 导航可见性权限不替代接口鉴权
- 权限前缀用于表达能力边界，不用于表达组织范围或数据范围

---

## 3. 权限前缀说明

| 前缀           | 用途                   | 当前语义边界                                                 |
| -------------- | ---------------------- | ------------------------------------------------------------ |
| `platform:*`   | 平台管理与平台接口能力 | 回答“能否进入平台管理能力、访问平台接口”                     |
| `nav:*`        | 导航可见性             | 回答“能否看到某个导航项或页面入口”                           |
| `project:*`    | 销售流程域业务动作     | 回答“能否对 `Project`、`BidProcess` 等对象执行业务动作”      |
| `contract:*`   | 合同资金域业务动作     | 回答“能否对合同、回款、付款、发票等对象执行业务动作”         |
| `commission:*` | 提成治理域业务动作     | 回答“能否对提成角色、计算、发放、规则版本等对象执行业务动作” |

---

## 4. 命名约束

### 4.1 推荐模式

第一阶段建议采用“域前缀 + 对象或能力 + 动作”的稳定命名方式，例如：

- `platform:users:view`
- `platform:roles:manage`
- `nav:platform:view`
- `project:assessment:submit`
- `contract:invoice:confirm`
- `commission:payout:approve`

说明：

- 上述示例用于说明命名模式，不代表当前仓库已经完整定义了这些 key
- 若某权限已正式发布，其 key 不应原地改名

### 4.2 命名规则

- 使用英文小写与冒号分层
- 语义必须稳定，避免与短期页面文案绑定
- 不允许通过创建“超级通配权限”绕开显式字典治理
- 导航可见性权限与业务动作权限不得混名
- 权限 key 应优先表达稳定业务动作，而不是某个临时按钮文案

---

## 5. 当前已明确的使用边界

### 5.1 `platform:*`

- 用于平台管理能力和平台接口访问边界
- 不直接回答某个业务对象是否允许被编辑、审批、关闭或冲销

### 5.2 `nav:*`

- 只影响导航可见性和页面入口体验
- 不得以 `nav:*` 权限替代业务接口授权
- `group` 类型导航父节点第一阶段默认不额外声明独立 `nav:*` 可见权限，其可见性由可见子节点派生
- 当前已明确的例子包括 `nav:platform:view`

### 5.3 业务域动作权限

- `project:*`、`contract:*`、`commission:*` 用于表达对象动作授权
- 这类权限应与业务对象动作矩阵、审批流和字段敏感度规则配合使用
- 仅有业务权限前缀并不自动回答组织范围和字段级写权限问题

---

## 6. 与导航和鉴权的关系

### 6.1 导航 `requiredPermissions`

- 导航项的 `requiredPermissions` 当前按 AND 语义解释
- 若 `requiredPermissions` 不满足，则导航项不返回
- 导航返回与接口鉴权共享同一权限事实源，但语义层必须区分

### 6.2 服务端事实源

- 用户最终有效权限应由后端基于当前有效关系计算
- token 中的权限上下文仅用于轻量传递与体验优化
- 用户停用、角色停用、角色剥权后，下一次请求必须体现最新授权结果

---

## 7. 生命周期与关系规则

### 7.1 `Permission` 生命周期

| 状态         | 含义                                                       |
| ------------ | ---------------------------------------------------------- |
| `active`     | 可被角色正常分配                                           |
| `inactive`   | 不再允许新增分配，但第一阶段默认保留存量有效关系的授权效力 |
| `deprecated` | 已废弃，不允许再被新角色或新变更使用                       |

### 7.2 关键关系实体

| 对象                       | 推荐状态             |
| -------------------------- | -------------------- |
| `RolePermissionAssignment` | `active` / `revoked` |
| `UserRoleAssignment`       | `active` / `revoked` |

关键规则：

- 权限迁移应通过“新增新 key + 废弃旧 key”完成
- 若目标是立即停止授权，应通过撤销关系或完成显式迁移，而不是仅依赖 `inactive`
- 同一 `roleId + permissionKey` 在有效状态下应唯一

---

## 8. 当前目录性质说明

- 本文档是权限目录和命名规则基线，不是完整权限字典落库结构
- 本文档也不是角色清单；角色建议在后续 `role-catalog.md` 中单独维护
- 若后续需要输出完整 `PermissionKey` 列表，应在本目录基础上新增更细粒度的权限字典文档，而不是在业务设计文档中各写一套

---

## 9. 维护约定

- 新增权限前缀前，应先确认其是否真的需要独立域级命名空间
- 新增权限 key 时，应优先检查是否能归入现有前缀体系
- 若某个权限 key 已进入接口、导航或角色配置使用，不应静默改名
