## 1. 文档目标

本文档用于统一 `POMS` 第一阶段在需求说明、授权矩阵、审批流设计和平台治理设计中反复出现的角色口径，减少“平台角色”“业务职责角色”“审批角色”之间的混用。

本文档当前不是最终系统角色配置表，而是跨文档角色语义目录。

---

## 2. 角色分层原则

第一阶段建议把角色分成三层理解：

- 平台角色：用于平台登录、菜单可见性、平台接口访问与平台治理能力配置
- 业务职责角色：用于表达某条业务链路中的责任分工或执行职责
- 审批/确认角色：用于表达某个动作的放行、确认、复核或例外裁量主体

说明：

- 同一个自然人可以在不同上下文下同时具备平台角色和业务职责
- 平台角色不直接替代业务对象动作授权矩阵中的业务责任表达
- 审批角色应优先表达“谁有放行权”，而不是直接等价于某个系统菜单角色

---

## 3. 平台角色口径

| 角色口径     | 当前统一含义                       | 备注                              |
| ------------ | ---------------------------------- | --------------------------------- |
| 平台用户     | 已登录并获得平台访问资格的用户     | 最小平台访问主体                  |
| 系统预置角色 | 由系统提供的基础角色集合           | 第一阶段允许存在系统预置角色      |
| 自定义角色   | 在现有权限字典基础上受控扩展的角色 | 只能从既有权限字典中选择权限      |
| 平台治理角色 | 能进入平台治理相关模块的角色       | 依赖 `platform:*` 与 `nav:*` 权限 |

说明：

- 平台角色的正式持有关系由 `UserRoleAssignment` 表达
- 平台角色是权限集合，不是组织范围或数据范围的直接承载体

---

## 4. 业务职责角色口径

| 角色口径        | 当前统一含义                                                             | 常见出现位置                       |
| --------------- | ------------------------------------------------------------------------ | ---------------------------------- |
| 销售人员        | 项目第一责任人，负责线索登记、立项推动、商务推进、签约与回款跟踪         | 需求说明、生命周期设计、授权矩阵   |
| 销售负责人      | 销售责任链条中的高一级业务负责人，负责投标决策、高层介入等关键发起或放行 | 生命周期设计、审批流、授权矩阵     |
| 技术支持 / 售前 | 负责技术可行性、范围确认和主要技术风险判断                               | 需求说明、生命周期设计、审批流     |
| 商务行政        | 负责合同资料、流程材料、移交组织和记录完整性                             | 需求说明、合同资金域设计、审批流   |
| 商务负责人      | 在商务结果确认、投标结果确认等环节承担业务确认职责                       | 审批流、授权矩阵                   |
| 财务人员        | 负责回款、付款、税务风险、提成计算与提成发放登记等财务归口动作           | 需求说明、合同资金域设计、提成设计 |
| 项目负责人      | 在项目移交、执行和验收等阶段承担项目交接或执行责任                       | 需求说明、移交相关设计             |
| 业务登记角色    | 负责把业务事实录入系统的角色口径                                         | 常用于台账登记、提成发放登记等场景 |
| 制度维护角色    | 负责维护提成规则版本等制度性配置的角色口径                               | 提成治理域设计                     |

说明：

- 业务职责角色更偏业务语义，不必与平台菜单角色一一对应
- 同一个业务职责角色在不同对象动作上可能仍需结合组织范围、字段敏感度和审批链路进一步约束

## 5. 审批与确认角色口径

| 角色口径         | 当前统一含义                                                                       | 常见出现位置                     |
| ---------------- | ---------------------------------------------------------------------------------- | -------------------------------- |
| 副总经理         | 第一阶段关键审批角色之一，负责立项、报价评审、流程合规等高影响放行                 | 需求说明、生命周期设计、审批流   |
| 公司高层         | 用于战略项目、重大例外、毛利红线以下等场景的高层裁量主体                           | 需求说明、生命周期设计、审批流   |
| 必要审批角色     | 某些动作中按制度或场景需要参与审批的补充角色                                       | 签约、投标结果确认等场景         |
| 预算审批角色     | 提成发放阶段中负责放行档位和审批结论的角色                                         | 提成治理域设计、审批流           |
| 项目移交确认角色 | 项目移交或提成角色冻结中参与确认的角色集合                                         | 项目移交、提成角色冻结           |
| 管理层           | 在提成治理、异常处理、关闭终止、规则版本生效等场景承担最终审批或裁量责任的角色集合 | 提成治理域设计、授权矩阵、审批流 |
| 相关责任人       | 多方确认场景中必须共同完成确认的责任集合                                           | 项目移交、交接类动作             |

说明：

- `管理层`、`必要审批角色`、`相关责任人` 目前属于聚合口径，不等于单一平台角色名
- 后续如果系统需要落到具体角色清单，应再细化为正式角色目录或审批规则配置

## 6. 易混角色澄清

### 6.1 销售人员 与 销售负责人

- 销售人员偏执行责任和一线推进
- 销售负责人偏关键决策发起、高影响审批链条参与和例外场景负责

### 6.2 商务行政 与 商务负责人

- 商务行政偏资料完整性、台账、流程组织和归档责任
- 商务负责人偏商务结果确认或商务判断责任

### 6.3 财务人员 与 预算审批角色

- 财务人员偏事实登记、校验、确认、计算与执行动作
- 预算审批角色偏提成发放档位和预算层面的审批裁量

### 6.4 平台角色 与 业务职责角色

- 平台角色负责系统访问与能力边界
- 业务职责角色负责具体业务对象动作中的责任表达
- 二者可以映射，但不应在文档里直接当成同一个概念使用

---

## 7. 使用约定

- 在需求说明、设计文档和评审材料中，优先使用本目录已收敛的角色名称
- 若需要引入新的聚合角色口径，应先说明它是平台角色、业务职责角色还是审批角色
- 若后续需要把角色落成正式配置字典，可在本目录基础上再补“系统角色清单”类文档
