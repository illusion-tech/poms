# 模块详细设计

本目录用于沉淀比 `docs/design/` 更细一层的模块实现设计，重点回答“某个模块准备如何落地实现”。

相对于 `docs/design/`，本目录更关注：

- 某个模块的页面、交互、接口、字段、状态与异常处理如何展开
- 某个模块在已接受设计基线之下，具体需要实现哪些约束
- 某个模块实现时，哪些边界已经固定，哪些地方仍需回到上游设计文档确认

因此，本目录不是新的总纲设计目录，也不负责记录高层决策；它承接的是已经在 `docs/design/` 和 `docs/adr/` 中稳定下来的设计结论。

## 与其他目录的关系

- `docs/adr/`：记录高影响、难回退的设计决策
- `docs/design/`：记录系统级设计、业务域设计、治理设计和评审治理文档
- `docs/modules/`：记录更细粒度的模块实现设计
- `docs/reference/`：记录术语、字典、模板和辅助约定

如果某项内容仍处于“方案尚未定”的阶段，应优先回到 `docs/design/` 或 `docs/adr/`，而不是直接在这里冻结实现细节。

## 上游输入

本目录下的模块设计文档，通常应建立在以下文档之上：

- 业务制度源文档
- `docs/design/poms-requirements-spec.md`
- `docs/design/poms-hld.md`
- 相关业务域或治理域设计文档
- 相关已接受 ADR

必要时，还应回看：

- `docs/design/poms-design-progress.md`
- 对应域的评审清单与评审摘要

## 适合放在这里的内容

- 页面级或子模块级功能设计
- 交互范围、表单字段、列表字段、按钮守卫等细化设计
- 模块内状态流转、边界场景、异常处理说明
- 接口草案、事件草案、校验规则和数据约束
- 实现前需要冻结的字段级权限或行为级守卫说明

## 不适合直接放在这里的内容

- 高影响、难回退的架构决策
- 跨多个业务域的总体边界划分
- 仅用于评审过程管理的清单与摘要
- 术语表、状态字典、角色字典、模板等参考资料

这些内容分别更适合放在 `docs/adr/`、`docs/design/` 或 `docs/reference/`。

## 建议优先级

在当前基线下，第一批更适合下沉到模块层的内容包括：

- 用户管理
- 角色与权限
- 组织单元
- 导航菜单
- 销售线索登记
- 项目立项评估
- 技术可行性与范围确认
- 报价与毛利评审
- 合同签约与项目移交
- 提成计算与结算

其中，平台治理域相关内容应优先在 `docs/design/platform-governance/` 完成域级收口，再决定是否继续下沉到本目录。

## 命名建议

本目录建议继续使用“主题 + 文档类型”的稳定命名方式，不使用全局顺序编号。

推荐示例：

- `user-management-module-design.md`
- `project-initiation-module-design.md`
- `commission-settlement-api-draft.md`
- `contract-handover-state-rules.md`

如果后续某一业务域的模块文档数量显著增多，再考虑在该业务域下增加子目录，而不是提前引入全局流水号。

## 建议模板

每篇模块设计文档建议至少包含：

- 标题与页头元信息
- 背景与目标
- 上游依据
- 关联业务制度条款
- 关联设计文档与 ADR
- 用户角色与权限
- 页面与交互范围
- 数据模型
- 状态流转
- API 草案
- 校验规则与异常场景
- 测试关注点

推荐页头元信息格式如下：

```md
# 模块设计文档标题

**文档状态**: Draft
**最后更新**: 2026-03-16
**适用范围**: `POMS` 第一阶段某模块
**关联文档**:

- `上游业务域设计`
- `相关 ADR`
```

模块设计文档的状态建议优先使用：

- `Draft`
- `Draft (Baseline)`
- `Review`
- `Accepted`
- `Archived`

一般不建议把模块设计文档标记为 `Active`，除非它本身是持续维护的治理类文档。

## 维护约定

- 新增模块设计文档前，应先确认上游设计是否已经稳定
- 若模块设计过程中发现新的高影响未决策问题，应回到 `docs/design/` 或新增 ADR，而不是在本目录静默拍板
- 每次本目录新增重要模块设计文档时，应同步检查 `docs/README.md` 与 `docs/design/poms-design-progress.md` 是否需要更新
