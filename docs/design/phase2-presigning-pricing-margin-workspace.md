## 1. 文档目标

本文档用于细化签约前统一工作区中的 `报价与毛利评审`。

这个工作区的核心不是单独做一个“价格审批页”，而是把签约前已经形成的技术、成本、税务、回款条件和竞标结果收束成一次正式的经营判断：

- 能否报价
- 能否放行
- 为什么能放行或不能放行
- 放行后的结果如何进入签约就绪

---

## 2. 工作区定位

### 2.1 面向用户

- 销售
- 财务
- 副总经理 / 流程负责人
- 技术支持 / 售前（以依据提供者视角进入）

### 2.2 使用时机

- 项目已形成前期成本估算
- 项目已形成可用于商务判断的技术与风险结论
- 项目需要正式给出报价或接受商务条件
- 项目需要判断是否可进入签约

### 2.3 工作区目标

- 将报价动作从“填价格”提升为“经营判断”
- 将成本、税务成本、回款条件和毛利放到同一上下文
- 让审批人先看摘要，再看依据
- 让放行结论直接进入 `签约就绪`

---

## 3. 与投标的关系

### 3.1 不是所有项目都必须先经过投标

当前设计明确允许两种前置路径进入 `报价与毛利评审`：

1. 竞标路径
   - 项目经过 `招投标 / 商务竞标`，再进入报价与毛利评审
2. 直接商务路径
   - 项目不需要投标，直接从 `技术与成本` 进入报价与毛利评审

### 3.2 当前规则

- 若项目不需要投标，`招投标 / 商务竞标` 节点应标记为“不适用”或“已通过商务路径替代”
- 项目不应因缺少投标记录而被强行阻断
- 但即使不投标，也仍需经过正式的报价与毛利评审，之后才能进入签约就绪

---

## 4. 工作区顶层结构

建议 `报价与毛利评审` 工作区采用“上摘要、中判断、下依据”的结构。

### 4.1 主区块建议

1. 当前报价摘要
2. 成本与税务成本摘要
3. 回款条件与税务条件
4. 毛利判断与例外提示
5. 审批摘要与放行结论
6. 进入签约就绪的承接提示

### 4.2 顶部固定摘要条

建议顶部固定显示：

- 当前报价版本
- 当前毛利判断状态
- 当前审批状态
- 当前是否存在阻断项
- 当前是否已具备进入签约就绪的条件

---

## 5. 关键区块详细口径

### 5.1 当前报价摘要

**作用**:

让用户先知道当前报价方案是什么，而不是先陷入明细项。

**建议包含**:

- 报价总额
- 当前报价版本
- 当前适用路径：
  - 竞标路径
  - 直接商务路径
- 当前是否为待审批版本

### 5.2 成本与税务成本摘要

**作用**:

把 `技术与成本` 工作区输出的经营基础显式带入，不让报价变成脱离成本的动作。

**建议包含**:

- 当前估算成本总额
- 当前税务成本 / 税负影响摘要
- 当前高不确定成本项
- 当前主要风险项

**关键规则**:

- 这里应引用 `技术与成本` 的正式输出，而不是再次手工录入
- 税务成本必须显式展示，不能只在审批备注中隐含出现

### 5.3 回款条件与税务条件

**作用**:

把影响毛利和放行判断的商务条件拉到同一屏判断。

**建议包含**:

- 首付款条件
- 阶段回款条件
- 尾款 / 质保金条件
- 税务条件摘要

**关键规则**:

- 回款条件与税务条件都应能解释其对毛利的影响
- 不能只显示条件文本，而不显示其经营含义

### 5.4 毛利判断与例外提示

**作用**:

帮助审批人快速判断项目是否处于正常区间，还是已经进入例外情况。

**建议包含**:

- 当前毛利判断结果
- 当前是否触发例外提示
- 当前主要例外原因

**例外原因示例**:

- 成本偏高
- 税务成本影响较大
- 首付款偏低
- 回款周期偏长
- 高风险不确定项过多

### 5.5 审批摘要与放行结论

**作用**:

把审批从“点通过/驳回”提升为结构化结论。

**建议结论类型**:

- 放行
- 条件放行
- 否决
- 需升级审批

**建议包含**:

- 当前审批摘要
- 当前结论
- 若为条件放行，列整改项
- 若为否决，列结构化原因

**审批摘要最小引用建议**:

- `approvalScenarioKey`
- `summaryPackageKey`
- `summarySnapshotId`
- `projectionLevel`
- `exportPolicy`

**关键规则**:

- 每次 `放行`、`条件放行` 或 `需升级审批`，都应形成一份可追溯的“商业放行基线”
- 商业放行基线至少应冻结：报价总额、税率与税务条件、首付款比例、阶段回款节点、尾款 / 质保金条件、当前毛利判断结论、来源成本估算版本
- 若为 `条件放行`，允许调整项与禁止偏离项必须结构化列出，不能只写在审批备注里
- 若为 `否决`，只保留结构化否决原因，不生成可进入合同主链的商业放行基线
- 审批人先看摘要、再按授权查看依据；审批身份本身不自动等于完整经营敏感字段可见权，审批摘要与附件展示仍需遵循 `phase2-data-permission-and-sensitive-visibility-design.md`
- 报价评审页、审批通知、打印材料与导出预览必须引用同一份 `summarySnapshotId`，不得由页面重新裁剪详情字段代替审批摘要包
- `projectionLevel` 与 `exportPolicy` 必须作为本工作区的正式输入显式返回，避免副总经理、财务与流程负责人在不同入口看到不同摘要口径
- 技术支持 / 售前作为依据提供者参与时，默认只查看与自身任务相关的范围、风险与成本依据摘要，不默认开放 `internalCostRate`、人员级人工成本明细或无关提成敏感信息

**差异分级矩阵建议**:

| 差异项                | `提示`                                               | `需复核`                                              | `必须重审`                                                              |
| --------------------- | ---------------------------------------------------- | ----------------------------------------------------- | ----------------------------------------------------------------------- |
| 合同金额 / 不含税报价 | 仅因四舍五入或格式化产生，且不改变金额口径           | 相对商业放行基线偏离 `<= 3%`，且未跨越毛利红线带      | 偏离 `> 3%`，或导致毛利率跨越 `< 10%`、`10% ~ 30%`、`>= 30%` 任一审批带 |
| 税率 / 税务条件       | 文案规范化，不改变税率、计税方式、税负承担方         | 无                                                    | 任何税率、计税方式、税负承担方变化都必须重审                            |
| 首付款比例            | 无                                                   | 未跨越 `20%` 首付款分界线，且绝对偏差 `<= 5` 个百分点 | 跨越 `20%` 分界线，或绝对偏差 `> 5` 个百分点                            |
| 阶段回款节点          | 仅文案规范化，不改变节点顺序、节点金额占比与节点日期 | 节点顺序不变，且单节点日期偏移 `<= 10` 个自然日       | 新增 / 删除 / 重排节点，或任一节点日期偏移 `> 10` 个自然日              |
| 尾款 / 质保金条件     | 无                                                   | 仅质保期表述微调，且不改变质保金比例与回款前提        | 质保金比例、尾款前提、质保期或结算条件发生变化                          |
| 范围承诺              | 文案澄清，不改变交付边界、验收口径与服务范围         | 无                                                    | 任何会改变交付边界、验收口径、服务范围或责任分工的调整都必须重审        |

说明：

- 本矩阵中的 `20%` 首付款分界线直接承接《销售提成方案制度（试行）》中标准首付款 / 低首付款的制度分层
- 毛利红线带直接承接《销售规范流程制度（试行）》中 `< 10%`、`10% ~ 30%`、`>= 30%` 的审批分层
- 任一字段一旦命中 `必须重审`，本次合同草案不得继续沿用原商业放行基线

**允许调整项建议**:

- 合同编号、签署日期、附件清单、归档说明等行政信息
- 不改变金额、税务条件、回款节奏、质保金比例和范围承诺的文案规范化调整
- 未改变节点顺序和节点金额占比的日期微调，且单节点日期偏移不超过 `10` 个自然日

**责任与入口建议**:

- 商务行政负责录入合同草案并触发差异校验
- 系统负责自动给出 `提示 / 需复核 / 必须重审` 分类
- 销售负责人对范围承诺和商务背景差异给出说明
- 财务对税率、税务条件、首付款比例和回款节点差异给出复核意见
- 副总经理 / 流程负责人对 `必须重审` 项决定是否退回本工作区重审或升级审批

### 5.6 进入签约就绪的承接提示

**作用**:

明确告诉用户当前是否已经完成签约前最后一道商务判断。

**建议包含**:

- 当前是否具备进入签约就绪的条件
- 若不具备，缺什么
- 若具备，下一步入口是什么

**关键规则**:

- 进入 `签约就绪` 时，不能只传递“已放行”状态，还必须同步带出最新通过的商业放行基线引用
- `签约就绪` 后续对合同草案的比较，必须以该商业放行基线作为必比依据

## 6. 角色视角差异

### 6.1 销售

默认重点：

- 当前报价版本
- 当前审批状态
- 当前阻断项
- 当前下一步动作

### 6.2 财务

默认重点：

- 成本与税务成本摘要
- 回款条件
- 毛利判断依据
- 当前是否需要财务确认

### 6.3 副总经理 / 流程负责人

默认重点：

- 审批摘要
- 毛利与例外提示
- 放行 / 条件放行 / 否决结论

### 6.4 技术支持 / 售前

默认重点：

- 被引用的技术与成本依据
- 因商务条件变化产生的风险提醒

## 7. 关键体验规则

### 7.1 审批人先看摘要，再看依据

评审页不应先展示大量明细字段，而应先给出经营判断摘要。

### 7.2 报价必须和成本绑定

任何报价结论都必须能回到成本和税务成本依据。

### 7.3 不投标项目也能直接进入本工作区

如果项目不走投标路径，本工作区必须支持直接商务路径进入，不得额外阻断。

### 7.4 放行结论必须驱动后续阶段

审批结论不是终点，必须决定是否能进入 `签约就绪`。

### 7.5 商业放行基线不得在合同页静默漂移

- 合同草案进入正式合同路径前，必须先与最新通过的商业放行基线做差异校验
- 必比字段至少包括：报价总额、税率与税务条件、首付款比例、阶段回款节点、尾款 / 质保金条件，以及会直接改变报价前提的范围承诺
- 若必比字段偏离商业放行基线，且该偏离不在“条件放行允许调整项”内，则必须回到本工作区重新审阅或升级审批
- 合同页可以补录合同编号、签署日期、附件等行政信息，但不得静默改写已放行的经营结论
- `需复核` 项可以进入签约就绪，但必须带着复核结论继续传递；`必须重审` 项不得进入签约就绪后的合同生效链

---

## 8. 与其他工作区的承接关系

`报价与毛利评审` 必须至少承接并输出以下关系：

1. 输入来自：
   - `技术与成本`
   - `招投标 / 商务竞标` 或直接商务路径
   - `项目总览`
2. 输出给：
   - `签约就绪`
   - `项目总览` 摘要区

**关键输出**:

- 当前报价结论
- 当前审批结论
- 当前是否具备进入签约的条件
- 当前商业放行基线引用
- 必比字段包与允许调整项
