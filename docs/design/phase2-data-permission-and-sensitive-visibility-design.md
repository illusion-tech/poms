## 1. 文档目标

本文档用于补齐第二阶段当前缺失的横切设计：业务数据权限与敏感信息可见性。

本文档重点回答以下问题：

- 业务用户是否可以看到其他人负责的项目
- 同一条业务记录中，哪些字段属于敏感字段
- 提成分配是否应比普通项目数据更严格
- 平台管理员是否天然拥有全部业务敏感数据可见权
- 列表、搜索、详情、导出、审批摘要、通知等读侧出口如何统一受控

本文档不是平台级 RBAC 设计的替代物，而是在其之上补齐业务数据范围与字段敏感度控制。

---

## 2. 当前缺口判断

第二阶段现有文档已经覆盖：

- 模块入口与用户主线
- 生命周期阶段承接
- 经营核算与提成操作链
- 平台级动作权限与路由/API 访问控制

但当前仍缺少一套完整回答以下问题的设计：

- 销售之间是否互看项目
- 项目参与人、主管、管理层分别能看多深
- 提成分配结构与金额是否允许同项目全员互看
- 合同金额、毛利率、税务成本、提成金额等敏感字段如何遮罩
- 离岗、转岗、项目移交后历史数据如何回看
- 搜索、导出、审批摘要、通知是否会泄露敏感信息

因此，这一专题属于进入实现排期前必须补齐的横切设计，而不是后续可选优化项。

## 3. 设计原则

### 3.1 四层权限分解

业务权限至少拆成四层，不应混成单一“有权限 / 无权限”：

1. 模块访问权限
   能否进入某个模块或工作区。

2. 动作权限
   能否新增、编辑、审批、冻结、发放、导出等。

3. 数据范围权限
   能看哪些记录。

4. 字段敏感权限
   同一条记录里能看哪些字段、哪些字段需遮罩或摘要化。

### 3.2 `Project` 作为数据范围主锚点

大多数业务对象都应从 `Project` 派生可见范围：

- `BidProcess`
- 报价与毛利评审
- 合同
- 实际成本
- 回款
- 经营核算
- 提成结算

也就是说，先判断用户与项目的关系，再决定其对派生对象的可见范围。

### 3.3 提成敏感度高于普通项目信息

提成分配、提成金额、异常扣回、离职影响等信息，默认应比普通项目状态更严格受控。

`项目可见` 不等于 `提成全明细可见`。

### 3.4 平台管理员不自动等于业务敏感信息全可见

平台管理员默认拥有平台治理能力，但不应天然等于：

- 可查看全部项目经营明细
- 可查看全部人员提成分配和金额

业务敏感数据仍应受业务范围权限和字段敏感权限约束。

### 3.5 所有读侧出口统一受控

数据权限不能只控制详情页，必须统一覆盖：

- 列表
- 搜索
- 详情
- 看板 / 总览
- 审批摘要
- 导出
- 打印
- 通知
- 审计查询中的业务上下文展示

## 4. 角色与默认范围建议

### 4.1 销售人员

默认建议：

- 可见本人主责项目
- 可见本人参与项目
- 不默认可见其他销售主责项目详情
- 不默认可见其他销售的提成分配明细

如果业务确需团队内协同查看，应进一步区分：

- 是否只显示存在与当前阶段
- 是否显示合同金额、成本、毛利
- 是否显示提成结构与金额

### 4.2 销售主管 / 部门负责人

默认建议：

- 可见本团队项目
- 可见本团队项目的经营摘要
- 是否可见全员提成金额，应单独授权

### 4.3 技术支持 / 售前

默认建议：

- 可见本人参与或被指派支持的项目范围、风险、里程碑、交付前置项与验收范围摘要
- 可见为完成方案、报价评审和移交确认所必需的任务上下文
- 不默认可见与自身无关项目的经营与提成敏感明细
- 不默认可见项目毛利精确值、税务成本、人工成本率与全员提成金额

### 4.3.1 交付 / 实施负责人

默认建议：

- 可见本人负责交付或已进入移交准备项目的范围、排期、交接清单、验收范围与当前阻断原因
- 可见执行启动所需的合同义务摘要、回款节点摘要与客户交付约束
- 不默认可见 `internalCostRate`、个人或角色级人工成本明细、项目毛利精确值与全员提成金额
- 如需判断资源压力，只允许查看项目级投入摘要和风险信号，不显示人工成本率与人员级成本拆解

### 4.4 商务行政

默认建议：

- 可见与合同、移交、资料完整性相关的项目记录
- 对经营和提成金额类字段仅按工作需要开放

### 4.5 财务

默认建议：

- 可见合同、回款、成本、经营核算、提成发放全链数据
- 但仍应保留字段敏感控制和操作审计

### 4.6 管理层

默认建议：

- 可见较大范围项目
- 可见经营摘要与必要敏感信息
- 是否查看全员提成金额，仍应按职责配置

### 4.7 平台管理员

默认建议：

- 可治理用户、角色、组织、导航和审计
- 不自动拥有全部业务敏感数据详情可见权

## 5. 关键敏感字段

以下字段默认应纳入敏感字段控制范围：

- 合同金额
- 回款金额与回款条件
- 预计成本
- 实际成本明细
- 毛利额与毛利率
- 税务成本 / 税负影响
- 提成角色权重
- 提成应发 / 已发金额
- 提成异常调整原因
- 例外审批理由
- 风险备注与保留意见

### 5.1 敏感字段包与默认角色边界

第一批先把跨角色高频读取场景收敛为以下字段包，不再允许各页面各自定义可见边界：

| 字段包                     | 典型字段                                                                      | 默认完整可见角色                                         | 默认摘要可见角色                                         | 默认不可见角色                                                                | 说明                                                   |
| -------------------------- | ----------------------------------------------------------------------------- | -------------------------------------------------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------------ |
| 项目标识与阶段包           | 项目编号、客户、当前阶段、下一步动作、阻断原因                                | 销售主责、项目参与人、商务行政、财务、管理层、平台管理员 | 无                                                       | 无                                                                            | 作为协作主锚点，不承载经营敏感明细                     |
| 合同与回款经营包           | 当前有效合同集金额、首付款比例、回款进度、应收节点状态、质保金状态            | 销售主责 / 销售负责人、商务行政、财务、管理层            | 售前、交付 / 实施负责人                                  | 无业务关系人员、未获业务授权的平台管理员                                      | 协作角色只看履约与推进摘要，不看完整经营明细           |
| 毛利与税务影响包           | 预计毛利额、毛利率、已确认毛利、税务成本、税负影响                            | 财务、受授权管理层、销售主责 / 销售负责人                | 商务行政、审批链角色                                     | 售前、交付 / 实施负责人、无业务关系人员、未获业务授权的平台管理员             | 审批链角色默认只拿摘要判断，不默认拿精确值             |
| 实际成本聚合包             | 项目累计成本、成本类型汇总、偏差方向、成本完整性状态                          | 财务、受授权管理层、销售主责 / 销售负责人                | 商务行政、交付 / 实施负责人                              | 售前、无业务关系人员、未获业务授权的平台管理员                                | 默认只开放项目级聚合，不开放明细展开                   |
| 人力成本率与人工成本明细包 | `internalCostRate`、`rateVersionId`、工时 × 成本率、人员 / 角色级人工成本记录 | 财务、受授权管理层                                       | 销售主责 / 销售负责人仅可见项目级 `LABOR` 汇总金额       | 售前、交付 / 实施负责人、商务行政、项目参与协作角色、未获业务授权的平台管理员 | 这是首批必须硬隔离的字段包                             |
| 提成分配与金额包           | 角色权重、应发金额、已发金额、扣回 / 异常调整金额                             | 财务、受授权管理层                                       | 项目负责人 / 销售主管仅可见结构摘要或本人 / 团队必要摘要 | 售前、交付 / 实施负责人、商务行政、无业务关系人员、未获业务授权的平台管理员   | `项目可见` 不等于 `提成金额可见`                       |
| 例外审批与保留意见包       | 例外理由、风险备注、保留意见、反对意见摘要                                    | 审批链相关角色、财务、管理层                             | 销售主责、商务行政仅可见结论摘要                         | 无业务关系协作角色                                                            | 用于解释放行与阻断，不等于对所有敏感基础数据开全量读取 |

同一条记录中，允许出现：

- 记录可见，但敏感字段遮罩
- 记录可见，但只显示摘要值
- 记录完全不可见

### 5.2 遮罩等级基线

第一批统一采用四档遮罩等级，避免不同页面各自发明表达：

1. 完整值
   展示精确金额、比例、版本号、责任人和时间戳，仅对默认完整可见角色开放。

2. 摘要值
   仅展示区间、档位、风险信号、是否越阈、是否已满足条件等最小判断信息。例如“毛利率处于审批通过区间”“首付款比例已跌破 20% 阈值”“已纳入 `LABOR` 成本但不展示成本率”。

3. 遮罩占位
   展示字段存在但不展示具体值，例如“经营敏感字段已隐藏”“人工成本率仅财务可见”，用于避免协作用户误判为数据缺失。

4. 完全不可见
   字段不在详情、搜索、导出、打印、审批摘要和通知中出现，也不通过标题、附件名或统计聚合侧漏存在性。

### 5.3 非财务 / 非销售负责人的硬隔离规则

第一批先冻结以下硬规则：

1. 售前、交付 / 实施、商务协作角色默认不得查看 `internalCostRate`、`rateVersionId`、个人 / 角色级人工成本明细、项目毛利精确值、税务成本精确值与全员提成金额。
2. 这些协作角色只可查看完成本职协作所必需的范围、里程碑、交接项、验收项、合同义务摘要、回款节点摘要与项目级风险信号。
3. 销售主责与销售负责人可以查看项目级经营结果和合同 / 回款摘要，但默认不查看个人 / 角色级人工成本率与财务归口明细来源。
4. 财务与受授权管理层可查看完整经营与人工成本敏感字段，但仍需保留字段访问审计、导出审计与审批留痕。
5. 平台管理员不因平台治理职责自动获得任何经营敏感字段明细可见权；若无业务授权，其可见范围停留在治理元数据和非敏感业务锚点。

## 6. 提成场景特别约束

提成相关视图默认建议采用更保守策略：

1. 销售个人
   仅直接看到自己的提成角色、权重、阶段、阻断原因与本人金额。

2. 项目负责人 / 销售主管
   可见本项目提成结构，但“是否显示全员金额”需单独授权。

3. 财务
   可见完整提成计算与发放链。

4. 管理层
   按管理职责可见完整或摘要化提成结果。

5. 无关人员
   不应默认看到其他人的提成分配明细或金额。

## 7. 历史与生命周期变化规则

数据范围设计还必须覆盖以下变化场景：

- 项目移交后，原销售还能看哪些历史信息
- 人员转岗后，还能否看到此前项目历史
- 售前退出项目后，是否仍保留历史查看权
- 离职后，提成历史如何被财务和管理层继续追溯
- 审批人是否能回看自己已审批事项的历史敏感数据

建议原则：

- 历史可追溯不等于历史全公开
- 生命周期关系变化后，应保留最小必要历史可追溯能力
- 历史敏感信息仍需按职责受控

## 8. 读侧统一约束建议

实现前应统一明确以下读侧约束：

1. 列表页先按数据范围过滤，再决定字段遮罩
2. 搜索结果不能泄露无权记录的存在细节
3. 审批摘要只展示审批所需最小信息
4. 导出能力必须单独受控，不自动继承页面查看权
5. 通知内容应避免携带超范围敏感字段
6. 审计日志中的业务对象标题与摘要，也应避免越权泄露敏感信息

### 8.1 审批最小可见基线

第一批先冻结审批页的最小可见原则；第三批进一步把 `R4-004` 的审批摘要字段包与 `R2-005` 的短时揭示机制收口为正式规则，但不反向改写本文件已冻结的主边界。

| 审批场景                    | 审批必须展示                                                                                 | 允许摘要化展示                                            | 默认禁止展示                                                               |
| --------------------------- | -------------------------------------------------------------------------------------------- | --------------------------------------------------------- | -------------------------------------------------------------------------- |
| 报价 / 毛利放行             | 项目标识、报价版本、报价总额、回款条件摘要、当前放行结论、主要阻断风险                       | 预计成本构成、毛利区间、税务条件摘要、是否跨审批阈值      | `internalCostRate`、人员级人工成本明细、全员提成金额、无关历史版本全文     |
| 合同差异复核 / 签约放行     | 商业放行基线版本、差异级别、差异字段列表、是否跨重审阈值、当前复核结论                       | 当前有效合同集摘要、首付款 / 质保金档位、回款节点变化摘要 | 人员级人工成本率、完整成本明细、其他项目提成信息、与本次判断无关的附件全文 |
| 移交 / 冻结确认             | 项目标识、移交清单完成状态、当前有效合同集摘要、冻结模式、角色分配结构摘要、阻断原因         | 项目经营状态信号、回款比例档位、验收前置项摘要            | 全员精确提成金额、人工成本率、人员级人工成本明细、与交接无关的经营细项     |
| 第二阶段发放 / 异常调整审批 | 阶段类型、`gate` 结论、验收 / 阶段成果确认摘要、证据链摘要、申请金额或上限摘要、当前阻断风险 | 毛利成熟度信号、回款比例档位、异常类型摘要                | 人员级人工成本率、无关项目发放历史、超出本次审批所需的全量提成明细         |

### 8.1.1 第三批审批摘要字段包规则

第三批把审批最小可见基线继续收口为场景化字段包规则：

1. 审批页、打印版、导出预览和审批通知都必须先绑定 `approvalScenarioKey`，再消费对应的 `summaryPackageKey`。
2. 同一审批场景下只允许返回该场景的最小字段集、投影级别和导出策略，不允许从详情页字段里临时挑选子集。
3. `projectionLevel` 至少区分完整值、摘要值和遮罩占位三档，不允许在审批链中默认回退到完整详情。
4. 审批摘要字段包生成后，应形成可追溯快照；审批意见、打印材料和导出审计都应引用同一份场景包结果。

### 8.1.2 第三批例外查看与短时揭示规则

第三批把例外查看继续收口为短时揭示规则：

1. 短时揭示必须以字段包为单位申请，不允许按页面或接口一次性放开完整详情。
2. 申请至少应说明 `fieldPackageKey`、用途、适用对象和到期时间；授权后也只在授权范围内临时放宽投影结果。
3. 到期、撤销或审批驳回后，查询层应立即回到原有字段包投影边界，不得把短时揭示结果沉淀为长期默认可见。
4. 所有揭示申请、授权、访问和失效都必须留痕；平台管理员也不因平台治理身份而自动获得短时揭示豁免。

### 8.2 导出、打印与通知边界

1. 导出与打印默认继承当前页面的字段可见边界，不允许通过导出模板绕过遮罩。
2. 审批材料打印版默认沿用审批最小可见基线，不因“便于线下判断”自动扩展到完整敏感字段。
3. 完整敏感字段导出必须单独授权，并记录导出人、导出时间、字段范围与业务理由。
4. 通知、待办标题、站内消息和审计摘要只允许带项目锚点、阶段、动作结果与阻断原因，不携带毛利精确值、人工成本率和全员提成金额。
5. 附件名称、导出文件名和审批摘要标题也应纳入脱敏范围，避免通过文件名侧漏敏感结论。

### 8.3 第一批守卫实现落点

第一批现在不应再停留在“页面自己判断是否显示字段”，而应把守卫正式拆到以下实现层：

1. `RecordScopeGuard`
   先判断当前用户能否看到这条记录；无权则列表、搜索、详情都不返回记录存在性。

2. `FieldPackageProjectionGuard`
   在记录可见前提下，再按字段包决定返回完整值、摘要值还是遮罩占位。

3. `ApprovalSummaryGuard`
   对审批页、审批摘要、打印摘要和审批通知统一裁剪到场景级最小字段集。

4. `SensitiveExportGuard`
   高敏字段导出、打印和文件生成不继承页面权限，必须单独检查导出授权与审计留痕。

5. `CommandPreconditionGuard`
   对会产生有效事实的命令，在写侧检查是否具备必要的敏感字段授权与前置对象引用，而不是让读侧绕过后再兜底。

#### 8.3.1 查询层守卫顺序

第一批统一固定以下顺序：

1. 先做数据范围过滤。
2. 再做字段包投影。
3. 再生成页面摘要、审批摘要、导出预览和通知摘要。

禁止做法：

- 先查全量详情，再在前端按角色裁字段
- 先返回完整经营数据，再在组件层决定是否隐藏
- 通过单独的“导出接口”绕开详情页守卫

#### 8.3.2 第一批主线的守卫落点

| 主线 / 场景                        | 记录范围守卫                       | 字段包投影守卫                                   | 审批 / 摘要守卫                                                          | 导出审计守卫                   | 说明                                      |
| ---------------------------------- | ---------------------------------- | ------------------------------------------------ | ------------------------------------------------------------------------ | ------------------------------ | ----------------------------------------- |
| `L1` 报价 / 毛利评审               | 按 `Project` 参与关系过滤          | 毛利与税务影响包按角色投影                       | 报价放行摘要只给区间、档位、风险信号                                     | 完整放行材料导出需单独授权     | 不允许通过评审页暴露人工成本率            |
| `L1` / `L3` 签约就绪与合同差异复核 | 按 `Project` / `Contract` 范围过滤 | 合同与回款经营包、差异字段包按场景投影           | 复核页只显示必比字段、差异等级和结论所需摘要                             | 合同全文与差异材料导出单独留痕 | 不因“合同复核”自动放开全部经营敏感值      |
| `L3` 移交 / 冻结确认               | 按移交参与关系与项目归属过滤       | 冻结模式、当前有效合同集与经营信号摘要投影       | 移交审批摘要不显示全员精确提成金额                                       | 冻结材料打印默认沿用审批摘要   | 只允许看完成判断所需最小字段              |
| `L4` 项目经营总览                  | 按项目范围过滤                     | 毛利、成本、税务、`LABOR` 汇总按角色投影         | 不适用                                                                   | 经营明细导出必须走独立导出申请 | `ProjectOperatingView` 不返回默认全量原值 |
| `L5` 阶段 gate / 第二阶段发放      | 按项目与提成职责范围过滤           | 提成分配与金额包默认按本人 / 团队 / 财务角色投影 | 第二阶段审批摘要只展示 `AcceptanceRecord` 摘要、证据链摘要和金额上限摘要 | 完整提成明细导出单独留痕       | `项目可见` 不等于 `提成全明细可见`        |

#### 8.3.3 第一批写侧命令的守卫要求

第一批以下命令在写侧必须额外做前置校验：

1. `reviewCommercialReleaseBaselineDiff`
   - 校验调用人是否具备差异复核权限
   - 校验差异结果是否仍处于可复核状态

2. `initializeContractTermSnapshotFromReadinessPackage`
   - 校验承接包是否可见且可用
   - 校验调用人是否具备初始化正式合同权限

3. `initializeReceivablePlanFromReadinessPackage`
   - 校验承接包与合同快照链是否完整
   - 校验是否存在阻断原因未收口

4. `submitCommissionPayoutApproval` at `stage=second`
   - 校验 `acceptanceRecordId` 非空且有效
   - 校验调用人无权通过自由文本绕过验收前置

5. `publishInternalCostRateVersion` / `registerLaborCostRecord`
   - 校验成本率字段包只允许财务或受授权管理角色写入
   - 校验 `LABOR` 记录必须绑定有效 `rateVersionId`

6. `requestSensitiveDataExport`
   - 校验字段包是否允许导出
   - 校验用途说明与审计落点是否齐备

#### 8.3.4 审计与运行时留痕要求

第一批至少要把以下守卫结果写入审计或安全事件：

1. 高敏导出申请通过 / 拒绝
2. 审批摘要因字段包限制被裁剪
3. 敏感字段读取因角色不足而返回遮罩或拒绝
4. 写侧命令因缺少关键前置对象或权限被拒绝

这些留痕可以复用：

- `sensitive_data_export_request`
- `sensitive_data_export_audit`
- `audit_log`
- `security_event`

第一批不要求把所有字段投影命中都做成完整行为分析，但至少要让“被拒绝的高敏读取 / 导出 / 命令”可以追溯。

## 9. 对第二阶段的直接影响

本设计对当前第二阶段有三个直接影响：

1. 它是 `LX-T03` 正式审阅的必审项
2. 它是 `LX-T04` 进入实现排期前的前置约束
3. 它会反向影响：
   - `L1` 签约前工作区的列表与详情展示
   - `L4` 经营核算视图的摘要与敏感字段展示
   - `L5` 提成阶段总览、发放与异常调整的可见范围

## 10. 当前建议结论

当前建议先采用偏保守的默认口径：

1. 销售默认不互看其他销售主责项目详情
2. 项目参与人按参与关系看项目
3. 主管按团队范围看项目
4. 提成分配默认比普通项目详情更严格
5. 平台管理员不自动拥有业务敏感数据详情可见权
6. 所有读侧出口统一受控
7. 非财务 / 非销售负责人默认不可见 `internalCostRate`、个人 / 角色级人工成本明细、项目毛利精确值与全员提成金额
8. 审批页默认只给最小判断字段，不因审批身份自动放开完整经营敏感明细
9. 例外查看、短时揭示和审批场景细粒度字段矩阵已作为第三批正式细化，但仍继续遵守本文件已冻结的主边界，不反向放宽默认权限

在这个基础上，再按真实业务需要局部放宽，而不是默认全开放后再回补限制。
