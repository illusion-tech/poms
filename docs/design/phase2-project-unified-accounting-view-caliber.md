## 1. 文档目标

本文档用于细化 `L4-T02`，即：

把项目经营核算视图的统一数字口径固定下来，避免后续在经营总览、偏差解释、提成判断和复盘页面里分别消费不同数字。

这里的“统一核算视图”是经营判断口径，不等于会计总账口径，也不直接等于提成最终结算口径。

---

## 2. 统一核算视图的定位

### 2.1 服务对象

- 财务
- 销售 / 项目负责人
- 管理层

### 2.2 服务场景

- 执行中查看项目当前经营状态
- 关闭前判断项目最终经营结果
- 为 `L4-T03` 偏差与风险解释提供同一组底层数字
- 为 `L5` 提成阶段判断提供统一经营输入

### 2.3 不负责的事情

- 不替代正式财务总账
- 不替代合同、回款、成本原始台账
- 不直接产出提成应发金额

### 2.4 统一口径不等于统一可见值

本文件固定的是合同、回款、成本、毛利等经营数字的唯一计算口径，不是所有角色都默认看到完整精确值的授权口径。

因此：

- 同一组统一核算结果，在不同角色页面上可分别落成完整值、摘要值或遮罩占位
- `internalCostRate`、人员 / 角色级 `LABOR` 明细、毛利精确值和其他高敏字段的展示，仍受 `phase2-data-permission-and-sensitive-visibility-design.md` 约束

---

## 3. 统一核算视图必须固定的核心指标

项目统一核算视图建议至少固定以下指标：

1. 当前有效合同集合不含税金额
2. 当前已确认回款不含税金额
3. 当前已纳入口径实际成本不含税金额
4. 原始签约基线成本
5. 当前有效变更后基线成本
6. 当前经营毛利
7. 当前经营毛利率
8. 相对当前有效基线偏差金额
9. 相对当前有效基线偏差比例
10. 当前经营状态

---

## 4. 数据来源优先级

### 4.1 合同金额

统一使用：

- 当前有效合同集合中全部有效合同版本的不含税金额汇总

补充要求：

- 必须支持按合同维度拆解来源
- 不得在项目级读侧压缩成单一“当前有效合同”

不直接使用：

- 草稿合同
- 已失效历史合同版本

### 4.2 回款金额

统一使用：

- 已确认生效的回款记录不含税金额累计

不直接使用：

- 计划回款
- 待确认回款
- 业务口头确认但未财务确认的到账信息

### 4.3 实际成本金额

统一使用：

- `includedCostTotal`

即：

- 已纳入口径的项目实际成本累计金额

不直接使用：

- 仅 `registered`
- 仅 `confirmed`
- 尚未纳入口径的成本事实

### 4.4 估算成本

统一使用：

- 原始签约基线成本 + 已批准生效的变更包基线累计结果

不直接使用：

- 任意历史估算版本
- 未正式采用的草稿估算
- 待审批或未生效的变更请求金额

进一步固定：

- `originalBaselineCost` 表达签约前冻结的原始基线，不得被执行期变更静默覆盖
- `currentEffectiveBaselineCost` 表达原始基线叠加已批准生效的变更包结果，作为执行中正式偏差口径
- 页面应同时保留两类解释：相对原始签约基线的偏移，以及相对当前有效基线的真实超支 / 节余

## 5. 税务口径

### 5.1 总原则

统一核算视图默认使用：

- 不含税收入
- 不含税成本

理由是：

- 经营毛利应尽量回到项目本身的经营结果
- 税务影响需要单独呈现，不应把经营结果和税务负担混在一条金额里

### 5.2 税务成本的表达方式

税务成本不应消失，而应作为以下两种信息同时存在：

1. 在估算侧：
   - 作为签约前成本清单中的独立估算项
2. 在执行与核算侧：
   - 作为影响经营判断的重要解释项和偏差项

### 5.3 当前核算视图对税务成本的要求

- 默认经营毛利不直接用含税金额相减
- 需要单独显示税务影响摘要
- 需要在偏差解释区明确税务影响是否为主要偏差来源之一

### 5.4 进项税与经营成本的纳入口径

第二批当前进一步固定：

1. `includedCostTotal` 默认继续采用不含税经营成本口径。
2. 可抵扣的进项税额不直接并入经营成本，不得与项目成本重复计算。
3. 不可抵扣进项税、明确属于项目经营负担的税务附加、以及签约前已单列为经营成本假设的税务成本，可以作为经营成本或税务影响调整项进入统一核算。
4. 销项税负、认证状态、税务补录和冲红等财务口径差异，应进入 `taxImpactSummary` 与偏差解释区，不得直接改写经营收入或经营成本基础公式。
5. 若税务影响尚未闭合，应单列 `taxImpactPendingAmount` 或同等级摘要字段，作为数据成熟度和风险解释的输入之一。

## 6. 核心公式建议

### 6.1 当前经营毛利

建议固定为：

`当前经营毛利 = 已确认回款不含税金额 - 已纳入口径实际成本不含税金额`

### 6.2 当前经营毛利率

建议固定为：

`当前经营毛利率 = 当前经营毛利 / 已确认回款不含税金额`

补充规则：

- 当已确认回款为 `0` 时，不显示误导性百分比
- 应显示为 `未形成有效毛利率`
- 这里使用的是项目级已确认回款汇总口径；`L5` 若采用合同分别判断模式，只影响 gate 解释和可发上限，不改变 `L4` 的底层金额

### 6.3 成本偏差金额

建议固定为：

`成本偏差金额 = 已纳入口径实际成本 - 当前有效变更后基线成本`

### 6.4 成本偏差比例

建议固定为：

`成本偏差比例 = 成本偏差金额 / 当前有效变更后基线成本`

补充规则：

- 当当前有效变更后基线缺失或为 `0` 时，不应强行显示百分比

### 6.5 相对原始签约基线偏移金额

建议同时固定：

`原始基线偏移金额 = 当前有效变更后基线成本 - 原始签约基线成本`

它不代表执行超支，而用于解释“项目经营假设在执行期间发生了多少正式批准的基线变化”。

### 6.6 经营公式边界与极端值规则

第二批当前统一以下边界：

1. 当已确认回款金额为 `0` 时，可显示经营毛利金额，但不得显示误导性的经营毛利率。
2. 当 `已确认回款 / 当前有效合同集合金额 < 20%` 时，经营毛利率只作为观察值，不得单独作为自动阻断或放行依据。
3. 当数据成熟度未达到“可用于经营判断”时，所有比例型指标都必须带“暂不稳定”标记，不得直接升级为最终经营结论。
4. 当经营毛利率小于 `-100%`、大于 `100%` 或出现明显异常跳变时，应标记为极端值，强制进入偏差解释或复核链，而不是按普通绿色 / 红色状态直接展示。

## 7. 统一状态分层

统一核算视图建议把项目经营状态分成两层：

### 7.1 数据成熟度层

用于表达当前数字是否足够稳定：

- `数据不足`
- `初步可看`
- `可用于经营判断`

建议派生依据：

- 是否已有有效合同
- 是否已有确认回款
- 是否已有纳入口径成本
- 未映射成本和待确认成本是否过多
- 分摊成本是否仍存在待冻结分摊依据
- 税务影响待确认金额是否过高
- 是否存在待确认的补录重述或历史重算候选

补充规则：

- 来自 `L2-T04 / L2-T05` 的税务影响摘要、`taxImpactPendingAmount`、分摊稳定性提示、未映射成本提示、成本数据成熟度状态与成本侧动作建议，必须作为统一核算视图的正式输入直接消费，不得在 `L4` 页面重新推断成本侧成熟度结论。
- 统一核算视图可以在项目级聚合这些输入，但不得覆盖 `L2` 已固定的成本侧 `PROMPT / REVIEW / BLOCK` 建议与引用基线 / 快照版本。

### 7.2 经营风险层

用于表达当前项目经营是否健康：

- `正常`
- `关注`
- `风险`

建议派生依据：

- 当前经营毛利率
- 成本偏差
- 回款进度
- 高风险成本项未收口数量

---

## 8. 分阶段展示口径

### 8.1 执行中项目

执行中项目的统一核算视图重点是：

- 当前有效合同集合金额
- 当前确认回款
- 当前纳入口径成本
- 当前经营毛利 / 毛利率
- 当前偏差与风险

### 8.2 已关闭项目

已关闭项目的统一核算视图重点是：

- 最终有效合同集合金额
- 最终确认回款
- 最终纳入口径成本
- 最终经营毛利 / 毛利率
- 最终偏差

### 8.3 同一项目不同时间查看

建议保证：

- 同一时间点看到的统一核算视图来自同一口径快照
- 不允许一个页面里混用不同成熟度的数据源
- 同一口径快照除金额结果外，还必须携带税务影响摘要、成本数据成熟度状态、成本侧动作建议与引用基线版本 / 快照版本，避免不同页面对同一项目做出不同解释

### 8.4 时点快照、期末冻结与补录重述

第二批当前固定两类历史回看口径：

1. 实时口径
   - 面向当前经营判断
   - 使用最新已确认的合同、回款、成本与变更包结果
2. 冻结口径
   - 面向月末、关账或管理复盘
   - 使用当时生成并冻结的统一核算快照

补充规则：

- 每个会计期间结束后，应允许生成统一核算时点快照，记录当期原始基线、有效变更后基线、已确认回款、已纳入口径成本、税务影响摘要、数据成熟度结果与成本侧动作建议
- 期末冻结后若发生补录、冲销或回溯修正，不得静默改写既有历史快照；应生成带版本号和重述原因的补充快照或重述快照
- 内部实现字段可继续使用 `asOfMode` 等技术命名，但系统前台、审批摘要、导出与通知统一使用“时点快照”“快照时点”“历史时点结果”等中文表达，不直接显示 `as-of`
- 页面默认展示“当前实时结果”，历史月视图或复盘页则默认展示该期间最后一个已批准快照，并显式提示是否存在后续重述
- 提成、管理复盘或审批留痕若引用了某个时点快照结果，必须能够回溯到具体快照版本，而不是只记一个会变化的当前值

### 8.4A 技术命名与前台展示映射

下表用于约束 DTO、查询视图、页面投影、审批摘要、导出与通知的统一表达。

| 内部技术命名 / 取值          | 前台统一显示     | 适用场景                     | 补充约束                                  |
| ---------------------------- | ---------------- | ---------------------------- | ----------------------------------------- |
| `asOfMode = realtime`        | `当前实时结果`   | 默认经营页、提成 gate 摘要   | 不得直接显示 `realtime` 或 `asOfMode`     |
| `asOfMode = period-end`      | `期末冻结快照`   | 月末、关账、管理复盘         | 必须同时显示 `快照时点` 与 `期末快照版本` |
| `asOfMode = restated`        | `重述快照`       | 历史补录、审批回看、跨期对账 | 必须同时显示 `重述原因` 与被替代快照版本  |
| `snapshotAt`                 | `快照时点`       | 页面标签、审批摘要、导出     | 不得写成英文术语                          |
| `periodEndSnapshotId`        | `期末快照版本`   | 页面标签、导出、审批留痕     | 应与 `snapshotAt` 成对出现                |
| `restatementReason`          | `重述原因`       | 页面标签、审批摘要           | 不得折叠进普通备注                        |
| `restatedFromSnapshotId`     | `被替代快照版本` | 重述详情、审批留痕           | 必须能回放到被替代的历史快照              |
| `ProjectOperatingAsOfView`   | `经营快照视图`   | 技术实现说明                 | 仅作技术命名，不得直接作为页面标题        |
| `PeriodClosingSnapshot`      | `期末冻结快照`   | 技术实现说明                 | 页面需显示业务含义，不直接暴露对象类名    |
| `OperatingRestatementRecord` | `经营重述记录`   | 技术实现说明                 | 页面需同时解释重述原因与影响范围          |

## 9. 与来源层的关系

统一核算视图不直接跨层读取所有来源事实，而应遵循：

1. 合同金额来自有效合同集合读侧，且每份合同都应可追溯到通过基线校验的当前有效 `ContractTermSnapshot`
2. 回款金额来自已确认回款读侧
3. 实际成本来自 `ProjectActualCostRecord` 聚合读侧
4. 估算成本来自签约前有效估算版本

这样做的目的是：

- 避免经营页重新发明映射和去重逻辑
- 避免采购合同、付款事实、采购发票被重复纳入
- 保证所有下游页面消费相同数字

---

## 10. 与提成口径的关系

### 10.1 统一核算视图不是提成结算口径

统一核算视图服务的是项目经营判断。

提成结算还会额外依赖：

- 提成阶段 gate
- 角色与权重冻结版本
- 制度中的特殊解锁与限制规则
- 异常、冲销、坏账、违规等调整链

### 10.2 但提成主线必须复用同一组经营事实

后续 `L5` 不应自行定义另一套收入和成本数字。

建议复用：

- 当前有效合同集合金额
- 已确认回款金额
- 已纳入口径实际成本
- 当前 / 最终经营毛利

同时建议明确：

- `L5` 的项目汇总 / 合同分别判断模式只影响提成 gate 的解释与可发上限拆分
- `L4` 输出的项目级经营数字保持唯一，不因 `L5` 的判断模式而切换底层金额
- `L5` 若需要引用历史经营结果或例外审批依据，应明确引用某个时点快照版本，而不是重新读取当前实时值

---

## 11. 页面表达建议

统一核算视图建议同时输出三类结果：

1. 摘要数值
   - 合同、回款、成本、毛利、毛利率
2. 派生状态
   - 数据成熟度
   - 经营风险层级
   - 成本侧动作建议
3. 解释入口
   - 进入偏差解释
   - 进入成本累计
   - 进入回款详情

说明：

- “摘要数值”默认只承诺统一口径，不承诺所有角色都拿到完整精确值
- 若当前访问角色不具备经营敏感字段查看权，应落为摘要值或遮罩占位，而不是返回另一套简化计算结果

不建议：

- 在总览页直接塞入完整台账明细
- 在没有数据成熟度提示的情况下直接给最终判断

---

## 12. 输出给后续任务的承接关系

### 12.1 输出给 `L4-T03`

`L4-T03` 应在本口径上继续展开：

- 为什么当前偏差形成
- 哪些风险仍未收口
- 哪些数字暂不稳定
- 如何直接消费 `L2` 已固定的税务影响摘要、成本数据成熟度状态、成本侧动作建议与引用基线 / 快照版本

### 12.2 输出给 `L4-T04`

`L4-T04` 应在本口径上继续明确：

- 经营核算如何反哺管理判断
- 如何影响复盘
- 如何影响提成阶段解释
- 如何把来自 `L2` 的成本侧动作建议与 `L4` 聚合动作等级稳定地下传到后续页面

### 12.3 输出给 `L5`

`L5` 应消费本口径产出的统一经营事实，而不是另起一套数字。

补充规则：

- `L5` 不得跳过本页重新推断税务影响摘要、成本数据成熟度状态、成本侧动作建议与引用基线 / 快照版本。

---

## 13. 当前结论

`L4-T02` 当前建议先固定以下统一口径：

- 合同：只认当前有效合同集合不含税金额汇总，并保留合同级拆解
- 回款：只认已确认生效回款不含税金额
- 成本：只认 `includedCostTotal`
- 估算：同时保留原始签约基线与当前有效变更后基线，待生效变更请求不提前并入正式基线
- 毛利：`已确认回款 - 已纳入口径实际成本`
- 毛利率：`当前经营毛利 / 已确认回款`
- 税务：默认不把可抵扣进项税并入经营成本，但必须单列税务影响摘要与待确认金额
- 成本侧成熟度：直接消费 `L2` 已固定的数据成熟度状态、分摊稳定性提示、未映射成本提示与成本侧动作建议，不在 `L4` 页面重新派生
- 历史回看：通过时点快照 / 期末冻结 / 重述快照控制，不静默改写历史经营结果

在此基础上，项目经营结果总览、偏差解释、提成解释和复盘页面才有可能长期保持一致。
