## 1. 文档目标

本文档用于把第二阶段路线图中的第一批主线，进一步细化为可讨论的信息架构草案。

当前只覆盖：

- `L1-S1` 项目签约前总入口
- `L1-S2` 技术确认与前期成本估算工作区
- `L1-S3` 招投标与报价评审承接

本文档不下沉到视觉稿或接口字段，而是先回答：

- 用户进入签约前阶段后，系统主入口应该长什么样
- 哪些工作区应该成为一级信息架构
- 各角色应该在同一个项目里看到什么、做什么、如何切换
- 阶段之间如何承接，避免重复录入和上下文丢失

---

## 2. 设计边界

### 2.1 当前覆盖范围

签约前统一工作区当前只覆盖以下阶段：

- 线索登记
- 立项评估
- 技术可行性与范围确认
- 前期成本估算
- 招投标 / 商务竞标
- 报价与毛利评审

### 2.2 当前不覆盖范围

本文档当前不覆盖：

- 合同签约后的移交工作区
- 执行期成本归集工作区
- 项目经营核算视图
- 提成阶段视图
- 最终页面视觉样式与组件实现

---

## 3. 信息架构总原则

### 3.1 一个项目，一个签约前工作区

用户不应先进入“线索模块”“投标模块”“报价模块”再自行拼上下文，而应直接进入某个 `Project` 的签约前工作区。

### 3.2 阶段不是孤立页面，而是连续工作带

签约前阶段应被用户感知为一条从机会判断到合同就绪的连续工作带，而不是若干分散菜单。

### 3.3 按任务组织，而不是按数据表组织

一级信息架构应围绕用户要完成的事，而不是围绕后端对象名或表名命名。

### 3.4 所有工作区都要回答 4 个问题

- 现在走到哪一步
- 下一步是什么
- 还缺什么
- 谁该处理

---

## 4. 签约前统一工作区顶层结构

### 4.1 顶层入口建议

建议在 `Project` 下形成独立的“签约前工作区”，其顶层结构如下：

1. 项目总览
2. 立项与推进
3. 技术与成本
4. 招投标 / 商务竞标
5. 报价与毛利评审
6. 签约就绪

其中：

- `项目总览` 是签约前驾驶舱
- `立项与推进` 承接 `L1-S1`
- `技术与成本` 承接 `L1-S2`
- `招投标 / 商务竞标` 与 `报价与毛利评审` 共同承接 `L1-S3`
- `签约就绪` 是签约前最终收口页，不属于合同页本身

### 4.2 顶层导航规则

- 顶层标签固定，避免不同项目因流程形态不同而彻底变形
- 不同项目形态的差异，通过页面内区块显隐和步骤状态体现
- 用户始终能返回 `项目总览`，查看全局进度、阻断项和责任人

## 5. 各工作区草案

### 5.1 项目总览

**主要用户**:

- 销售
- 副总经理 / 流程负责人
- 技术支持 / 售前
- 商务行政
- 财务

**目标**:

让所有参与签约前阶段的人，先在一个地方看懂项目当前状态，而不是先钻进明细页。

**建议包含的核心区块**:

1. 项目阶段条
   - 当前处于：线索 / 立项 / 技术确认 / 成本估算 / 投标 / 报价评审 / 签约就绪
2. 下一步动作卡
   - 当前最关键下一步
   - 当前责任人
   - 截止时间
3. 阻断项卡
   - 缺失材料
   - 未完成审批
   - 未确认技术范围
   - 未形成成本清单
4. 关键结论卡
   - 当前立项结论
   - 当前技术结论
   - 当前投标结论
   - 当前报价 / 毛利结论
5. 协作责任卡
   - 销售
   - 售前
   - 财务
   - 商务行政
   - 审批责任人

**关键体验要求**:

- 首页先给摘要，不先给字段明细
- 一眼能判断能否继续推进
- 一眼能看到谁卡住了当前项目

### 5.2 立项与推进

**主要用户**:

- 销售
- 副总经理 / 流程负责人

**目标**:

把线索和立项收成同一工作带，让销售明白“我怎么把机会推进到可正式进入签约前主链”。

**建议包含的核心区块**:

1. 基础机会信息
2. 立项评估材料
3. 立项审批区
4. 条件通过 / 否决原因区
5. 整改与待补清单

**关键体验要求**:

- 条件通过不应只是状态，要转成可执行整改项
- 否决必须结构化表达原因
- 立项结论应直接进入后续技术确认上下文

### 5.3 技术与成本

**主要用户**:

- 技术支持 / 售前
- 销售
- 财务

**目标**:

把技术判断、范围边界、风险说明和前期成本估算收成同一工作区。

**建议包含的核心区块**:

1. 需求与背景摘要
2. 可行性判断
3. 范围内 / 范围外 / 前提条件
4. 风险与保留意见
5. 前期成本清单
   - 人力实施
   - 第三方采购 / 集成
   - 外包 / 专项支持
   - 差旅 / 部署 / 云资源 / 其他费用
   - 税务成本 / 税负影响
   - 不确定成本项
6. 成本估算版本记录

**关键体验要求**:

- 售前不应只上传附件，而要形成结构化快照
- 成本不能只有总额，必须有必要成本清单
- 税务成本应在工作区中显式可见，而不是只在后续报价备注中隐含出现
- 销售和财务能在同一页理解售前结论，而不是二次口头转译

### 5.4 招投标 / 商务竞标

**主要用户**:

- 销售
- 技术支持 / 售前
- 商务行政
- 副总经理 / 流程负责人

**目标**:

把多形态投标统一挂在 `Project` 下，形成连续协作过程，而不是附件和备注堆积。

**建议包含的核心区块**:

1. 投标形态与当前阶段
   - 公开招标 / 邀标 / 比选 / 商务竞标 / 混合型
2. 投标决策
3. 材料清单与责任分工
4. 标书 / 方案 / 递交版本
5. 澄清 / 答疑 / 述标时间线
6. 结果登记
   - 中标 / 失标 / 弃标 / 废标

**关键体验要求**:

- 不同投标形态共享同一骨架
- 所有过程信息可回看，不丢版本
- 结果能直接带入报价评审或签约就绪判断

### 5.5 报价与毛利评审

**主要用户**:

- 销售
- 财务
- 副总经理 / 流程负责人

**目标**:

让报价不再是孤立价格动作，而是围绕成本、回款条件、税务条件和毛利的综合判断。

**建议包含的核心区块**:

1. 报价草案
2. 成本估算引用
3. 回款条件与税务条件
4. 毛利与例外提示
5. 审批摘要快照
6. 放行 / 条件放行 / 否决结论

**关键体验要求**:

- 审批人先看判断摘要，再看明细
- 能清楚解释为什么能放行或不能放行
- 审批摘要、审批通知、打印材料与导出预览必须引用同一份 `summarySnapshotId`，不得在信息架构层默认允许各入口自行裁剪详情字段
- `approvalScenarioKey`、`summaryPackageKey`、`summarySnapshotId`、`projectionLevel` 与 `exportPolicy` 应作为报价评审到签约就绪承接时的稳定引用，而不是页面临时拼装结果
- 审批结论直接进入签约就绪页

### 5.6 签约就绪

**主要用户**:

- 销售
- 商务行政
- 财务

**目标**:

在进入正式合同页之前，先明确签约前主链是否已真正收口。

**建议包含的核心区块**:

1. 签约前置项检查
   - 立项已完成
   - 技术范围已确认
   - 成本估算已形成
   - 投标结果已明确
   - 报价评审已放行
2. 可直接复用到合同的关键事实
3. 当前仍缺失的签约前阻断项

**关键体验要求**:

- “可签约”必须是显式状态，而不是靠用户自己判断
- 已确认事实能直接进入合同登记
- 签约就绪消费的报价放行结论，应与报价评审页、审批通知和打印材料共享同一份摘要快照与投影级别，不得在进入签约就绪后重新组织一份“看起来差不多”的摘要
- 未通过前不应让用户误以为合同流程已经具备条件

## 6. 角色视角差异

同一个签约前工作区，不同角色应看到不同的默认重点。

### 6.1 销售默认重点

- 项目总览
- 下一步动作
- 投标推进状态
- 报价评审卡点
- 签约就绪情况

### 6.2 副总经理 / 流程负责人默认重点

- 立项结论摘要
- 毛利与风险摘要
- 当前待审批事项
- 条件通过 / 否决原因

### 6.3 技术支持 / 售前默认重点

- 需求背景
- 范围快照
- 风险与保留意见
- 前期成本清单及版本

### 6.4 商务行政默认重点

- 投标材料完成度
- 结果登记
- 签约就绪检查
- 后续进入合同的关键信息

### 6.5 财务默认重点

- 成本估算摘要
- 税务成本 / 税负影响摘要
- 回款 / 税务条件
- 毛利评审摘要
- 与后续经营核算相关的前置事实

---

## 7. 关键承接规则

签约前统一工作区必须至少支持以下承接关系：

1. 立项结论 -> 技术与成本
2. 技术范围快照 -> 投标 / 报价评审
3. 前期成本清单 -> 毛利评审
4. 投标结果 -> 签约就绪
5. 报价放行结论与审批摘要快照 -> 签约就绪
6. 签约就绪事实 -> 合同签约

如果这些承接关系不能在界面和数据上自然表达，用户仍会感知为多个孤岛页面。

---

## 8. 当前建议的下一步细化顺序

在本信息架构草案基础上，建议继续按以下顺序细化：

1. `项目总览` 与 `立项与推进`
   - 因为这是销售进入签约前主链的第一入口
2. `技术与成本`
   - 因为这是第二阶段最重要的结构化能力新增点之一
3. `招投标 / 商务竞标`
   - 因为多形态协作复杂，需单独展开
4. `报价与毛利评审`
   - 因为涉及审批摘要和经营判断表达
5. `签约就绪`
   - 因为它是签约前主链到合同主链的桥
