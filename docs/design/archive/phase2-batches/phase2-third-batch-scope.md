## 1. 文档目标

本文档用于把第三批专题从 `follow-up` 表格中的一组增强项，收成正式范围说明。

本文档重点回答以下问题：

- 第三批到底纳入哪些专题
- 第三批的目标是“补什么健壮性与审批增强”，而不是回头改写第一批、第二批主事实
- 第三批与第二批、第四批的边界分别是什么
- 第三批完成到什么程度，才可认为已具备进入实现设计下钻的正式入口

---

## 2. 第三批范围原则

第三批范围统一采用以下四条原则：

1. 不反向推翻第一批已冻结的主对象、主事实与敏感边界。
2. 不反向推翻第二批已建立的经营可信源、历史回看口径与 `L4 -> L5` 绑定结果。
3. 只处理会直接影响真实流程健壮性、审批最小可见和冻结后争议处理的专题。
4. 不把第四批未来扩展与表达增强项混入第三批。

---

## 4. 第三批明确不纳入的内容

### 4.1 当前不回卷第一批与第二批的内容

以下内容虽与第三批存在依赖，但不再回卷为前两批未完成项：

- 多合同主假设与回款判断模式冻结
- `签约就绪 -> ContractTermSnapshot / ReceivablePlan` 对象链
- 商业放行基线与合同差异校验
- 第二阶段验收对象与证据链
- `internalCostRate` 治理基线
- 敏感数据主边界与最小可见集
- 分摊成本、阶段归属、税务处理、经营基线、时点快照 / 期末 / 重述
- `L4 -> L5` gate 绑定矩阵

### 4.2 当前不提前纳入第四批的内容

以下内容当前不进入第三批范围：

- `R1-001` 多币种与汇率扩展
- `R3-003` 分期移交
- `R1-005` 异常调整摘要展示
- `R3-005` 细粒度通知与协作触发策略

## 5. 第三批入场与出场条件

### 5.1 入场条件

第三批开始前应满足：

1. 第一批六个专题不存在方向级未决策问题。
2. 第二批六份实现设计总文档写回已完成，不再只停留在桥接层。
3. `../reviews/phase2-review-follow-up-plan.md` 与本文件对第三批专题范围保持一致。
4. 第三批主回写文档已明确，不再继续拆散成临时讨论结论。

### 5.2 出场条件

第三批完成后至少应满足：

1. 五个专题均已形成正式规则回写并落到主文档。
2. `../reviews/phase2-review-follow-up-plan.md` 已同步真实状态。
3. `../../poms-design-progress.md` 已同步第三批当前阶段判断。
4. `../../phase2-detailed-design-index-map.md` 已提供第三批范围入口。
5. 第三批实现映射桥接入口已固定为 `phase2-third-batch-implementation-mapping.md`，不再重新定义第三批专题。
6. 六份实现设计总文档已完成第三批首轮写回，不再只停留在范围说明和桥接入口层。
7. `../../phase2-contract-to-handover-workspace.md`、`../../phase2-presigning-workspace-handoff-map.md`、`../../phase2-data-permission-and-sensitive-visibility-design.md`、`../../phase2-commission-freeze-at-handover.md`、`../../phase2-commission-staged-payout-adjustment-paths.md` 与 `../../workflow-and-approval-design.md` 已完成第三批首轮写回。
8. `../../phase2-project-handover-gate-workspace.md`、`../../phase2-commission-stage-gate-overview-workspace.md` 与 `../../phase2-estimated-to-actual-cost-bridge.md` 已完成第三批必要联动补点。

## 6. 当前结论

第三批当前已经不适合继续只以“后续增强项”口头保留，而应被视为正式的流程健壮性与审批增强范围。

在此基础上：

- 第一批负责主事实和主边界的稳定落地
- 第二批负责把经营可信口径、历史回看口径和 `L4 -> L5` 动作绑定写稳
- 第三批负责把真实异常路径、审批最小可见和冻结后争议处理补成正式链路，并已完成六份实现设计总文档、关键业务主文档与剩余关键联动文档的首轮写回 / 补点
- 第四批继续作为范围限制或未来扩展处理

这样，第二阶段后续若继续进入实现设计层，就不会在主事实已稳定后又因为流程负路径、审批摘要或争议处理缺口重新掉回线下。
