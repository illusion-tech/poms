## 1. 文档目标

本文档用于承接首轮详细设计正式评审的 `Passed with follow-up` 结论，把后续项明确拆分为：

- 实现前必须补齐的内容
- 下一轮详细设计继续细化的内容
- 仅需治理同步的内容

本文档不重新评审详细设计，也不提前冻结接口与数据模型；它只用于约束“哪些事现在必须做，哪些事可以后移”。

---

## 2. 本轮评审结论来源

本轮 follow-up 基于以下事实形成：

- `design-convergence-review-checklist.md` 中高优先级项已关闭
- `design-review-execution-checklist.md` 已完成首轮正式评审勾选
- 当前整体结论为 `Passed with follow-up`

---

## 3. 实现前必须补齐项

以下事项建议在进入真正的 schema / DDL 级细化前明确：

1. 明确字段级全量矩阵是否继续下钻到页面 / DTO 级别，还是以当前字段包基线作为实现约束。
2. 明确命令型动作接口的最小集合，至少覆盖生效、冻结、关闭、作废、冲销、重算、审批、确认等高敏感动作。当前已形成 `../../interface-command-design.md` 与 `../../interface-openapi-dto-design.md` 两层基线，后续进入真正 OpenAPI 文件层时不得再回退接口形态边界。
3. 明确对象状态迁移与审批实例状态的映射边界，避免接口设计时再次混用业务状态与审批状态。
4. 明确读侧查询视图边界，避免表结构冻结只围绕写侧对象展开。当前已形成 `../../query-view-boundary-design.md` 首版基线，开始把列表、详情、经营看板和统一待办的读侧约束转为稳定输入。
5. 明确哪些 follow-up 只影响实现约束，不再反向改动已通过评审的业务边界。当前已形成 `../../data-model-prerequisites.md`、`../../table-structure-freeze-design.md` 与 `../../schema-ddl-design.md` 三层基线，开始把接口合同与查询视图边界继续映射到真实建表前输入物。

---

## 4. 下一轮详细设计继续细化项

以下事项可以留在下一轮详细设计中继续细化，不阻塞当前进入实现前冻结准备：

1. 字段级权限是否继续细化到页面按钮、批量操作、导入导出等更细交互层级。
2. 多组织例外权限、复杂审批链和少量例外关闭场景的进一步展开。
3. `CommissionPayout` 后续是否拆分为业务发放与真实财务支付两层状态。
4. 多合同项目下提成池汇总口径、质保金多次释放等进阶规则。

## 5. 文案与治理同步项

以下事项不构成设计阻塞，但应在后续持续同步：

1. `../../README.md`、`../../poms-design-progress.md` 与评审类文档的阅读路径提示保持一致。
2. 历史评审摘要继续明确“历史结论”与“当前状态源”的边界。
3. 后续若新增评审摘要或冻结类文档，应同步补入目录索引与进度板。

---

## 6. 进入接口与数据模型冻结准备的前提

建议满足以下条件后，再正式进入 schema / DDL 级细化：

- 已确认实现前必须补齐项的范围
- 已确认字段包基线是否足以支撑第一阶段接口设计
- 已形成首版 OpenAPI / DTO 输入输出边界基线
- 已形成首版查询视图边界基线
- 已形成首版数据模型冻结前提基线
- 已形成首版表结构冻结设计基线
- 已形成首版 schema / DDL 细化基线
- 已确认高敏感动作采用命令型接口，而不是普通更新接口
- 已确认评审 follow-up 不会再次推翻当前业务边界与对象分层

---

## 7. 当前结论

当前最稳妥的推进方式不是继续扩写新的业务设计专题，而是先把首轮正式评审的 follow-up 约束逐层落实为接口边界、查询视图边界、表结构冻结设计与 schema / DDL 细化基线，再进入真实迁移脚本或 ORM schema 编写。这样可以把返工风险限制在脚本落地前，而不是在数据库结构已开始实现后再回退。
