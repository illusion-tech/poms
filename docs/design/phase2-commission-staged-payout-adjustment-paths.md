## 1. 文档目标

本文档用于细化 `L5-T02`，即：

把提成治理里的分阶段发放、暂停、扣回、冲销、补发和重算路径收成一条连续可执行的操作链，而不是分散在多个对象说明里。

这里的重点不是重新定义制度，而是把制度允许的动作和系统中的操作路径对齐。

---

## 2. 工作区定位

### 2.1 面向用户

- 主用户:
  - 财务
- 高频协作角色:
  - 管理层
  - 销售

### 2.2 使用时机

- 当前提成 gate 已允许进入某一发放阶段
- 当前项目已发放后又出现异常，需要暂停、调整、扣回、冲销或重算

### 2.3 工作区目标

- 统一表达分阶段发放路径
- 统一表达异常调整路径
- 统一表达历史已发结果与后续调整链的关系

---

## 3. 必须回答的核心问题

分阶段发放与异常调整路径必须能回答：

1. 当前项目现在允许发放哪个阶段
2. 当前阶段最多能发到哪里
3. 已经发放过的部分后续如何继续推进
4. 出现退款、坏账、违规、离职等情况后，系统如何处理
5. 已发记录如何保持可追溯，不被后续调整覆盖

---

## 4. 分阶段发放路径

### 4.1 总原则

- 不得提前预支
- 不得跨阶段发放
- 不得跳过当前 gate 直接发下一个阶段
- 每次发放都应绑定一个明确阶段和当时的有效依据

### 4.2 阶段建议

建议至少拆成：

1. 第一阶段发放
2. 第二阶段发放
3. 最终阶段发放
4. 质保金结算

### 4.3 第一阶段发放

应支持：

- 标准首付款项目档位选择
- 低首付款项目档位选择
- 发起申请
- 审批放行
- 登记业务发放

### 4.4 第二阶段发放

应支持：

- 验收 / 阶段成果确认条件检查
- 80% 回款条件检查
- 第二阶段累计可发上限判断
- 发起申请、审批和登记

**对象与证据链口径**:

- 第二阶段发放统一复用 `AcceptanceRecord`，不额外引入新的阶段成果对象
- 本阶段允许作为前置的 `AcceptanceRecord` 类型仅包括：`阶段成果确认`、`阶段验收`
- 当前发放申请必须附带被引用的 `AcceptanceRecord` 标识、确认时间和证据摘要
- 若 `AcceptanceRecord` 只有结论没有证据引用，不得进入第二阶段发放申请链
- 发放申请页、审批摘要和打印材料中出现的经营依据字段，必须遵循 `phase2-data-permission-and-sensitive-visibility-design.md` 的最小可见集与遮罩规则，不得因进入发放审批链而自动扩展到完整敏感明细

**确认动作建议**:

1. 业务侧先形成阶段成果或验收材料
2. 相关责任人 / 业务确认角色执行 `confirmAcceptance`
3. 系统生成确认留痕、关闭相关待办，并把 `AcceptanceRecord` 标记为当前有效结论
4. 财务在第二阶段发放申请中引用该记录，而不是手工录入“已验收”文本

**第二阶段阻断规则**:

- 无有效 `AcceptanceRecord` 不得发起第二阶段发放申请
- 有效 `AcceptanceRecord` 与当前项目、当前发放阶段不匹配，不得发起申请
- 证据链不完整时，只能补证据，不得越过确认链直接申请发放

### 4.5 最终阶段发放

应支持：

- 非质保部分 100% 回款或合同履行完成的判断
- 剩余非质保部分提成的发放路径

### 4.6 质保金结算

本阶段只做入口承接，不展开全部细则。

质保金部分在 `L5-T03` 继续详细展开。

## 5. 分阶段发放的动作链建议

建议每个阶段统一走以下动作链：

1. 系统判断当前阶段可发上限
2. 用户发起本阶段发放申请
3. 系统附带当前 gate 依据与经营依据摘要
4. 审批决定是否放行及按何档位执行
5. 财务登记业务发放结果
6. 系统把阶段累计可发与已发结果更新为新状态

所有阶段审批都应继续消费场景级审批摘要字段包，而不是把详情页字段随机裁剪成审批材料；打印、导出和通知摘要也应沿用同一份场景包结果。

其中第二阶段至少应补充以下前置检查：

1. 读取当前有效 `AcceptanceRecord`
2. 校验证据链是否完整
3. 校验是否已满足 80% 已确认回款
4. 再进入第二阶段档位审批与发放登记
5. 对审批摘要、打印材料和导出结果应用当前角色可见边界，避免通过发放链路侧漏毛利精确值、人工成本率或全员提成金额

---

## 6. 异常调整路径

### 6.1 总原则

- 异常不得直接覆盖原发放记录
- 必须通过独立调整链处理
- 必须保留原已发结果、调整原因、执行动作和替代关系

### 6.2 典型异常类型

建议至少覆盖：

1. 退款 / 回款冲减
2. 坏账 / 无法收回
3. 违规处理
4. 离职处理
5. 合同变更或成本变化触发重算

### 6.3 退款 / 回款冲减

建议路径：

1. 识别当前已发部分受影响
2. 暂停后续未发放阶段
3. 生成异常调整记录
4. 判断是否需要扣回、冲销或重算差额

### 6.4 坏账 / 无法收回

建议路径：

1. 确认异常事实
2. 暂停后续发放
3. 对已发放部分形成受控调整判断
4. 必要时触发取消或扣回

### 6.5 违规处理

建议路径：

1. 记录违规结论
2. 选择处理结果:
   - 暂停
   - 降低
   - 取消
3. 对后续阶段发放产生阻断
4. 对已发部分形成调整链

### 6.6 离职处理

建议路径：

1. 区分“离职前已满足发放条件”和“尚未满足发放条件”
2. 已满足但未发放的部分，允许按既得权益结算
3. 尚未满足条件的部分，不自动进入发放
4. 涉及质保金的离职特例，在 `L5-T03` 再细化

### 6.7 合同变更 / 成本变化 / 其他经营变化

建议路径：

1. 识别是否影响已生效提成计算基础
2. 生成重算候选
3. 形成新计算版本
4. 必要时联动后续阶段发放调整

### 6.8 冻结后争议 / 仲裁联动

当冻结后争议进入第三批正式链路后，分阶段发放与异常调整必须补充以下联动规则：

1. 一旦存在待仲裁的冻结后争议，后续未发放阶段应默认进入受控暂停，不得继续按原冻结版本推进新的发放。
2. 仲裁前必须先完成影响评估，明确争议只影响后续阶段，还是同时影响既有计算结果与已发记录。
3. 若仲裁结论要求替代冻结版本，应先生成替代版本和新的计算基础，再判断是扣回、冲销、补发还是仅恢复后续发放。
4. 已发记录始终保留原动作事实；任何负向或正向修正都必须通过独立调整链表达，不得直接改原发放记录。
5. 争议处理、仲裁摘要和影响评估继续沿用场景级审批摘要字段包，不因异常链路而默认放开完整提成明细。

## 7. 暂停、扣回、冲销、补发、重算的区别

这部分必须对用户讲清楚，否则最容易混淆。

### 7.1 暂停

- 作用：阻止后续继续发放
- 不直接消灭原已发结果
- 常用于异常待确认期间

### 7.2 扣回

- 作用：对已发结果形成负向调整
- 常用于损失已确认但无需完全作废原链路时

### 7.3 冲销

- 作用：明确使某次已发生的业务发放结果进入反向处理链
- 适用于需要明确撤销原发放业务含义的场景

### 7.4 补发

- 作用：对之前未发足或差额应补部分形成新增发放记录
- 不能直接改原记录

### 7.5 重算

- 作用：对计算基础重新生成版本
- 不直接等于扣回或补发，但通常会成为后续调整依据

## 8. 页面建议结构

### 8.1 当前可执行动作区

建议显示：

- 当前可发阶段
- 当前累计可发上限
- 当前已发累计
- 当前剩余可发空间

### 8.2 已发记录时间线

建议显示：

- 各阶段发起、审批、发放、暂停、冲销、补发记录
- 每条记录对应的阶段、金额、原因和状态

### 8.3 异常调整时间线

建议显示：

- 异常类型
- 认定时间
- 对应动作
- 是否触发重算
- 仲裁结论与替代冻结版本摘要
- 是否影响后续阶段

### 8.4 当前阻断与待处理区

建议显示：

- 当前被暂停的原因
- 当前是否有待审批调整
- 当前是否有待执行重算
- 当前是否存在冻结后争议待仲裁或待回溯影响确认

---

## 9. 角色视角差异

### 9.1 财务

默认重点：

- 当前可以发什么
- 当前不能发什么
- 已发结果和调整链是否一致
- 异常后该走哪条路径

### 9.2 管理层

默认重点：

- 需要审批什么
- 当前异常处理会影响哪些后续阶段
- 当前是暂停、扣回还是取消更合适

### 9.3 销售

默认重点：

- 当前提成为什么停在这里
- 当前是临时暂停还是已实质调整
- 当前下一步要推动什么条件恢复

---

## 10. 与后续任务的承接关系

### 10.1 输出给 `L5-T03`

`L5-T03` 应在本路径基础上继续展开：

- 最终阶段与质保金结算
- 非质保部分与质保金部分的最后收口

### 10.2 输出给 `L5-T04`

`L5-T04` 应把本页的动作语义、状态语义和异常语义统一收成可解释表达规则。

## 11. 当前结论

`L5-T02` 当前建议先固定以下原则：

- 分阶段发放必须按阶段、按 gate、按累计上限受控进行
- 异常调整必须走独立调整链，不能覆盖原记录
- 暂停、扣回、冲销、补发、重算必须明确区分
- 冻结后争议进入第三批正式链路后，发放与调整必须消费争议仲裁和回溯影响结论，而不是继续沿用旧冻结版本静默推进
- 后续质保金与最终收口应建立在本页已固定的阶段与异常路径之上

在这个基础上，提成治理才能把“能不能发”和“出了问题怎么办”真正串成一条用户可执行链路。
