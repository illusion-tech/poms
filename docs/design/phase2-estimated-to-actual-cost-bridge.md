## 1. 文档目标

本文档用于细化 `L2-T05`，即：

把签约前成本估算与执行期实际成本建立稳定承接关系。

它重点回答：

- 签约前应以哪个估算版本作为执行期对比基线
- 估算成本项如何映射到执行期实际成本类型
- 实际成本累计结果如何与估算值形成偏差
- 哪些偏差属于正常落地，哪些偏差需要解释或预警

---

## 2. 核心设计结论

### 2.1 必须固定“估算成本基线版本”

执行期实际成本不能和任意一个历史估算版本对比。

当前建议固定：

**“进入签约就绪并最终被签约路径采用的最新有效估算版本，作为执行期成本对比基线。”**

### 2.2 必须固定“估算项 -> 实际类型”的映射关系

如果不先固定映射关系：

- 偏差无法解释
- 人力成本会与采购、费用混算
- 税务成本会被掩盖

因此，第二阶段应形成正式映射表。

### 2.3 偏差解释不只看金额差

偏差判断至少应同时看：

- 金额差
- 比例差
- 来源类型差异
- 阶段分布差异
- 是否属于高不确定项兑现

---

## 3. 估算成本基线版本规则

### 3.1 基线来源

当前建议由签约前 `技术与成本` 工作区输出的成本估算版本中，选取：

- 被报价 / 毛利评审引用
- 并最终进入 `签约就绪`

的那一个版本，作为执行期基线。

### 3.2 基线冻结时点

当前建议基线冻结时点为：

- `签约就绪` 判断通过时
  或
- 最终签约前的最后一次有效成本评审通过时

### 3.3 为什么不能后补选择

如果执行中再回头选基线版本，后续偏差判断会失真，也容易被人为改写。

因此：

- 基线版本一旦冻结，不应静默替换
- 若必须替换，应保留替换原因和前后版本差异

### 3.4 执行中合同变更基线与变更包基线

第二批当前固定两层基线，避免把“正式批准的范围变更”与“真实超支”混在一起：

1. `originalBaselineCost`
  - 来自签约前最后被正式采用的有效估算版本
  - 在签约就绪 / 最终签约采用时冻结
  - 后续不得被静默覆盖
2. `changePackageBaselineCost`
  - 来自执行期已经正式批准并生效的合同变更包、范围变更包或同等级经营批准包
  - 只累计已批准、已生效的增减项

当前项目用于执行中正式对比的基线应固定为：

`currentEffectiveBaselineCost = originalBaselineCost + changePackageBaselineCost`

补充约束：

- 待审批、已发起但未生效的变更请求，只能作为“待生效变更影响”展示，不得提前并入正式基线
- 历史对比必须同时保留“相对原始签约基线偏移”和“相对当前有效变更后基线偏差”两套结果
- 变更包基线只能通过显式增量累加或冲减表达，不允许直接改写 `originalBaselineCost`

### 3.5 移交前再基线化与执行期基线衔接

第三批当前进一步固定一条边界：

- 合同生效后、项目正式移交前发生的正式合同变更，不直接视为执行期 `changePackageBaselineCost`
- 这类变更若已进入 `L3` 的再基线化链，应先形成“当前移交前有效基线”，再作为执行期承接起点进入 `L2`

换句话说：

1. `originalBaselineCost` 仍表示签约前最后被正式采用的原始签约基线。
2. 若合同生效后到移交前发生正式变更，且已完成再基线化，则执行期起始比较口径应消费 `currentEffectiveHandoverBaselineCost`。
3. `currentEffectiveHandoverBaselineCost` 不是对 `originalBaselineCost` 的静默覆盖，而是通过再基线化链显式形成的移交前有效承接基线。
4. 只有项目进入执行态之后才产生的正式变更包，才继续累加到 `changePackageBaselineCost`。

补充约束：

- 若移交前再基线化尚未收口，执行期不得把旧基线直接拿来当正式偏差比较口径。
- 偏差解释至少应能区分：原始签约基线偏移、移交前再基线化吸收、执行期变更包吸收。

## 4. 估算项到实际成本类型映射

当前建议固定以下一阶映射：

| 签约前估算项                    | 执行期默认实际成本类型                     | 说明                           |
| ------------------------------- | ------------------------------------------ | ------------------------------ |
| 人力实施成本                    | `LABOR`                                    | 软件开发服务类项目的主要承接项 |
| 第三方采购 / 集成成本           | `PROCUREMENT` / `PAYMENT_FACT` / `INVOICE` | 视来源链阶段逐步体现           |
| 外包 / 专项支持成本             | `PROCUREMENT` / `PAYMENT_FACT` / `LABOR`   | 内外部承接方式不同             |
| 差旅 / 部署 / 云资源 / 杂项费用 | `EXPENSE` / `PAYMENT_FACT` / `INVOICE`     | 费用型支出                     |
| 税务成本 / 税负影响             | `INVOICE` 税务影响 + 其他税务相关成本项    | 不应被吞并                     |
| 不确定成本项                    | 对应实际类型 + `高风险/不确定兑现` 标记    | 用于偏差解释                   |

### 4.1 人力实施成本

这是当前必须重点固定的一项。

映射规则：

- 估算侧的人力实施成本
- 默认承接到执行期 `LABOR`
- 以“人员/角色 + 项目 + 周/月”汇总归集结果参与对比

### 4.2 采购 / 外部集成成本

映射规则：

- 签约前的第三方采购 / 集成估算
- 执行期承接到采购链相关实际成本
- 对比时应优先看已纳入口径的实际支出，不是简单看承诺金额

### 4.3 税务成本

映射规则：

- 签约前显式估算的税务成本
- 执行期通过发票税务影响和其他税务相关成本表达
- 对比时不能把税务影响吞并到“其他费用”

### 4.4 分摊成本与项目共享事实承接

若签约前估算项本身就是共享成本或后续执行中需要按多个项目分摊，当前建议固定：

- 项目级对比只消费分摊后的估算份额与分摊后的实际份额
- 不允许在单项目偏差页直接拿共享来源全额与某一个项目的实际成本对比

典型规则：

1. 若估算阶段已经明确分摊依据，应同步冻结项目级估算份额。
2. 若估算阶段尚不能稳定拆分，则该项应标记为“待分摊估算项”，在分摊规则批准前不作为项目级正式偏差基线，只进入数据成熟度提示。
3. 执行中若因合同变更包导致共享成本重新拆分，应以变更包增量方式调整项目份额，而不是回写历史原始基线。

## 5. 对比结构建议

每个项目至少应形成以下对比结构：

- 原始签约基线总额
- 已批准变更包基线总额
- 当前有效变更后基线总额
- 实际已纳入口径总额
- 相对当前有效基线偏差金额
- 相对当前有效基线偏差比例
- 相对原始签约基线偏移金额

并按成本项进一步拆解为：

- 原始基线值
- 变更包调整值
- 当前有效基线值
- 实际值
- 相对当前有效基线差额
- 相对原始基线偏移
- 差异说明

### 5.1 最小对比列建议

- 成本项名称
- 原始签约基线金额
- 已批准变更包金额
- 当前有效基线金额
- 实际已纳入口径金额
- 相对当前有效基线偏差金额
- 相对当前有效基线偏差比例
- 相对原始基线偏移金额
- 偏差主要来源
- 是否高风险

### 5.2 默认优先级

建议优先突出：

1. 人力成本偏差
2. 第三方采购 / 集成成本偏差
3. 税务成本偏差
4. 高不确定项兑现偏差

---

## 6. 偏差解释规则

### 6.1 正常偏差

以下情况可以先按正常偏差处理：

- 估算项粒度粗，执行期实际更细
- 小额费用波动
- 周/月节奏错位导致的阶段性偏差

### 6.2 必须解释的偏差

以下偏差必须被显式解释：

- 人力成本明显超估
- 第三方采购 / 外包支出明显超估
- 税务成本与原假设明显不一致
- 高不确定项集中兑现
- 原估算未覆盖的新成本类型出现
- 已发生成本主要来自待批准变更请求，而不是已生效变更包
- 分摊成本缺少稳定分摊依据，导致项目级偏差暂不可信

### 6.3 典型偏差原因标签建议

- 人力投入超预期
- 开发周期拉长
- 采购价格变化
- 外包范围扩大
- 费用补录
- 税务假设变化
- 估算遗漏
- 不确定项兑现
- 已批准变更包吸收
- 待生效变更请求影响
- 分摊依据调整

---

## 7. 未映射与新增成本处理

### 7.1 未映射成本

如果执行期出现的实际成本找不到对应估算项，当前建议：

- 仍允许进入项目实际成本累计
- 但必须标记为“未映射成本”
- 并进入偏差解释区

### 7.2 估算存在但实际尚未发生

如果估算项存在，但执行期尚无对应实际成本：

- 不应视为错误
- 但应保留“尚未兑现”状态
- 供阶段判断使用

### 7.3 新增成本类型

如果执行中出现签约前完全未覆盖的新成本类型：

- 应允许新增实际成本记录
- 但必须进入风险或偏差说明

## 8. 与累计视图的承接关系

`L2-T04` 已经固定了：

- `registeredCostTotal`
- `confirmedCostTotal`
- `includedCostTotal`

当前建议在估算对比中：

- 默认使用 `includedCostTotal` 作为正式对比口径
- 必要时辅以 `confirmedCostTotal` 解释“为什么看起来快超了，但还没完全纳入口径”

### 8.1 为什么不用 `registeredCostTotal` 做正式偏差

因为它太早，容易把尚未确认或尚未吸收的记录也算进去，导致偏差过度放大。

---

## 9. 与后续主线的承接关系

### 9.1 输出给 `L4`

`L4` 将在此基础上形成：

- 项目经营结果总览
- 成本偏差解释
- 毛利变化解释

当前至少应稳定输出以下结果给 `L4`：

- `originalBaselineCost`
- `currentEffectiveHandoverBaselineCost` 或 `currentEffectiveBaselineCost` 的正式比较口径引用
- 相对原始签约基线偏移结果
- 相对当前有效基线偏差结果
- 税务影响摘要
- 分摊依据是否稳定与未映射成本摘要
- 当前成本侧数据成熟度状态
- 当前成本侧动作建议: `PROMPT / REVIEW / BLOCK`
- 当前引用基线版本与快照版本

补充规则：

- `L4` 不得重新推断“当前到底在比较哪一层基线”，而应直接消费本页输出的正式比较口径。
- 税务影响摘要、分摊稳定性、未映射成本与当前成本侧动作建议必须和基线版本、快照版本一起下传，避免下游只拿偏差数字重新做解释。

### 9.2 输出给 `L5`

虽然提成不直接消费全部估算字段，但后续：

- 成本失控
- 毛利偏离
- 异常成本兑现

都可能影响提成判断或管理决策解释。

因此，本页至少还应向 `L5` 稳定提供：

- 当前比较使用的正式基线版本引用
- 当前税务影响摘要
- 当前成本侧动作建议: `PROMPT / REVIEW / BLOCK`
- 当前引用快照版本

`L5` 可以不直接展示全部成本映射明细，但不得绕过这些稳定输出自行重算成本侧解释。

---

## 10. 当前结论

`L2-T05` 当前建议固定为：

- 签约前最后被正式采用的有效估算版本，作为执行期对比基线
- 合同生效后到移交前的正式变更若已完成再基线化，应以当前移交前有效基线作为执行期承接起点，而不是把它误算为执行期变更包
- 同时保留“原始签约基线”与“当前有效变更后基线”两层结果，不把执行期变更静默改写成原始基线
- 估算项必须映射到统一实际成本类型
- 正式偏差判断默认使用 `includedCostTotal`
- 人力实施成本必须与执行期 `LABOR` 累计结果直接可比
- 分摊成本与共享事实必须先形成项目级份额，才能进入正式偏差判断
- 未映射成本和高不确定项兑现必须进入偏差解释
- 并把正式比较口径、税务影响摘要、数据成熟度状态、成本侧动作建议与引用快照版本稳定输出给 `L4 / L5`

因此，`L2` 执行期成本归集主线到此可以视为第一轮基线闭环完成；下一步应按路线图进入 `L3-T01`，补“合同到移交的承接工作区”详细草案。
