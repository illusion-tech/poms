## 1. 文档目标

本文档用于细化签约前统一工作区中的 `项目总览`。

`项目总览` 不是详情页摘要版，也不是简单数据概览卡片集合，而是签约前阶段的驾驶舱。它必须先回答：

- 当前项目走到哪一步
- 当前谁该处理
- 当前为什么能继续或不能继续
- 当前哪些关键结论已经形成

---

## 2. 工作区定位

### 2.1 面向用户

- 销售
- 副总经理 / 流程负责人
- 技术支持 / 售前
- 商务行政
- 财务

### 2.2 使用时机

- 用户从项目列表进入某个签约前项目时
- 用户想快速判断项目是否卡住时
- 审批人想快速理解当前项目背景时
- 协作人想知道自己是否被当前项目阻断时

### 2.3 工作区目标

- 作为签约前主链默认首页
- 为所有角色提供同一个项目上下文
- 将后续各工作区的关键结论收束到同一处
- 减少用户为了判断当前状态而频繁切页

---

## 3. 页面结构草案

建议 `项目总览` 采用“上摘要、中状态、下行动”的三段结构。

### 3.1 顶部摘要区

**目标**: 先让用户知道“这是什么项目、现在大概到哪了”。

**建议包含**:

1. 项目基础识别
   - 项目名称
   - 客户名称
   - 当前销售负责人
   - 当前项目状态标签
2. 当前阶段主标签
   - 线索
   - 立项评估
   - 技术确认
   - 成本估算
   - 招投标 / 商务竞标
   - 报价与毛利评审
   - 签约就绪
3. 当前推进结论
   - 可继续推进
   - 条件推进
   - 当前阻断

### 3.2 中部状态区

**目标**: 解释项目当前为什么处于这个状态。

**建议包含 4 个核心区块**:

1. 阶段进度条
2. 当前下一步动作卡
3. 当前阻断项卡
4. 关键结论卡

### 3.3 底部行动区

**目标**: 帮用户从“看懂”直接进入“去处理”。

**建议包含**:

1. 快速进入待处理工作区
2. 最近关键活动时间线
3. 协作责任人分布

---

## 4. 关键区块详细口径

### 4.1 阶段进度条

**作用**:

把签约前连续工作带显式展示出来，避免用户只看到当前单点状态。

**建议阶段固定为**:

1. 线索登记
2. 立项评估
3. 技术与范围确认
4. 前期成本估算
5. 招投标 / 商务竞标
6. 报价与毛利评审
7. 签约就绪

**状态表达建议**:

- 已完成
- 当前进行中
- 被阻断
- 尚未开始
- 不适用

**关键要求**:

- 用户应能看到不只是“现在在哪”，还要看到“前面哪些已经完成、后面哪些尚未开始”
- 若当前项目不走投标形态，也应保留统一骨架，但将对应节点标记为不适用或已通过商务路径替代

### 4.2 下一步动作卡

**作用**:

把当前最重要的一步明确给用户，避免用户自己判断“应该先去哪”。

**建议字段**:

- 当前下一步动作名称
- 当前责任人
- 建议完成时间
- 动作所属工作区
- 进入处理入口

**例子**:

- 补齐立项评估材料
- 完成技术范围确认
- 形成前期成本清单
- 登记投标结果
- 发起毛利评审

### 4.3 阻断项卡

**作用**:

解释为什么当前项目不能继续推进。

**阻断项分类建议**:

- 材料缺失
- 审批未完成
- 技术结论未形成
- 成本清单未形成
- 投标结果未明确
- 报价未放行

**每条阻断项至少应显示**:

- 阻断原因
- 当前责任人
- 所属阶段
- 建议入口

### 4.4 关键结论卡

**作用**:

把分散在各工作区的关键判断摘要出来，方便快速理解上下文。

**建议最少包含**:

1. 立项结论
2. 技术结论
3. 风险与保留意见摘要
4. 当前成本估算结论
5. 投标 / 商务竞标结论
6. 报价 / 毛利评审结论

**关键要求**:

- 只显示摘要，不显示冗长明细
- 摘要必须能回跳到对应工作区查看完整内容

### 4.5 最近关键活动时间线

**作用**:

让用户能快速看到项目签约前最近发生了什么，而不是四处翻找历史。

**建议包含事件**:

- 立项发起 / 通过 / 条件通过 / 否决
- 技术结论更新
- 成本估算版本更新
- 投标结果登记
- 报价审批结论更新

### 4.6 协作责任卡

**作用**:

让用户看清“当前项目谁在主责、谁在协作、谁在审批”。

**建议包含角色**:

- 销售负责人
- 技术支持 / 售前负责人
- 商务行政责任人
- 财务责任人
- 当前审批责任人

---

## 5. 角色默认视角

### 5.1 销售

默认最先看到：

- 下一步动作卡
- 阻断项卡
- 当前阶段进度条
- 签约就绪风险

### 5.2 副总经理 / 流程负责人

默认最先看到：

- 当前立项/报价审批摘要
- 风险与例外提示
- 当前需要自己处理的事项

### 5.3 技术支持 / 售前

默认最先看到：

- 技术结论摘要
- 风险与保留意见
- 当前是否缺自己提供的输入

### 5.4 商务行政

默认最先看到：

- 投标结果状态
- 签约就绪检查
- 还缺哪些可进入合同的前置项

### 5.5 财务

默认最先看到：

- 成本估算摘要
- 回款 / 税务条件摘要
- 毛利评审状态

---

## 6. 关键体验规则

### 6.1 先解释状态，再给入口

总览页不能只给很多按钮，而必须先解释当前状态。

### 6.2 一屏内必须知道能否推进

用户打开总览后，不应再通过多次切页才能判断项目是否卡住。

### 6.3 阻断项必须能指向责任人

不能只告诉用户“未完成”，还要告诉用户“谁该补、去哪补”。

### 6.4 摘要不等于丢失上下文

总览页只做摘要，但每个摘要都应可追溯到完整工作区。

---

## 7. 与其他工作区的承接关系

`项目总览` 必须承接并指向：

1. `立项与推进`
2. `技术与成本`
3. `招投标 / 商务竞标`
4. `报价与毛利评审`
5. `签约就绪`

用户应从总览出发进入处理，也应从其他工作区返回总览复核整体状态。
