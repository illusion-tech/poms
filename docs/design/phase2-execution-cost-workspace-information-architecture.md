## 1. 文档目标

本文档用于细化第二阶段 `L2-T01`，即“执行期成本归集工作区”的信息架构。

这个工作区的核心不是新增一批零散台账页面，而是把执行中的真实成本事实，围绕同一个 `Project` 持续归集、持续解释、持续承接到后续经营核算。

它重点回答：

- 财务如何围绕项目持续登记实际成本
- 销售、商务行政、管理层如何看到可理解的成本进展
- 哪些执行期成本事实应进入统一工作区，而不是散落在附件、线下表格和备注里
- 执行期成本如何承接签约前估算成本，而不是重新起一条线

---

## 2. 工作区定位

### 2.1 面向用户

- 主用户:
  - 财务
- 高频协作角色:
  - 商务行政
  - 销售 / 项目负责人
- 摘要视角用户:
  - 管理层

### 2.2 使用时机

- 合同生效并进入执行阶段后
- 项目出现采购、费用、发票、必要付款等真实成本事实时
- 财务需要确认成本是否已纳入项目经营口径时
- 销售、项目负责人或管理层需要判断项目执行成本是否偏离预期时

### 2.3 工作区目标

- 让执行期成本围绕 `Project` 归集，而不是按单据孤立登记
- 让“已发生”“已登记”“已确认”“已纳入口径”这些状态被明确区分
- 让执行期成本天然成为后续经营核算、提成判断和项目复盘的可信输入
- 保持第一阶段合同资金域边界，不把当前工作区扩成完整采购或完整财务系统

---

## 3. 与第一阶段基线的承接关系

### 3.1 复用的既有事实

当前第二阶段 `L2` 不是从零开始，而是承接第一阶段已存在的最小事实基线：

- 合同台账
- 回款记录
- 基础成本 / 付款记录
- 发票台账

因此，`L2` 的设计原则不是替换这些对象，而是把它们从“最小可用事实点”提升为“项目级持续成本工作区”的一部分。

### 3.2 不在本阶段展开的方向

本工作区当前不直接展开为：

- 完整采购审批系统
- 完整付款审批系统
- 完整供应商管理系统
- 完整总账 / 凭证体系

这些内容依然保持后置，不挤占当前主线。

### 3.3 与签约前成本估算的关系

签约前的 `技术与成本` 工作区负责形成：

- 估算成本
- 税务成本假设
- 高风险不确定成本项

执行期成本归集工作区负责形成：

- 实际已发生或已确认的成本事实
- 成本状态变化
- 与前期估算的承接基础

因此，两者关系应是：

`估算成本基线 -> 执行期实际成本归集 -> 经营核算 -> 提成与复盘使用`

---

## 4. 工作区需要回答的核心问题

执行期成本归集工作区必须让用户能快速回答以下 5 个问题：

1. 当前项目已累计了哪些实际成本
2. 哪些成本只是登记，哪些已经确认并纳入口径
3. 成本主要来自采购、费用、发票还是必要付款事实
4. 当前累计成本与签约前估算相比是否出现明显偏差
5. 现在还有哪些成本事实缺口，导致经营判断不完整

---

## 5. 工作区顶层结构

建议采用“上总览、中归集、下明细、右解释”的结构。

### 5.1 一级区块建议

1. 成本总览条
2. 成本累计与状态摘要
3. 成本归集入口区
4. 成本记录主列表
5. 成本偏差与风险提示
6. 成本依据与版本侧栏

### 5.2 顶部固定摘要条

建议固定显示：

- 项目标识
- 合同生效状态
- 当前执行阶段
- 估算成本总额
- 当前已归集实际成本
- 当前已确认并纳入口径成本
- 当前成本偏差状态:
  - 正常
  - 关注
  - 偏高
- 当前未收口成本项数量

### 5.3 默认视图原则

- 财务默认进入“成本归集与状态”视图
- 销售 / 项目负责人默认进入“总览与偏差”视图
- 管理层默认进入“累计结果与风险解释”视图

---

## 6. 一级工作区设计

### 6.1 成本总览条

用于在最上层快速显示项目级成本状态。

建议包含：

- 估算成本
- 实际已登记成本
- 实际已确认成本
- 已纳入口径成本
- 当前偏差金额
- 当前偏差比例
- 当前最高风险标签

这个区块服务的不是录入，而是让所有角色先理解当前项目成本处于什么状态。

### 6.2 成本累计与状态摘要

用于把累计结果和状态差异压缩表达。

建议至少按以下状态区分：

- 草稿
- 已登记
- 已确认
- 已纳入口径
- 已作废 / 已替代

建议同时显示：

- 当前累计成本按状态分布
- 最近 30 天新增成本
- 待确认成本金额
- 高风险未收口成本数量

### 6.3 成本归集入口区

这是财务和商务行政最需要的动作入口。

建议统一入口，不按系统模块切散，至少支持：

- 新增采购相关成本记录
- 新增采购发票记录
- 新增费用记录
- 新增必要付款事实
- 关联已有发票 / 付款事实到项目
- 发起确认或纳入口径

关键要求：

- 入口按“我要归集什么成本”组织，而不是按技术模块命名
- 每个入口都必须强制绑定项目上下文

### 6.4 成本记录主列表

这是工作区主体，应作为项目级成本台账总表。

建议主列表至少支持以下列：

- 成本类型
- 成本子类
- 成本说明
- 来源对象 / 依据
- 金额
- 税务成本影响
- 当前状态
- 发生日期
- 登记人
- 确认人
- 是否已纳入口径
- 是否高风险

建议支持的默认筛选：

- 全部
- 待确认
- 已纳入口径
- 高风险
- 最近变更

### 6.5 成本偏差与风险提示

这一块是连接 `L2` 和 `L4` 的关键。

建议固定输出：

- 与估算成本偏差金额
- 与估算成本偏差比例
- 偏差主要来源类别
- 当前高风险成本项
- 仍未收口的成本缺口

注意：

这里先做“提示与解释入口”，不在 `L2-T01` 就直接展开完整经营核算工作区。

### 6.6 成本依据与版本侧栏

用于避免成本记录只剩金额，没有解释基础。

建议显示：

- 对应合同 / 发票 / 付款 / 费用依据
- 相关附件或说明
- 最近状态变更记录
- 是否替代历史记录
- 与签约前成本估算的关联提示

---

## 7. 成本类型与统一口径建议

当前建议执行期成本至少统一分成以下四类：

1. 采购相关成本
2. 发票相关成本
3. 费用成本
4. 必要付款事实

### 7.1 采购相关成本

聚焦：

- 采购合同
- 外协 / 集成 / 第三方采购
- 与交付直接相关的外部采购承诺

### 7.2 发票相关成本

聚焦：

- 与项目成本归集相关的采购发票或成本票据
- 税务成本影响
- 是否已进入当前项目成本口径

### 7.3 费用成本

聚焦：

- 差旅
- 现场实施费用
- 部署 / 运输 / 杂项支出
- 其他需要进入项目经营判断的费用

### 7.4 必要付款事实

聚焦：

- 已实际支付或已形成付款事实的记录
- 与成本纳入口径直接相关的付款确认信息

---

## 8. 状态设计原则

执行期成本归集工作区必须严格区分“事实存在”和“已进入经营口径”。

建议统一语义：

- `草稿`: 仅录入，尚未作为正式依据
- `已登记`: 成本事实已进入系统，但未确认
- `已确认`: 责任角色已确认该事实有效
- `已纳入口径`: 已被纳入项目经营核算口径
- `已作废 / 已替代`: 不再作为当前有效依据，但必须保留历史

设计要求：

- 不允许直接覆盖历史金额
- 状态迁移必须可审计
- 经营核算默认只消费“已纳入口径”的成本

---

## 9. 与其他主线的承接关系

### 9.1 与 `L1` 的承接

需要复用：

- 签约前估算成本总额
- 高风险不确定成本项
- 税务成本假设

### 9.2 与 `L3` 的承接

移交后，执行期成本归集工作区应成为项目执行责任下的正式工作区，而不是独立财务孤岛。

### 9.3 与 `L4` 的承接

本工作区输出给 `L4` 的核心不是原始单据，而是：

- 项目级实际成本累计结果
- 状态清晰的有效成本集合
- 偏差提示和缺口提示

### 9.4 与 `L5` 的承接

虽然提成规则不直接在本工作区处理，但后续提成判断需要消费：

- 已纳入口径成本
- 成本异常或调整链
- 对项目真实毛利的影响

---

## 10. 当前结论

`L2-T01` 的信息架构建议固定为：

- 以 `Project` 为主上下文
- 以项目级成本归集总表为主工作面
- 以状态区分和偏差提示为核心解释方式
- 以采购、发票、费用、必要付款事实为统一入口

因此，下一步应继续执行 `L2-T02`，把“项目级实际成本记录”从信息架构继续细化到对象与记录层草案。
