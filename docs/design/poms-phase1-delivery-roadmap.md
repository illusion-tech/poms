## 1. 文档目标

本文档用于把 `POMS` 第一阶段从“设计已基本收敛”进一步转换为“执行路线明确、阶段目标清楚、近期切片可直接实施”的交付路线图。

本文档重点回答：

- `POMS` 第一阶段到底指什么
- 第一阶段最终要收口成什么状态
- 为达到该终局目标，中期应按什么里程碑推进
- 当前近期应优先做哪些实现切片
- 远期、中期、近期三层内容应如何分别维护

本文档不是需求说明、HLD 或模块详细设计的替代物，而是第一阶段交付层面的统一路线图。

---

## 2. 第一阶段定义

`POMS` 第一阶段不是“先做一批功能试试看”，而是第一个正式基线版本。

第一阶段的统一定义为：

**在不把系统扩张为完整 CRM、ERP 或财务系统的前提下，先把“项目型销售主链路 + 风险审批闸口 + 合同资金基础闭环 + 提成治理基础闭环 + 平台治理基础能力”做成一个可运行、可审计、可联调、可继续扩展的正式版本。**

这一定义至少包含以下特征：

- 以 `Project` 作为第一阶段主对象，贯穿签约前推进与签约后执行前过渡。
- 以显式业务状态机 + 固定审批流实现关键闸口控制，不预设通用工作流引擎为前置条件。
- 合同、回款、发票、成本、提成等关键事实以“业务登记 + 确认生效 + 留痕审计”为第一阶段主口径。
- 平台治理能力必须先形成最小正式能力，包括用户、角色权限、组织和导航。
- 数据结构以 `PostgreSQL + SQL-first migration + MikroORM` 为固定路线。

第一阶段明确不是以下内容：

- 不是完整财务系统，不承诺真实付款、银行流水、ERP、开票系统的强联动闭环。
- 不是完整审批中心，不先建设可配置工作流平台。
- 不先承诺多公司、多账套、多币种和复杂数据范围权限。
- 不把发票、付款、提成、异常调整等所有后续深水区一次性做满，只要求形成第一阶段正式闭环。

## 3. 路线图分层原则

为避免路线图要么过虚、要么过早写死细节，第一阶段路线图固定采用三层维护：

- 远期目标：
  - 表达第一阶段最终要收口成什么状态
  - 只写目标状态、范围边界、完成定义
  - 不写实现细节和排期承诺
- 中期里程碑：
  - 表达为了到达终局目标，需要经过哪些阶段性收口
  - 写清里程碑顺序、依赖关系与完成定义
  - 保持相对稳定，不随日常切片频繁波动
- 近期切片：
  - 表达当前 2-4 周最应该推进的具体实现工作
  - 允许细到 migration、API、DTO、测试、OpenAPI、seed、联调入口
  - 应随实现进展持续回写

统一维护原则：

- 越远期越强调目标与边界
- 越近期越强调实现与验证
- 不允许把远期设想伪装成近期承诺

## 4. 远期目标（第一阶段终局目标）

当 `POMS` 第一阶段被认为“完成”时，至少应达到以下终局状态。

### 4.1 业务终局目标

- 平台治理域已具备最小正式能力：
  - 用户、角色权限、组织单元、导航可作为真实系统入口使用
- 销售流程域已具备最小正式闭环：
  - 从线索登记到签约登记、合同台账建立、项目移交形成受控主链路
- 合同资金域已具备最小正式闭环：
  - 合同草稿、提交审核、审批通过 / 驳回、生效口径、回款确认、发票台账基础状态可追溯
- 提成治理域已具备最小正式闭环：
  - 角色分配、冻结、基础计算、分阶段发放记录、异常调整 / 重算留痕可成立
- 横切支撑域已具备最小正式闭环：
  - 审批记录、统一待办、审计留痕、附件与通知可支撑核心业务动作

### 4.2 工程终局目标

- 关键主表与支撑表均以真实 migration 落地，不依赖 ORM 自动建表作为权威源。
- 核心写侧接口已按普通更新接口 / 命令型接口 / 系统派生接口分层稳定。
- 核心读侧至少具备：
  - 当前用户资料
  - 导航树
  - `Project` 列表 / 详情
  - `Contract` 列表 / 详情
  - 当前用户统一待办列表
- OpenAPI 与共享 contract 产物可稳定生成，供前后端联调使用。
- 核心切片具备自动化验证，不仅覆盖 happy path，也覆盖关键约束失败路径。

### 4.3 前后端终局目标

- `poms-admin` 不再主要依赖模板 / demo 页面承载主业务。
- 管理后台至少能通过真实 API 跑通一条 `Project -> Contract -> Approval/Todo` 主链路。
- 当前用户、导航、待办、对象详情等基础页面已改为真实接口驱动。

### 4.4 第一阶段完成定义

只有同时满足以下条件，才建议认为第一阶段进入收口评估：

1. `Project`、`Contract`、审批 / 待办、平台最小能力均已形成真实实现闭环。
2. 关键业务状态迁移与审批状态迁移已分层稳定，不再混用。
3. 设计文档、OpenAPI、migration、实体、测试和 seed 之间不存在明显漂移。
4. 至少存在一条可演示、可验证、可继续扩展的真实业务链路。

## 5. 中期里程碑

为达到上述终局目标，第一阶段建议按以下中期里程碑推进。

### 5.1 里程碑 A：后端基础设施与最小开发环境稳定

目标：

- 固化 `MikroORM` 基础接入、migration 执行方式、seed 最小样例数据与 OpenAPI 导出流程。

完成定义：

- 新环境可通过 migration + seeder 启动到可调 API 状态。
- OpenAPI 与共享 contract 产物能稳定生成。
- 最小开发用户、组织、导航、样例业务数据可支撑本地联调。

### 5.2 里程碑 B：`Project` 主对象完整切片

目标：

- 把 `Project` 做成第一个真正贯通的业务对象切片。

完成定义：

- 具备稳定的创建、列表、详情、基础信息更新。
- 日期字段、状态约束、唯一约束、失败路径测试已固定。
- 能作为后续 `Contract`、审批和前端页面的上游对象。

### 5.3 里程碑 C：`Contract` 核心链路与生效前支撑

目标：

- 把 `Contract` 从草稿台账推进到提交审核、生效前控制与基础详情查询稳定。

完成定义：

- 合同草稿创建、详情、基础信息更新、提交审核链路成立。
- 合同与 `Project` 的关联、唯一约束与状态约束固定。
- 高敏动作不再混入普通更新接口。

### 5.4 里程碑 D：审批 / 统一待办最小闭环

目标：

- 形成“对象命令触发 -> 审批记录 -> 待办生成 -> 审批处理 -> 待办关闭”的最小横切闭环。

完成定义：

- 至少一个真实业务对象已接入审批闭环。
- 审批通过 / 驳回与业务对象状态迁移一致。
- 同一审批节点不会生成重复未关闭待办。

### 5.5 里程碑 E：平台最小可用 + 前端真实业务化

目标：

- 把前端壳层从模板承载转成真实 API 承载。

完成定义：

- 当前用户资料、导航树、统一待办、`Project`、`Contract` 基础页面已接真实接口。
- 前端至少能跑通一条真实的业务演示链路。

### 5.6 里程碑 F：第一阶段收口评估

目标：

- 在主链路、审批支撑与前后端联调完成后，再评估是否进入提成、经营视图和后续扩展深化。

完成定义：

- 当前阶段的正式范围、明确不做项和残留扩展项已写清。
- 对后续阶段的议题分类明确，不再混回第一阶段主承诺。

## 6. 近期切片（当前优先级）

以下内容按当前仓库成熟度，属于近期应优先推进的实现切片。

### 6.1 近期已完成或已启动

- 后端基础设施已形成：
  - `MikroORM` 基础接入
  - 首批 migration
  - 最小 seed
  - OpenAPI 导出稳定化
- `Project` 已形成首批完整切片基础：
  - 基础读写链路
  - 日期语义测试
  - 唯一约束与状态约束测试
- `Contract` 已形成首批完整切片基础：
  - 基础读写链路
  - 与 `Project` 关联校验
  - 日期语义与唯一约束测试
- 审批 / 统一待办最小支撑已启动：
  - `Contract` 提交审核
  - `ApprovalRecord`
  - `TodoItem`
  - `GET /me/todos`

### 6.2 近期下一批切片

1. 完成 `activateContract`
   - 把 `pending-review -> active` 的命令型动作和状态约束固定下来
   - 明确是否同步生成当前有效合同快照骨架
2. 补齐审批详情与读侧稳定化
   - 稳定审批记录详情输出
   - 收敛统一待办列表字段
3. 把当前用户、导航、待办接入前端壳层
   - 不先铺大量业务页面
   - 先把壳层真实化
4. 建立 `Project` / `Contract` 基础列表与详情页
   - 使用真实 API
   - 不再由 demo 页面充当正式入口

### 6.3 近期暂不提前展开

- 完整付款审批流
- 完整开票申请流
- 外部财务系统或银行流水同步
- 通用审批中心
- 复杂数据范围权限

## 7. 依赖关系

为避免切片乱序，第一阶段建议固定以下依赖关系：

1. 平台最小能力是前端真实化的前提。
2. `Project` 是 `Contract` 与后续审批链路的上游对象。
3. `Contract` 草稿与审核链路稳定后，才适合继续推进合同生效口径。
4. 审批 / 待办必须依附真实业务对象动作，不单独空转实现。
5. 前端真实业务页应建立在真实 API 与稳定 OpenAPI 契约之上，而不是反向逼迫后端先做临时接口。

---

## 8. 路线图维护规则

本路线图后续建议按以下规则维护：

1. 远期目标仅在第一阶段定义或范围边界发生变化时更新。
2. 中期里程碑仅在实施顺序或完成定义发生变化时更新。
3. 近期切片应随着实现进展持续回写，保持其始终可执行。
4. 若实际实现推翻既有高影响结论，不直接静默改本路线图，应同步补 ADR 或评审摘要。
5. 本文档应与 `poms-design-progress.md`、`implementation-delivery-guide.md` 共同维护，而不是相互替代。

---

## 9. 当前结论

`POMS` 第一阶段已经不再缺“为什么做”的设计说明，当前更缺“按什么顺序把它做成”的统一交付路线。

因此，第一阶段路线图的定位应明确为：

- 用远期目标定义阶段终局
- 用中期里程碑控制推进顺序
- 用近期切片指导具体实现

后续若需要继续细化计划，应优先细化“近期切片”和“里程碑完成定义”，而不是重新发散第一阶段的正式范围。

## 10. 近期切片看板

本看板用于把第 6 章“近期切片”进一步转换为可执行入口。维护时不追求大而全，而要求每个切片都能回答以下问题：

- 当前处于什么状态
- 依赖哪些已存在输入
- 本切片交付后要产出什么
- 满足什么条件才算完成
- 下一步最应该推进什么

### 10.1 状态定义

| 状态          | 含义                                                               |
| ------------- | ------------------------------------------------------------------ |
| `Planned`     | 已确认纳入近期范围，但尚未开始落地                                 |
| `In Progress` | 已有代码或文档推进，当前仍在补齐闭环                               |
| `In Review`   | 主要实现已形成，当前在做联调、验证、文档回写或收尾校验             |
| `Done`        | 已满足当前阶段 DoD，后续仅做缺陷修补或纳入下一阶段扩展，不再算近期 |

### 10.2 编号规则

为便于后续持续跟踪，近期执行层统一采用两级编号：

- 切片编号：`P1-Sxx`
  - `P1` 表示 `POMS Phase 1`
  - `S` 表示 `Slice`
  - `xx` 为两位递增编号
- 任务编号：`P1-Txx`
  - `P1` 表示 `POMS Phase 1`
  - `T` 表示 `Task`
  - `xx` 为两位递增编号

使用约定：

- 切片编号用于路线图、进度板、联调纪要和阶段评审。
- 任务编号用于近期 2-4 周内的具体推进项，不要求长期保留不变，但同一轮迭代内应保持稳定。
- 若某切片被拆分为多个独立执行单元，应新增新的 `P1-Sxx`，而不是复用旧编号硬塞多条主线。

### 10.6 看板维护规则

- 看板只维护近期 2-4 周的执行切片，不承载远期设想。
- 每个切片至少维护 `状态 / 主要输出 / DoD / 当前下一步` 四项，不允许只写标题。
- 某切片状态从 `In Progress` 进入 `In Review` 或 `Done` 时，需同步回写实现证据，如测试、OpenAPI、联调或文档。
- 若切片被拆分或合并，应同步更新第 6 章“近期切片”和 `poms-design-progress.md` 的当前阶段判断，避免两个入口漂移。
- 提交记录、联调纪要、评审意见和进度同步时，优先引用 `P1-Sxx` 或 `P1-Txx`，避免只写自然语言标题导致后续难以检索。
