## 1. 文档目标

本文档用于把“统一运行时审计落库与安全事件结构化留痕”从抽象要求收敛为第一阶段可执行的最小设计基线，明确：

- 第一阶段为什么必须纳入该能力
- 第一阶段到底做什么，不做什么
- 最小持久化模型、写入时机和查询出口如何定义
- 哪些平台治理动作和安全事件必须先被覆盖
- 后续实现与验收应如何拆任务、补证据

---

## 2. 当前判断

当前仓库已具备平台治理真实主数据、真实权限过滤、真实路由守卫与浏览器层 smoke，但仍缺统一运行时审计基础设施。

这会导致以下问题：

- 平台高敏动作虽已可执行，但系统内没有统一可查询的结构化审计证据
- 登录失败、无效令牌、权限拒绝、路由访问拒绝等关键安全事件没有统一落库
- 导航治理当前只能依赖 PR / 发布记录作为维护留痕，尚不能满足系统内审计查询要求

因此，第一阶段当前不能仅以“页面已通、接口已通”作为通过口径，必须补上最小统一审计基线。

---

## 3. 第一阶段范围

### 3.1 第一阶段必须覆盖的能力

- 统一 `audit_log` 持久化模型
- 统一 `security_event` 持久化模型
- 命令执行后的结构化审计写入能力
- 关键安全事件的结构化写入能力
- 最小只读查询出口
- 自动化验收证据

### 3.2 第一阶段必覆盖对象

第一阶段最小覆盖对象限定为：

- 平台治理域高敏动作
  - 用户创建、启停、角色分配、组织分配
  - 角色创建、启停、权限分配 / 撤销
  - 组织创建、更新、移动、启停
  - 导航事实源同步 / 发布
- 关键安全事件
  - 登录失败
  - 无效或失效令牌访问
  - 接口权限拒绝
  - 路由权限拒绝或匿名登录回跳

### 3.3 第一阶段明确不做项

- 完整审计中心、复杂聚合检索、导出与报表
- 外部 SIEM / 日志平台对接
- 所有业务域命令的一次性全量补接
- 面向终端用户的安全分析页
- 通用字段级 diff 引擎

说明：

- 第一阶段先把平台治理域与关键安全闸口收敛为统一事实源
- 合同、审批、提成等业务域已有领域级留痕与审批记录，不在本轮强制整体迁移，但后续应逐步接入统一审计基础设施

---

## 4. 核心模型

### 4.1 `audit_log`

用于记录命令型动作或受控同步动作的结构化审计事实。

建议最小字段：

| 字段              | 说明                                                               |
| ----------------- | ------------------------------------------------------------------ |
| `id`              | 主键                                                               |
| `event_type`      | 事件类型，如 `platform.user.deactivated`                           |
| `target_type`     | 对象类型，如 `PlatformUser`、`Role`、`OrgUnit`、`NavigationConfig` |
| `target_id`       | 对象主键或稳定标识                                                 |
| `operator_id`     | 操作者；系统动作可为空                                             |
| `request_id`      | 请求级关联标识                                                     |
| `result`          | `success` / `rejected` / `failed`                                  |
| `reason`          | 人类可读原因或拒绝说明                                             |
| `before_snapshot` | 变更前关键字段快照                                                 |
| `after_snapshot`  | 变更后关键字段快照                                                 |
| `metadata`        | 扩展信息，如权限 key、来源命令、发布批次                           |
| `occurred_at`     | 发生时间                                                           |

约束：

- 采用追加写，不允许普通更新覆盖历史记录
- `before_snapshot` / `after_snapshot` 只保留关键字段，不镜像整对象
- `metadata` 用于补充对象外上下文，不替代主字段语义

### 4.2 `security_event`

用于记录认证、鉴权、访问拒绝等安全事件。

建议最小字段：

| 字段             | 说明                                                        |
| ---------------- | ----------------------------------------------------------- |
| `id`             | 主键                                                        |
| `event_type`     | 事件类型，如 `auth.login.failed`、`authz.permission.denied` |
| `severity`       | `info` / `warning` / `high`                                 |
| `actor_id`       | 当前用户 ID；匿名场景可为空                                 |
| `principal`      | 用户名、邮箱或令牌主体等文本标识                            |
| `request_id`     | 请求级关联标识                                              |
| `path`           | 请求路径或前端目标路由                                      |
| `method`         | HTTP 方法；前端路由事件可为空                               |
| `permission_key` | 触发拒绝的权限 key                                          |
| `result`         | `blocked` / `failed` / `expired`                            |
| `ip`             | 来源 IP                                                     |
| `user_agent`     | 来源 UA                                                     |
| `details`        | 扩展上下文，如 guard 名称、token 原因                       |
| `occurred_at`    | 发生时间                                                    |

约束：

- 同样采用追加写
- 允许匿名事件存在，不要求强制关联平台用户
- 不把敏感令牌原文、密码原文写入明文字段

---

## 5. 写入时机

### 5.1 `audit_log` 写入规则

以下动作成功或被业务规则拒绝时，都应写入 `audit_log`：

- 平台用户创建、停用、角色变更、组织变更
- 角色创建、启停、权限集合变更
- 组织创建、更新、移动、停用
- 导航事实源同步或发布

说明：

- `NavigationItem` 第一阶段仍允许通过受控代码/seed 维护
- 但每次导航事实源同步到运行时后，必须写入 `audit_log`，记录发布批次、操作者、关键差异摘要和结果

### 5.2 `security_event` 写入规则

以下事件必须写入 `security_event`：

- 用户名密码登录失败
- 使用失效或无效令牌访问接口
- 后端权限守卫拒绝访问
- 前端路由守卫拒绝进入平台治理页并触发回跳

说明：

- 前端事件可先通过现有后端接口上报，第一阶段不要求另做专用日志代理
- 后端安全事件以服务端落库为准；前端上报只补充浏览器侧 guard 语义

## 6. 最小查询出口

第一阶段至少提供以下只读查询出口：

- `GET /audit-logs`
- `GET /security-events`

建议最小筛选：

- 时间区间
- `eventType`
- `targetType`
- `targetId`
- `operatorId` 或 `actorId`
- `result`

建议最小返回字段：

- 事件类型
- 对象类型与对象标识
- 操作者 / 触发主体
- 结果
- 发生时间
- 摘要字段

说明：

- 第一阶段不要求复杂导出、全文检索、图表分析
- 只读管理页可后补，但至少要有稳定 API 和查询视图边界

---

## 7. 与现有设计的对齐

### 7.1 与表结构冻结设计对齐

- 继续沿用 `audit_log` 作为统一审计主表命名
- 新增 `security_event` 作为安全事件专用支撑表

### 7.2 与查询视图边界对齐

- 保留 `AuditEventListView`
- 新增 `SecurityEventListView`

### 7.3 与命令设计对齐

- `audit_log` 写入视为命令执行副作用
- 安全拒绝事件写入视为认证 / 鉴权中间件或守卫副作用

---

## 8. 第一阶段任务拆分建议

为避免 `P1-T26` 再次停留在抽象描述，建议按以下任务推进：

1. `P1-T26`：冻结最小设计基线并同步路线图、缺口矩阵、验收清单
2. `P1-T27`：实现 `audit_log` / `security_event` 持久化模型、migration 与基础写入服务
3. `P1-T28`：接入平台治理高敏命令与关键安全事件
4. `P1-T29`：补齐查询 API、最小管理端读侧与 e2e / smoke 验收证据

---

## 9. 第一阶段验收要求

第一阶段至少应证明：

- `audit_log` 与 `security_event` 已真实落库
- 平台治理高敏动作可查询到结构化审计记录
- 登录失败、权限拒绝、无效令牌访问可查询到结构化安全事件
- 导航事实源同步动作不再只依赖 PR / 发布记录作为唯一留痕
- 读侧查询出口可按时间与对象检索
- 自动化测试已覆盖至少一条成功审计写入和一条安全拒绝写入链路

---

## 10. 当前结论

统一运行时审计落库与安全事件结构化留痕应纳入第一阶段硬门槛，但必须按“最小基线”而非“完整审计中心”实施。

在 `P1-T27` ~ `P1-T29` 完成前，第一阶段仍不应被标记为正式通过。
