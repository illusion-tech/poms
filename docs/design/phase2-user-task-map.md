## 1. 文档目标

本文档用于在第一阶段正式收口后，把第二阶段讨论的出发点固定为“真实用户如何完成真实任务”，而不是直接把讨论拉成新的功能清单或技术专题。

本文档重点回答以下问题：

- `POMS` 里真正的业务主用户是谁
- 这些用户在项目全生命周期中承担什么职责
- 第一阶段成果放回真实任务链后，系统已经支撑到了哪里
- 当前哪些断点主要是“体验断裂”，哪些本质上是“能力未形成”
- 第二阶段后续的体验优化路线图应围绕哪些主任务链展开

本文档不是第二阶段开发排期，也不是单纯的页面改版清单，而是第二阶段体验优化规划的前置分析输入。

---

## 2. 当前讨论边界

本轮第二阶段讨论的目标不是立即扩张新功能范围，而是先从用户任务角度全面复盘第一阶段成果。

当前统一边界如下：

- 先识别真实用户、真实任务与真实协作关系
- 先审视第一阶段成果在真实任务链中的承接情况
- 先识别生命周期中的关键体验断点
- 在上述分析完成前，不直接把第二阶段收敛为开发任务列表

因此，第二阶段当前的正确提问方式不是“先做哪些模块”，而是：

- 用户是谁
- 他们在做什么
- 当前系统在哪些节点让人困惑、低效或容易出错
- 哪些断点最值得优先优化

## 3. 用户分层

### 3.1 业务主用户

- 销售人员
- 副总经理 / 流程负责人
- 技术支持 / 售前
- 商务行政
- 财务人员
- 公司高层

### 3.2 平台支撑用户

- 平台管理员

### 3.3 当前主设计中心

按制度与业务现场判断，第二阶段体验优化当前优先围绕以下 5 类业务主用户展开：

- 销售人员
- 副总经理 / 流程负责人
- 技术支持 / 售前
- 商务行政
- 财务人员

公司高层属于低频高影响用户，平台管理员属于系统支撑用户；两者重要，但不应成为当前体验优化的第一设计中心。

## 4. 关键用户画像

### 4.1 销售人员

- 制度定位：项目第一责任人，对从线索登记、立项、技术确认配合、投标推进、报价、签约到回款跟踪全过程负责
- 用户特点：高频使用、时间碎片化、同时推进多个项目、强目标导向
- 核心诉求：快速推进项目、明确下一步、减少重复录入、清楚知道卡点和责任边界
- 主要风险：绕流程、遗漏材料、隐含承诺、风险披露不足
- 最关注的体验点：
  - 当前项目走到哪一步
  - 为什么不能继续推进
  - 还缺什么材料
  - 谁该接下一棒
  - 当前回款、风险与提成条件处于什么状态

### 4.2 副总经理 / 流程负责人

- 制度定位：立项、毛利、流程合规性的关键闸口责任人
- 用户特点：低频但高影响，不会长时间停留在系统里
- 核心诉求：用最短时间看清是否值得放行、哪里有风险、谁该补什么
- 主要风险：信息过载、关键风险被埋、审批只剩动作没有判断依据
- 最关注的体验点：
  - 决策摘要是否清楚
  - 毛利、回款、风险、范围边界是否一眼可见
  - 条件通过或否决后整改要求是否清楚
  - 自己的审批结果与项目后续经营结果是否可回看

### 4.3 技术支持 / 售前

- 制度定位：负责技术可行性评估、交付范围确认、主要技术风险判断，并参与形成项目初期成本估算
- 用户特点：任务重、判断责任大，关注边界、复杂度、风险和成本依据
- 核心诉求：输入上下文完整、范围和风险能结构化表达、成本估算可解释且可迭代
- 主要风险：需求背景不完整、范围不清、成本估算只有结论没有清单、前序判断在后续阶段丢失
- 最关注的体验点：
  - 当前项目到底要自己给出什么判断
  - 范围、风险、成本是否能形成一套统一结论
  - 自己的否决、保留意见和风险提示是否真的影响流程

### 4.4 商务行政

- 制度定位：负责合同、流程文件、项目移交的组织与记录，确保关键流程资料完整合规
- 用户特点：高准确性导向，重完整性、规范性、时点一致性
- 核心诉求：资料齐全、字段规范、流程节点明确、关键材料可回看
- 主要风险：前序阶段信息散乱、签约与移交流程口径不一致、文档与结构化信息脱节
- 最关注的体验点：
  - 当前进入签约或移交还缺什么
  - 哪些材料是必需的
  - 哪些内容可直接继承前序阶段
  - 移交是否真正形成收口节点

### 4.5 财务人员

- 制度定位：负责回款条件、现金流、税务风险审核，并承担回款确认、成本口径、提成计算与发放审核职责
- 用户特点：高审慎、强口径一致性意识，关注数据能否成为正式依据
- 核心诉求：金额准确、口径统一、依据完整、历史可追溯、估算与实际能对比
- 主要风险：合同、回款、成本、提成之间数据脱节，成本事实没有形成项目级归集
- 最关注的体验点：
  - 当前项目有哪些真实资金条件和成本事实
  - 哪些数据是草稿，哪些已确认生效
  - 实际成本偏离前期估算多少
  - 当前项目真实毛利和提成结果是否可信

## 5. 项目全生命周期任务总图（增强版）

### 5.1 状态口径

- `已形成主链`：第一阶段已能作为正式流程主入口使用
- `基础可用`：对象和能力已有，但还不是完整用户任务链
- `部分支撑`：有局部能力，没形成连续体验
- `尚未形成`：制度要求明确，但系统还没有真正承接

### 5.2 生命周期任务总图

| 生命周期阶段                 | 主用户                     | 关键任务                                         | 第一阶段当前支撑 | 当前疑似体验断点                                               |
| ---------------------------- | -------------------------- | ------------------------------------------------ | ---------------- | -------------------------------------------------------------- |
| 线索登记                     | 销售                       | 登记机会、补基础信息、作为后续起点               | `尚未形成`       | 缺正式 `Lead` 主入口，销售签约前起点不清楚                     |
| 立项评估                     | 销售 / 副总经理            | 发起评估、条件立项、否决                         | `尚未形成`       | 缺结构化评估与审批工作区，制度里的第一道闸口没有形成系统主线   |
| 技术可行性与范围确认         | 技术支持 / 售前            | 明确范围、排除项、风险、可行性                   | `尚未形成`       | 技术判断没有正式承接位置，范围和风险难被后续复用               |
| 前期成本估算                 | 技术支持 / 售前            | 形成必要成本清单与估算依据                       | `尚未形成`       | 没有签约前成本估算主能力，报价和毛利判断难以建立在可追溯基础上 |
| 招投标 / 商务竞标            | 销售主导，多角色协作       | 投标决策、分工、标书/方案/报价、递交、澄清、结果 | `尚未形成`       | `BidProcess` 没成为项目主链工作区，多形态投标缺统一承接        |
| 报价与毛利评审               | 销售 / 财务 / 副总经理     | 报价、成本、回款条件、毛利审批                   | `尚未形成`       | 看不到“报价-成本-回款-毛利-风险”的整体关系                     |
| 合同签约                     | 销售 / 商务行政 / 财务     | 合同登记、审核、激活                             | `已形成主链`     | 合同页和签约前信息复用不足，仍像从中途重新起一条线             |
| 项目移交                     | 商务行政 / 销售 / 技术支持 | 交接边界、负责人、提成冻结                       | `部分支撑`       | 制度上很关键，但系统里还没形成强收口节点和移交工作区           |
| 执行期实际成本归集           | 财务                       | 采购合同、采购发票、费用、付款事实               | `基础可用`       | 目前只有最小事实模型，没形成项目级成本主线和归集体验           |
| 回款确认                     | 财务                       | 登记并确认实际到账                               | `已形成主链`     | 财务事实已在，但和项目经营视角、提成视角连接还不够顺           |
| 项目经营核算                 | 财务 / 销售 / 管理层       | 看估算成本、实际成本、毛利偏差                   | `尚未形成`       | 缺统一项目经营核算视图，用户只能手工拼接判断                   |
| 提成角色与权重冻结           | 销售 / 管理层 / 商务行政   | 在移交前确认提成角色与权重                       | `基础可用`       | 对象基础已在，但缺和移交强绑定的完整操作体验                   |
| 第一阶段提成发放             | 财务 / 管理层              | 启动条件判断、20/25/30 或 10/15 档位             | `基础可用`       | 制度规则复杂，但系统体验未必做到“为什么能发/不能发”一眼看懂    |
| 50% 解锁                     | 财务 / 管理层              | 低首付款项目约束解除                             | `部分支撑`       | 制度规则明确，但用户任务路径和判断提示还不成型                 |
| 第二阶段提成发放             | 财务 / 管理层 / 销售       | 70/75/80 档位、验收与 80% 回款联动               | `部分支撑`       | `AcceptanceRecord` 与提成阶段判断没有形成清晰主线体验          |
| 最终提成发放                 | 财务 / 管理层              | 非质保 100% 回款后的剩余发放                     | `部分支撑`       | 最终发放、质保金、尾款条件等规则还不够可视化                   |
| 质保金提成处理               | 财务                       | 质保期、质保金到账、单独结算                     | `部分支撑`       | 制度里的重要特例还没形成显式操作链                             |
| 异常、退款、坏账、违规、离职 | 财务 / 管理层              | 暂停、扣回、冲销、重算、责任认定                 | `基础可用`       | 后台能力有基础，但复杂规则对操作者未必足够直观                 |
| 项目关闭 / 复盘              | 销售 / 管理层 / 财务       | 关闭、看偏差、看经营结果                         | `尚未形成`       | 缺把全过程信息收束成复盘视图的正式出口                         |

## 6. 当前已识别的关键体验断点

基于当前讨论，第一阶段成果放回真实用户任务链后，至少存在以下 6 个高价值断点：

1. 缺“签约前统一工作区”
   - 线索、立项、范围、成本估算、投标、报价审批没有形成连续项目主链
2. 缺“前期成本估算”正式能力
   - 直接影响投标、报价、毛利判断和后续偏差分析
3. 缺“项目移交”强节点
   - 制度要求重，但系统体验还没把它做成真正的收口阶段
4. 缺“执行期成本归集主线”
   - 当前更像零散财务事实，不像项目经营成本工作台
5. 缺“项目经营核算视图”
   - 估算成本、实际成本、回款、毛利、提成都没形成统一经营视图
6. 提成治理“有基础能力但体验不制度化”
   - 首付款分档、50% 解锁、第二阶段发放、质保金、异常调整等规则尚未形成用户友好的连续操作链

## 7. 第二阶段当前建议的主体验主线

在更细的优先级矩阵形成前，第二阶段体验优化可先按以下 4 条主线理解：

1. 签约前主线
   - 线索、立项、范围确认、前期成本估算、招投标、报价与毛利评审
2. 签约与移交主线
   - 合同、项目移交、责任边界、提成冻结
3. 执行与经营主线
   - 实际成本归集、回款确认、项目经营核算
4. 提成治理主线
   - 角色冻结、分阶段发放、解锁规则、质保金、异常调整、最终结算
