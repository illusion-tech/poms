## 1. 文档目标

本文档用于把四轮正式审阅发现的 22 个问题，从“已记录”推进到“已形成可执行的综合判断”。

本文档重点回答以下问题：

- 22 个问题分别值不值得优先处理
- 哪些问题会直接阻断第二阶段进入最终统一开发判断
- 哪些问题虽重要，但可进入受控 follow-up
- 哪些问题已因最新设计决策而降级或转入规则细化

本文档不是替代 `phase2-review-checklist.md` 的执行入口，而是作为正式审阅问题的综合评估结果。

## 2. 评估维度与口径

本轮综合评估分为两层：

- 第一层是“主评估维度”，用于回答问题值不值得优先处理
- 第二层是“实施型补充维度”，用于回答问题之间能否并行、是否可隔离、应先收口什么

### 2.1 主评估维度

主评估维度统一采用以下 6 个维度：

1. 业务价值
   修复后对制度正确性、业务闭环、协作效率和经营可信性的提升程度。

2. 技术复杂度
   对对象模型、版本 / 快照、查询视图、权限口径和上下游读写链路的改动复杂度。

3. 紧迫性
   对最终统一开发判断的阻断程度，以及是否必须在进入开发前先收口。

4. 影响面
   波及的主线、角色、对象与跨文档范围。

5. 返工风险
   若带着当前缺口进入实现，后续回改模型、规则或读侧的概率与成本。

6. 当前优先级
   统一采用以下标记：

- `P0`：进入最终统一开发判断前必须收口，属于实现前硬阻断
- `P1`：高优先级 follow-up，原则上应进入首批细化 / 实现范围
- `P2`：可受控后置，不阻断首批实现，但需明确范围限制或后续 owner
- `P3`：当前阶段非核心范围或纯表达增强，不应抢占主线资源

### 2.2 实施型补充维度

除上述 6 个主评估维度外，当前还应补充以下 5 个实施型维度：

1. 问题依赖关系
   判断某个问题是否依赖其他问题先收口，或是否本身是后续多个问题的前置。

2. 决策成熟度
   判断该问题当前处于“业务政策未定”“方向已收口待细化”“规则待回写”还是“可直接进入实现”。

3. 范围可隔离性
   判断该问题能否通过明确范围限制、试点条件或已知限制安全后置。

4. 验证与验收成本
   判断该问题是否需要跨时间点、跨角色、跨版本或跨合同进行复杂验证，而不仅是写完规则即可。

5. 数据回填 / 历史兼容冲击
   判断该问题是否会影响历史数据解释、既有台账迁移、快照重算或旧记录回补。

这些补充维度不直接替代主评估表，但必须进入实施排序判断。

### 2.3 观察型补充维度

除上述内容外，以下两个维度建议作为实现排期讨论时的观察项：

- 发布原子性：是否必须跨 `L3/L4/L5` 同步落地，还是可拆分上线
- 组织协同成本：是否需要财务、商务行政、销售、管理层共同拍板后才能冻结规则

补充说明：

- 本轮在原有 7 个正式审阅维度之外，额外加入“交易链条完整性 / 可追溯性”和“返工不可逆性”作为综合评估参考。
- 2026-04-01 已明确：多币种暂列未来扩展；一个 `Project` 必须支持多份同时履约的有效销售合同；项目级交易链条最小覆盖范围包括销售合同、应收计划、回款、销项发票、成本、付款、进项发票。

## 3. 当前总判断

基于最新已确认的设计决策，22 个问题当前可分为：

- `P0`：11 项
- `P1`：7 项
- `P2`：3 项
- `P3`：1 项

当前最重要的变化有三点：

1. `R4-005` 已不再是“是否支持多合同”的开放问题，而是“多合同项目如何在 `L3/L4/L5` 落成稳定规则”的收口问题。
2. `R1-001` 已明确降为未来扩展，不再占用当前阶段核心优先级。
3. 交易链条完整性已被正式识别为进入实现前必须核对的新判断维度，因此涉及合同、应收、回款、成本、付款和发票链路的问题整体优先级上升。

## 6. 关键依赖链与实施排序判断

仅靠主评估表还不足以决定“先做什么”，当前至少应识别以下依赖链。

### 6.1 合同与提成主链

建议按以下顺序理解：

`R4-005 -> R2-001 -> R4-001 -> R1-003 -> R4-002 -> R4-003`

判断理由：

- 不先收口多合同项目主假设，就无法稳定定义项目级合同汇总、回款比例与提成 gate
- 不先收口 `签约就绪 -> ContractTermSnapshot / ReceivablePlan`，合同放行与应收计划都缺正式承接对象
- 不先收口放行基线与合同草案差异校验，签后变更与变更基线都会建立在漂移事实之上
- 最终 `R4-003` 的 `L4 -> L5` 绑定矩阵只能建立在上述合同和变更口径稳定后

### 6.2 成本可信源主链

建议按以下顺序理解：

`R3-002 -> R1-002 -> R2-002 -> R1-004 -> R2-003 -> R3-004 -> R4-003`

判断理由：

- 内部成本率治理不定，人力成本就没有可信基础
- 分摊成本规则不定，共享成本无法进入项目级可信口径
- 阶段归属不定，历史回看、毛利率解释和提成 gate 都会漂移
- 税务影响、`as-of` 时点和公式边界行为必须建立在稳定的成本可信源之上

### 6.3 权限与审批可见性主链

建议按以下顺序理解：

`R1-006 -> R2-005 -> R4-004 -> R3-005`

判断理由：

- 不先明确敏感经营数据边界，例外查看与审批摘要都无法安全设计
- 不先明确例外查看和短时揭示机制，审批页就只能在“信息不够”和“越权暴露”之间摇摆
- 不先稳定审批字段包，通知与协作触发也无法判断哪些内容可推送、哪些必须脱敏

### 6.4 可并行与可后置判断

当前可并行推进的典型项：

- `R3-001`：可与 `L1` 的主链细化并行推进
- `R2-005` 与 `R4-004`：在 `R1-006` 边界稳定后可并行细化
- `R3-006`：在 `R2-004` 和 `R4-003` 基本稳定后再进入提成争议路径设计

当前可通过范围限制后置的典型项：

### 6.5 建议实施排序

   `R4-005`、`R2-001`、`R4-001`、`R2-004`、`R3-002`、`R1-006`

   `R1-001`、`R3-003`、`R1-005`、`R3-005`

当前判断：

- 第一批和第二批不应倒置
- 若第一批未收口，不应继续推进全主线实现设计
