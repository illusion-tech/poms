## 1. 文档目标

本文档用于把第一阶段“缺口补齐计划”进一步转换为可执行、可核验、可签收的验收清单。

本文档重点回答：

- 第一阶段补齐后，具体检查什么
- 每个检查项如何判定通过或不通过
- 哪些项属于硬门槛，缺任一项都不能宣布通过
- 验收时需要查看哪些代码、接口、测试与文档证据
- 收口评估应如何避免再次出现“口头通过、事实未完工”

本文档不是实现计划，也不是任务分解文档，而是第一阶段重新收口时的正式验收入口。

---

## 2. 验收总原则

第一阶段重新验收时，遵循以下原则：

1. **以原始承诺为准，不以后续口头裁剪为准**。
2. **以真实实现与验证结果为准，不以设计意图或页面占位为准**。
3. **以真实 PostgreSQL、真实 API、真实前端接入为准，不以 fixture 探活为准**。
4. **以自动化验证和文档回写为准，不以人工口头说明为准**。
5. **平台治理域与提成治理域任一未达标，第一阶段均不通过**。

---

## 3. 通过门槛总表

| 验收项 | 类别                   | 是否硬门槛 | 通过标准                                                     |
| ------ | ---------------------- | ---------- | ------------------------------------------------------------ |
| A      | 平台治理域主数据真实化 | 是         | `User`、`Role`、`OrgUnit` 与关系模型真实落地                 |
| B      | 平台导航治理闭环       | 是         | 导航与真实权限、真实路由、真实页面落点形成闭环               |
| C      | 提成规则与角色分配     | 是         | `CommissionRuleVersion`、`CommissionRoleAssignment` 真实落地 |
| D      | 提成计算与发放闭环     | 是         | `CommissionCalculation`、`CommissionPayout` 真实落地         |
| E      | 提成异常调整与重算闭环 | 是         | `CommissionAdjustment` 与重算替代链真实落地                  |
| F      | 自动化验证             | 是         | 核心成功路径与关键失败路径可重复验证                         |
| G      | 文档回写一致性         | 是         | 设计、接口、表结构、实现状态无明显漂移                       |
| H      | 前后端联调可演示性     | 是         | 平台治理与提成治理均能通过真实页面或真实接口完整演示         |

说明：

- 任意一个硬门槛未满足，均不得判定第一阶段通过。
- 不允许用“技术债留到第二阶段”替代本清单中的硬门槛。

## 4. 验收清单

### A. 平台治理域主数据真实化

#### A1. 用户主数据真实落地

- [ ] 已存在 `platform_user` 真实持久化模型
- [ ] 已存在真实 migration、entity / mapping、repository 与 service
- [ ] 登录后 `profile` 数据来自真实主数据，而非 `dev-platform.fixtures.ts`
- [ ] 用户列表、详情、创建、更新、启停接口已实现
- [ ] 用户启停后，下次请求授权结果能立即收敛

验收证据：

- migration 文件
- 后端模块与接口实现
- OpenAPI / client 产物
- 自动化测试
- 真实 API 探活记录

#### A2. 角色与权限真实落地

- [ ] 已存在 `role`、`role_permission_assignment` 真实持久化模型
- [ ] 角色列表、详情、创建、更新、启停接口已实现
- [ ] 角色权限绑定接口已实现
- [ ] 权限字典来源、只读治理口径与角色关系口径一致
- [ ] 用户有效权限来自真实角色关系计算，而非开发期硬编码

验收证据：

- 角色 / 权限相关 migration 与代码
- 角色管理接口探活记录
- 权限变更后导航 / 接口授权收敛验证

#### A3. 组织单元真实落地

- [ ] 已存在 `org_unit`、`user_org_membership` 真实持久化模型
- [ ] 组织树查询、详情、创建、更新、启停、移动接口已实现
- [ ] 用户主责组织与附属组织关系来自真实关系表
- [ ] 组织停用级联规则已生效
- [ ] 停用组织不可再作为用户可选组织

验收证据：

- 组织树接口探活记录
- 组织移动 / 停用 / 用户挂靠测试
- 前端组织查询接入记录

- [ ] `/platform/navigation` 已有真实页面或受控治理入口

重新发起第一阶段收口评估时，结论必须按以下模板记录：

### 5.1 完成项

### 5.2 未完成项

- 仍未补齐的测试：

### 5.3 明确不做项

- 本阶段原本就不纳入的内容：
- 仍保留到后续阶段的扩展项：

### 5.4 通过判断

- [ ] 通过
- [ ] 不通过

若“不通过”，必须明确是哪一条硬门槛未满足，且不得用模糊描述替代。

---

## 6. 当前结论

在本清单全部满足前，`POMS` 第一阶段都不应被标记为“已正式收口”或“已通过验收”。

只有当平台治理域与提成治理域的真实实现、真实联调、自动化验证和文档回写全部满足要求后，才建议将第一阶段状态从“待补齐”更新为“通过”。
