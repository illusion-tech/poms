## 1. 文档目标

本文档用于细化签约前统一工作区中的 `招投标 / 商务竞标`。

这个工作区的重点不是单独做一个“投标资料库”，而是把多形态投标和商务竞标都挂在同一个 `Project` 主链下，形成统一、连续、可追溯的协作工作区。

本文档重点回答：

- 投标 / 商务竞标在项目里如何被统一承接
- 多种竞标形态如何共享同一骨架
- 谁负责什么、缺什么、下一步是什么
- 结果如何自然进入报价评审与签约就绪

---

## 2. 工作区定位

### 2.1 面向用户

- 销售
- 技术支持 / 售前
- 商务行政
- 副总经理 / 流程负责人
- 财务

### 2.2 使用时机

- 项目已经过立项、技术判断和前期成本估算
- 项目进入正式投标、邀标、比选、商务竞标或竞争性谈判阶段
- 项目需要对多轮递交、澄清、述标、结果进行受控记录

### 2.3 工作区目标

- 把投标决策、任务分工、过程版本和结果登记收成一个连续协作空间
- 让不同形态共享统一骨架，而不是为每种形态拆成不同入口
- 让结果能直接进入报价与毛利评审、签约就绪和后续生命周期分流

---

## 3. 工作区顶层结构

建议 `招投标 / 商务竞标` 工作区采用“上状态、中协作、下时间线”的结构。

### 3.1 主区块建议

1. 当前竞标形态与阶段
2. 投标 / 竞标决策
3. 材料清单与责任分工
4. 方案 / 标书 / 递交版本
5. 澄清 / 答疑 / 述标 / 谈判时间线
6. 当前结果与后续流转

### 3.2 顶部固定摘要条

建议顶部固定显示：

- 当前竞标形态
- 当前竞标阶段
- 当前责任人
- 当前是否存在阻断项
- 当前是否已具备进入报价评审或签约就绪的条件

---

## 4. 多形态统一骨架

### 4.1 统一骨架节点

无论是公开招标、邀标、比选还是商务竞标，统一骨架建议都包含：

1. 是否参与的决策
2. 材料清单与责任分工
3. 技术 / 商务材料准备
4. 正式递交或响应
5. 澄清 / 答疑 / 述标 / 谈判
6. 结果登记
7. 后续分流

### 4.2 可变节点

不同形态通过节点显隐和字段差异体现，不另起一套工作区。

**示例**:

- 公开招标：
  - 更强调招标文件、正式递交、澄清答疑、开标 / 定标
- 邀标：
  - 更强调受邀确认、响应版本、反馈结果
- 比选：
  - 更强调多轮比较、演示、候选排序
- 商务竞标 / 竞争性谈判：
  - 更强调多轮商务条件调整、谈判结论、最终确认

### 4.3 关键规则

- 用户先进入同一个工作区，再由系统按形态展示差异区块
- 不允许每种形态都形成独立菜单或独立心智模型

---

## 5. 关键区块详细口径

### 5.1 当前竞标形态与阶段

**作用**:

先回答“现在是在做什么类型的竞标，以及走到哪一步了”。

**建议包含**:

- 竞标形态：
  - 公开招标
  - 邀标
  - 比选
  - 商务竞标 / 竞争性谈判
  - 混合型
- 当前阶段：
  - 决策中
  - 材料准备中
  - 已递交 / 已响应
  - 澄清 / 述标 / 谈判中
  - 已出结果

### 5.2 投标 / 竞标决策

**作用**:

显式记录“为什么参与 / 为什么不参与 / 为什么继续”。

**建议包含**:

- 是否参与
- 决策理由
- 当前目标
- 关键风险提醒

### 5.3 材料清单与责任分工

**作用**:

解决多角色协作时“缺什么、谁负责、做到哪了”不透明的问题。

**建议包含**:

- 材料名称
- 材料类型
- 当前责任人
- 当前状态
- 截止时间

### 5.4 方案 / 标书 / 递交版本

**作用**:

把过程中的正式输出物按版本管理，而不是靠附件覆盖。

**建议至少表达**:

- 版本号
- 版本说明
- 更新时间
- 更新人
- 当前是否为正式递交版本

### 5.5 澄清 / 答疑 / 述标 / 谈判时间线

**作用**:

把投标过程中的往返事件收成连续时间线，便于回看和解释。

**建议包含事件**:

- 澄清问题
- 答疑回复
- 述标安排
- 谈判轮次
- 条件变更

### 5.6 当前结果与后续流转

**作用**:

明确本次竞标最终结果，并决定项目下一步去向。

**建议结果类型**:

- 中标 / 赢单
- 失标
- 弃标
- 废标
- 商务继续推进

**后续流转建议**:

- 中标 / 商务继续推进：
  - 进入报价与毛利评审或签约就绪
- 失标 / 弃标 / 废标：
  - 进入关闭或继续跟进判断

## 6. 角色视角差异

### 6.1 销售

默认重点：

- 当前竞标阶段
- 材料缺口
- 当前责任人
- 结果与下一步去向

### 6.2 技术支持 / 售前

默认重点：

- 技术材料版本
- 澄清 / 述标事件
- 因竞标过程产生的范围变化

### 6.3 商务行政

默认重点：

- 材料清单完成度
- 正式递交版本
- 结果登记与后续进入签约的关键信息

### 6.4 副总经理 / 流程负责人

默认重点：

- 参与决策理由
- 当前风险摘要
- 结果结论

### 6.5 财务

默认重点：

- 报价相关材料是否齐备
- 税务成本和财务条件是否已进入竞标上下文
- 是否已具备进入毛利评审的前置条件

---

## 7. 关键体验规则

### 7.1 多形态共享一套主骨架

不能让不同竞标形态把系统体验拆散。

### 7.2 过程要能解释，不只记录结果

澄清、答疑、谈判、版本变化都属于正式过程，不能只留下最终一条结果。

### 7.3 材料必须和责任绑定

不能只看到“还有材料缺失”，必须知道谁负责补、缺什么、何时要补齐。

### 7.4 结果必须自动推动后续阶段

工作区不能停在“已经中标/失标”的静态状态，而要推动进入后续评审、签约就绪或关闭判断。

---

## 8. 与其他工作区的承接关系

`招投标 / 商务竞标` 必须至少承接并输出以下关系：

1. 输入来自：
   - `技术与成本`
   - `项目总览`
   - `立项与推进`
2. 输出给：
   - `报价与毛利评审`
   - `签约就绪`
   - `项目总览` 摘要区
