## 1. 文档目标

本文档用于明确第二阶段第一批硬前置专题范围，不再把所有 `P0 / P1` 问题混成一个“大待办池”。

本文档重点回答以下问题：

- 第一批到底纳入哪些专题
- 第一批的目标是“冻结什么”，而不是“把所有细节一次写完”
- 哪些内容虽重要，但不属于第一批
- 第一批完成到什么程度，才可认为具备继续支撑全主线实现设计下钻的条件

---

## 2. 第一批范围原则

第一批范围统一采用以下四条原则：

1. 必须直接决定主对象与主事实是否稳定。
2. 必须直接支撑多合同项目和交易链条最小覆盖范围。
3. 必须能通过跨文档、跨角色和跨阶段的一致性校验。
4. 不把未来扩展项和纯展示增强项混入第一批。

## 5. 第一批入场与出场条件

### 5.1 入场条件

第一批开始前应满足：

1. `phase2-review-comprehensive-assessment.md`、`phase2-review-follow-up-plan.md` 与本文件口径一致。
2. 多合同、交易链条最小覆盖范围和回款判断模式冻结三项业务前提不再反复摇摆。
3. 第一批六个专题的主回写文档已明确。
4. 第一批实现映射入口已明确，并可继续回写到接口、查询与表结构设计。

### 5.2 出场条件

第一批完成后至少应满足：

1. 六个专题均已形成正式规则回写并完成交叉复核。
2. `phase2-review-checklist.md` 已同步综合状态与批次判断。
3. `poms-design-progress.md` 已同步真实阶段状态。
4. 不再存在阻断第一批继续作为实现设计输入的方向级未决策问题。
5. `phase2-first-batch-implementation-mapping.md` 中的关键补点已完成“是否足以继续支撑全主线实现设计”的判断。

---

## 6. 当前结论

第二阶段当前不是“所有 `P0 / P1` 一起推进”，而是应先把第一批六个前置专题写稳。

只有这样，后续第二批经营与成本可信源专题才不会继续建立在摇摆的主事实之上；而第四批范围限制项也不会再次混入主线，拖慢全主线实现设计收口的节奏。
