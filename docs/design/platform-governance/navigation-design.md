## 1. 文档目标

本文档用于在平台治理域总设计基础上，进一步明确第一阶段导航菜单模块的契约模型、节点语义、权限过滤、前端适配、路由对照和有限维护边界，为后端导航 SSOT、前端菜单渲染和后续导航管理能力提供稳定输入。

本文档重点回答：

- `NavigationItem` 第一阶段如何正式建模
- 导航节点类型和字段语义如何约束
- 导航与前端真实路由如何建立稳定映射
- 导航权限匹配、隐藏、禁用之间的优先级是什么
- 当前 `poms-admin` 模板如何适配后端导航树

---

## 2. 设计范围

### 2.1 第一阶段纳入范围

- 后端导航树单一事实源
- `NavigationItem` 契约与字段约束
- 导航节点类型与层级规则
- 权限过滤与返回规则
- `poms-admin` 菜单适配规则
- 导航与真实路由对照治理
- 有限维护能力与审计要求

### 2.2 第一阶段不展开的内容

- 任意后台新增未知路由导航
- 完整导航可视化编排器
- 多端导航差异化策略
- 动态个性化导航推荐
- 导航国际化与多语言治理

---

## 3. 上游约束

本设计继承以下已固定结论：

- 后端导航为单一可信源，前端忠实渲染
- 导航可见性与接口鉴权共享同一权限事实源，但语义不能混淆
- 导航链接必须与前端真实路由保持一致
- 当前前端静态 fallback 菜单只允许作为迁移期机制
- 第一阶段导航配置能力以有限维护为主，不开放未知路由自由新增

---

## 4. 当前代码现状

结合当前仓库代码，导航模块已有以下现实基础：

- 共享契约已定义 `NavigationItem` 与 `NavigationItemType`
- 后端已存在 `NAVIGATION_TREE` 作为内置导航树
- 当前 `requiredPermissions` 在后端常量中已按 AND 语义解释
- `poms-admin` 前端通过 `AuthStore` 将 `NavigationItem[]` 转换为模板 `MenuItem[]`
- 当前动态菜单转换已覆盖 `title -> label`、`icon -> icon`、`link -> routerLink`、`children -> items`、`divider -> separator`
- 当前动态菜单尚未完整覆盖父级菜单激活与展开所需的 `path` 语义
- 后端导航已开始使用 `/platform/*`，但前端部分路由仍保留 `/profile/*` 等旧入口

当前缺口判断：

- 导航虽已有真实只读接口，但底层仍是 `NAVIGATION_TREE` 常量树
- 导航过滤已与真实 `User` / `Role` / `OrgUnit` 关系收敛，`/platform/users`、`/platform/roles`、`/platform/org-units`、`/platform/navigation` 均已有真实页面或只读治理入口
- `/platform/navigation` 当前仍是只读治理页，第一阶段不再追加运行时导航 CRUD
- 当前剩余需要收敛的不再是页面落点，而是“受控代码/seed 维护 + 运行时 `audit_log` 写入 + 文档同步 + 构建/smoke”的闭环

这意味着第一阶段导航设计必须同时面向：

- 后端契约稳定
- 前端模板可用
- 路由迁移已收敛
- 治理写侧和审计边界清楚

## 5. 核心对象与模型

### 5.1 核心对象

- `NavigationItem`
- `NavigationRouteMapping`（设计输出物，不要求必须单独持久化为实体）

### 5.2 建议字段

第一阶段 `NavigationItem` 采用以下正式字段口径：

| 字段                  | 说明                                              |
| --------------------- | ------------------------------------------------- |
| `id`                  | 导航项唯一标识                                    |
| `key`                 | 导航稳定键                                        |
| `type`                | 节点类型：`basic / group / collapsable / divider` |
| `title`               | 展示标题                                          |
| `subtitle`            | 辅助标题，第一阶段可为空                          |
| `link`                | 跳转目标                                          |
| `icon`                | 图标类名                                          |
| `displayOrder`        | 同层级排序值                                      |
| `isHidden`            | 是否隐藏                                          |
| `isDisabled`          | 是否禁用                                          |
| `requiredPermissions` | 导航可见性所需权限集合                            |
| `meta`                | 少量稳定扩展元数据                                |
| `children`            | 子节点列表                                        |

### 5.3 `meta` 字段口径

`meta` 只用于承载跨前后端都稳定的扩展语义，不承载 Angular 专有运行时参数。

第一阶段允许的典型用法：

- `activeMatchPath`
- `featureFlag`
- `badge`

第一阶段不建议放入：

- `routerLinkActiveOptions`
- `queryParamsHandling`
- `replaceUrl`
- 纯样式类名

## 6. 节点类型设计

### 6.1 `basic`

适用于可直接点击跳转的叶子节点。

规则：

- 返回时必须提供 `link`
- 可有 `requiredPermissions`
- 返回时不得携带非空 `children`

### 6.2 `group`

适用于顶层或分组容器节点。

规则：

- 返回时默认不提供可点击 `link`
- 主要承担信息架构分组和父级容器职责
- 返回时必须包含至少一个可见子节点

### 6.3 `collapsable`

适用于需要展开/收起的中间层节点。

规则：

- 可包含 `children`
- 第一阶段默认不依赖自身 `link` 跳转
- 若确需同时存在 `link`，必须在 `navigation-route-mapping.md` 中单独注明场景、跳转语义与匹配规则
- 当前如无真实业务需求，可暂不在平台导航第一批节点中大量使用

### 6.4 `divider`

适用于分隔符节点。

规则：

- 不应有 `link`
- 不应有 `requiredPermissions`
- 不应有 `children`
- `title`、`icon` 可为空

## 7. 权限过滤与可见性规则

### 7.1 匹配语义

第一阶段 `requiredPermissions` 默认采用 AND 语义：

- 当数组为空或为 `null` 时，表示仅需登录即可见
- 当数组存在多个权限 key 时，必须全部满足才可见
- 第一阶段不引入 `requiredPermissionMode`

### 7.2 优先级规则

导航返回规则按以下顺序判定：

1. 若 `isHidden=true`，则直接不返回
2. 若 `requiredPermissions` 不满足，则不返回
3. 若 `isDisabled=true`，则可返回但前端应渲染为不可点击
4. 若父节点存在但子节点全部不可见，应根据节点类型决定是否保留空容器

### 7.3 父子节点规则

- `group` 类型父节点第一阶段默认不承担独立导航准入权限，其可见性由过滤后的可见子节点集合派生
- 除 `group` 类型的派生可见性规则外，父节点本身不可见时，其子树整体不返回
- `group` 或 `collapsable` 节点若过滤后无任何可见子节点，第一阶段默认不返回空节点
- 不允许返回孤儿节点

说明：

- `ADR-009` 已正式确定第一阶段平台配置父组采用“任一子项可见则父组可见”的规则
- 若后续某父节点本身是独立可点击入口，或具有独立信息架构语义，再通过后续 ADR 决定是否允许恢复独立导航权限

## 8. 导航与真实路由对照

### 8.1 基本原则

- 后端 `link` 必须与前端真实路由保持一致
- 任何导航项启用前，都必须确认其 `link` 已有真实页面承载
- 在路由未收敛完成前，不允许自由修改核心导航 `link`

### 8.2 第一阶段输出物

第一阶段必须维护独立的 `navigation-route-mapping.md`，作为导航启用、路由迁移和前端收敛的正式基线。

### 8.3 当前已知迁移约束

当前代码中，平台导航已采用 `/platform/*` 目标路径，但前端仍存在 `/profile/*` 历史入口。  
因此第一阶段需要显式维护迁移策略：

- 路由已完成收敛的菜单，按目标正式 `link` 启用
- 路由尚未收敛的菜单，不应假设点击即可正确落页
- 旧路径与新路径关系必须记录在对照表中

### 8.5 与补齐计划的衔接

本模块属于 `P1-S09` 的核心范围，并依赖 `P1-S08` 提供真实用户、角色和组织事实源。

第一阶段导航补齐的最小要求为：

- 导航继续由后端作为单一事实源输出
- 导航过滤使用真实角色/权限结果，而不再只依赖开发期 fixture
- 平台治理相关导航项具备真实页面落点或清晰受控维护入口
- `navigation-route-mapping.md` 与真实前端路由保持一致
- 写侧治理收敛为受控代码/seed 维护，不额外要求运行时导航 CRUD；变更必须附带文档同步、构建 / smoke 校验，并在同步到运行时后写入统一 `audit_log`

### 8.4 第一阶段建议路由对照表

| 导航键                | 目标导航链接           | 当前前端现状                                      | 第一阶段处理建议                                         |
| --------------------- | ---------------------- | ------------------------------------------------- | -------------------------------------------------------- |
| `dashboard`           | `/dashboard`           | 已有真实工作台页，根路径默认重定向到 `/dashboard` | 保持为正式首页入口                                       |
| `platform.users`      | `/platform/users`      | 已有真实页面、真实 API 与权限守卫                 | 可按当前链接正式启用                                     |
| `platform.roles`      | `/platform/roles`      | 已有真实页面、真实 API 与权限守卫                 | 可按当前链接正式启用                                     |
| `platform.org-units`  | `/platform/org-units`  | 已有真实页面、真实 API 与权限守卫                 | 可按当前链接正式启用                                     |
| `platform.navigation` | `/platform/navigation` | 已有只读治理页，可查看完整导航树与路由对齐结果    | 可按当前链接正式启用；写侧治理按受控代码/seed 维护口径执行 |
| `my_profile`          | `/profile`             | 当前已有 `/profile/*` 体系                        | 可保留为当前阶段正式入口                                 |

说明：

- 当前 `/platform/*` 平台治理路由均已具备真实页面承载或只读治理入口，后续导航变更仍需同步维护本对照表
- 当前 `/profile/*` 体系仅保留个人中心语义，不等价于平台治理页的替代入口

## 9. `poms-admin` 适配设计

### 9.1 适配定位

`poms-admin` 前端不直接把 `NavigationItem` 当作模板数据使用，而是通过适配层转换为菜单组件使用的 `MenuItem` 视图模型。

这层适配必须被视为正式设计的一部分，而不是实现阶段的临时补丁。

### 9.2 映射规则

| `NavigationItem`       | `MenuItem`             | 说明                           |
| ---------------------- | ---------------------- | ------------------------------ |
| `title`                | `label`                | 标题映射                       |
| `icon`                 | `icon`                 | 图标映射                       |
| `link`                 | `routerLink` 或 `url`  | 第一阶段平台菜单以内部路由为主 |
| `children`             | `items`                | 递归映射                       |
| `type=divider`         | `separator=true`       | 分隔符映射                     |
| `isDisabled`           | `disabled`             | 禁用映射                       |
| `meta.activeMatchPath` | 适配层内部 `path` 语义 | 用于当前模板激活态计算         |

### 9.3 排序责任归属

- 导航树由后端完成过滤与排序后返回
- 前端适配层不应对已返回节点再次做业务排序
- 若迁移期存在兼容性补丁，必须在实现中显式标注为临时逻辑，而不是形成长期双重排序

### 9.4 当前适配缺口

基于当前代码，仍存在以下缺口：

- 动态菜单未完整映射父级高亮/展开所依赖的 `path`
- 外部链接 `url + target` 的正式契约支撑仍有限
- 根节点分隔策略目前仍由前端适配逻辑补齐

因此第一阶段导航实现应补以下能力：

- 适配层支持从 `meta.activeMatchPath` 或 `link` 推导当前模板所需的活动路径
- 适配层清晰区分内部路由与外部链接
- 适配层与静态 fallback 的行为差异要逐步收敛

## 10. 第一阶段有限维护边界

### 10.1 建议开放的维护能力

- 调整标题
- 调整图标
- 调整排序
- 隐藏/取消隐藏
- 禁用/取消禁用
- 调整权限要求

补充说明：

- 第一阶段即使暂不建设完整导航 CRUD 页面，也必须明确谁以何种方式维护这些字段
- 第一阶段当前接受的维护路径为：修改后端 `NAVIGATION_TREE` 或等价 seed / 契约事实源，同步 `navigation-route-mapping.md` 与相关设计文档，执行前端构建与平台治理 smoke，再通过 PR / 发布流程合入
- 该路径属于正式受控维护机制，但只有在同步到运行时后写入统一 `audit_log`，才可视为第一阶段完整闭环

### 10.2 第一阶段不建议开放的能力

- 任意新增未知业务路由节点
- 任意修改核心平台导航 `key`
- 在未完成页面落地前直接启用新菜单
- 通过后台维护框架专属参数
- 将外部链接作为正式平台导航能力大范围开放

说明：

- 第一阶段默认不把外部链接作为正式平台导航能力
- 当前若存在历史模板中的外部链接能力，只视为兼容遗留能力，不作为平台治理域正式承诺

## 11. 生命周期与变更规则

### 11.1 生命周期

- `enabled`：参与导航返回
- `disabled`：返回但不可点击
- `hidden`：不返回

说明：

- 第一阶段继续使用 `isHidden`、`isDisabled` 表达状态，不要求额外拆正式枚举字段
- 如后续导航管理复杂度显著提升，可再抽象为更显式的生命周期状态

### 11.2 关键变更规则

- 导航项停用或隐藏后，不得继续出现在返回导航树中
- 导航项变更 `link` 前必须校验真实路由存在
- 父节点停用后，子节点不得单独继续暴露
- 核心导航键不应在第一阶段频繁变更

---

## 12. 审计要求

第一阶段必须把以下动作接入统一结构化审计：

- 导航项创建、编辑、排序调整
- 标题、图标、链接、权限要求变更
- 隐藏、禁用、重新启用
- 路由不匹配导致的配置拒绝

第一阶段至少对以下内容保留结构化前后值或差异摘要：

- `link` 变更
- `requiredPermissions` 变更
- 父子层级变更
- `isHidden / isDisabled` 变更

第一阶段当前验收口径收敛为：

- 不要求新增运行时导航写侧 API 或可视化编排器
- 导航事实源变更只能通过受控代码/seed 维护完成
- 每次导航变更必须同时提交路由对照文档更新、构建 / smoke 结果
- 每次导航事实源同步到运行时后，必须写入统一 `audit_log`
- 不允许借“维护便利”开放未知路由自由新增能力

---

## 13. 测试与验收要点

第一阶段至少覆盖以下场景：

- 不同权限用户返回不同导航树
- `requiredPermissions` 多权限按 AND 语义过滤
- `isHidden=true` 的节点不返回
- `isDisabled=true` 的节点返回但不可点击
- 父节点不可见时不返回孤儿子节点
- 菜单 `link` 与真实前端路由一致
- 未授权用户直接输入 URL 时，页面守卫和接口授权同时生效
- 动态导航与静态 fallback 在主要菜单行为上不出现明显偏差
- 导航维护路径可回溯到代码变更、文档同步、发布批次和运行时审计记录
- 导航治理不引入未知路由自由新增能力
