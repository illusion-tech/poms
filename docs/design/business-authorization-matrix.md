## 1. 文档目标

本文档用于把 `POMS` 中“平台访问控制”和“业务对象动作授权”这两层能力正式拆开，避免后续在接口设计、页面守卫、审批流和角色配置中继续混用。

本文档当前阶段的目标不是一次性写完所有业务域的完整对象-动作-角色矩阵，而是先形成第一阶段可评审的**边界基线**，统一回答以下问题：

- 哪些能力属于平台权限，哪些不属于
- 业务对象动作授权应按什么粒度建模
- 平台治理域应向业务授权层输出什么稳定输入
- 后续销售流程域、合同资金域、提成治理域和审批流设计应如何在本基线上继续下钻

---

## 2. 当前阶段定位

当前文档处于：

**“业务授权边界已经固定，销售、合同资金、提成治理和横切审批的首批稳定对象动作已可统一回填。”**

这意味着：

- 本文档可作为后续详细设计的上游约束
- 本文档当前仍未覆盖全量对象动作矩阵
- 本文档已经可以作为接口、按钮、待办和审批动作设计的统一上游输入

---

## 3. 核心原则

- **平台权限不替代业务授权**: `platform:*` 与 `nav:*` 只能回答“能否进入平台、看到菜单、访问平台接口”，不能直接回答“能否操作某个业务对象”
- **业务授权面向对象动作**: 业务授权应围绕“对象 + 动作 + 条件 + 责任角色”建模，而不是只围绕页面或菜单建模
- **后端是授权事实源**: 页面可见性、按钮展示和接口调用最终都以后端当前有效授权结果为准
- **审批不等于授权**: 审批流用于控制特定状态迁移和敏感动作，不等于一般业务读写权限
- **主对象与子流程分层授权**: `Project` 与 `BidProcess` 应分别建模授权动作，不能把投标动作继续折叠成 `Project` 的普通字段编辑
- **先边界、后细化**: 在销售流程域、合同资金域、提成治理域详细设计完成前，先固定授权分层和矩阵模板，避免后续口径漂移

---

## 4. 分层边界

### 4.1 平台权限层

平台权限层负责：

- 登录后是否允许进入系统
- 是否允许进入某类平台管理模块
- 是否允许看到某些导航入口
- 是否允许访问平台治理相关接口

平台权限层不负责：

- 某个 `Project` 是否允许当前用户编辑
- 某个 `BidProcess` 是否允许当前用户提交投标决策或登记定标结果
- 某个合同是否允许当前用户提交审批
- 某个提成角色分配是否允许当前用户冻结或变更
- 某个回款记录是否允许当前用户确认生效

### 4.2 业务对象动作授权层

业务对象动作授权层负责：

- 哪个角色或责任人可对哪个业务对象执行何种动作
- 是否受对象状态、审批节点、责任归属或组织归口限制
- 同一对象在不同阶段是否允许不同动作
- 敏感动作是否需要二次审批、复核或审计留痕

### 4.3 两层之间的衔接

平台治理域向业务授权层至少输出以下稳定输入：

- 当前用户的有效平台身份与 `roleKey[]`
- 当前用户的有效平台权限集合
- 当前用户的主责组织与附属组织上下文
- 当前用户可进入的模块与页面边界

业务授权层在此基础上继续判断：

- 当前用户是否是对象责任人或参与角色
- 当前用户是否满足对象状态和审批节点要求
- 当前用户是否满足归属组织、归口或其他业务条件

## 5. 第一阶段基线矩阵

### 5.1 分层矩阵

| 场景                     | 是否属于平台权限 | 是否属于业务对象动作授权 | 说明                             |
| ------------------------ | ---------------- | ------------------------ | -------------------------------- |
| 登录后看到平台配置菜单   | 是               | 否                       | 属于导航与平台入口控制           |
| 进入用户管理页面         | 是               | 否                       | 属于平台治理域访问控制           |
| 维护角色权限关系         | 是               | 否                       | 属于平台治理域能力               |
| 查看项目列表页面入口     | 是               | 否                       | 页面入口属于导航与平台访问控制   |
| 查看某个项目详情         | 否               | 是                       | 是否可见具体对象应由业务授权判定 |
| 编辑某个项目信息         | 否               | 是                       | 属于业务对象写动作               |
| 提交项目进入下一阶段审批 | 否               | 是                       | 属于对象状态迁移动作             |
| 创建或推进投标流程       | 否               | 是                       | 属于 `BidProcess` 对象动作       |
| 登记投标结果             | 否               | 是                       | 属于投标结果敏感动作             |
| 查看合同台账列表页面入口 | 是               | 否                       | 入口控制与对象访问控制分层       |
| 编辑某份合同条款         | 否               | 是                       | 属于合同对象动作授权             |
| 确认回款生效             | 否               | 是                       | 属于资金对象敏感动作             |
| 冻结提成角色分配         | 否               | 是                       | 属于提成治理域对象动作           |
| 查看统一待办入口         | 是               | 否                       | 待办入口本身是平台/导航能力      |
| 执行某条待办对应审批动作 | 否               | 是                       | 待办背后仍回到具体业务对象动作   |

### 5.3 销售流程域第一批稳定对象动作

基于当前已接受的 ADR-011 和 `project-lifecycle-design.md`，销售流程域可先固定以下第一批稳定对象动作，用于后续接口、按钮和审批流设计对齐。

| 业务域     | 对象                         | 动作            | 触发条件                                   | 授权主体                             | 状态约束                                             | 是否需审批 | 审计要求                         | 备注                                              |
| ---------- | ---------------------------- | --------------- | ------------------------------------------ | ------------------------------------ | ---------------------------------------------------- | ---------- | -------------------------------- | ------------------------------------------------- |
| 销售流程域 | `Project`                    | 创建            | 已存在有效 `Lead`                          | 销售人员                             | `Lead` 已提交且可转入正式推进                        | 否         | 记录创建人、来源线索、创建时间   | 不等于通过立项                                    |
| 销售流程域 | `ProjectAssessment`          | 提交立项评估    | 已创建 `Project`                           | 销售人员                             | `Project.stage = assessment`                         | 是         | 留痕提交版本、审批链路与结论     | 审批通过后才可推进到范围确认                      |
| 销售流程域 | `ScopeConfirmation`          | 确认范围        | 立项已通过                                 | 技术支持 / 售前                      | `Project.stage = scope-confirmation`                 | 是         | 留痕确认人、版本、附件与时间     | 已确认后变更应新开版本                            |
| 销售流程域 | `QuotationReview`            | 提交报价评审    | 范围已确认                                 | 销售人员                             | `Project.stage = commercial-closure`                 | 是         | 留痕报价版本、毛利测算与审批结论 | 属于商务收口子动作，不再单独占据 `Project.stage`  |
| 销售流程域 | `BidProcess`                 | 创建            | `Project.commercialMode = bidding`         | 销售人员                             | `Project.stage = commercial-closure`                 | 否         | 留痕创建人、关联项目、创建时间   | 一个项目同一时点通常只有一个当前有效 `BidProcess` |
| 销售流程域 | `BidProcess`                 | 提交投标决策    | 已创建 `BidProcess`                        | 销售负责人                           | `BidProcess.status = 待投标决策`                     | 是         | 留痕决策依据、审批链路与结论     | 未通过不得进入编制或递交                          |
| 销售流程域 | `BidProcess`                 | 更新标书 / 递交 | 投标决策已通过                             | 销售人员 / 商务配合 / 售前支持       | `BidProcess.status in (编制中, 待递交)`              | 否         | 留痕版本、递交时间、附件清单     | 递交后进入已递交状态                              |
| 销售流程域 | `BidProcess`                 | 登记投标结果    | 已递交或澄清完成                           | 销售负责人 / 商务负责人              | `BidProcess.status in (已递交, 澄清中)`              | 是         | 留痕结果、来源依据、确认时间     | 中标才允许项目进入签约                            |
| 销售流程域 | `ExecutiveEscalationRequest` | 发起高层介入    | 毛利红线以下、重大例外、战略项目或关键卡点 | 销售负责人                           | `Project.stage in (commercial-closure, contracting)` | 是         | 留痕卡点说明、目标、审批结论     | 不替代正常对象权限                                |
| 销售流程域 | `Project`                    | 提交签约登记    | 商务收口完成；若为招投标项目则已中标       | 销售人员 / 商务行政                  | `Project.stage = contracting` 或待进入 `contracting` | 是         | 留痕签约条件、合同版本与审批结论 | 未完成合同审核不得形成有效签约                    |
| 销售流程域 | `ProjectHandover`            | 完成移交确认    | 合同台账已建立                             | 销售、技术支持、商务行政、相关责任人 | `Project.stage = handover`                           | 是         | 留痕交接清单、责任人、冻结版本   | 未完成移交不得进入执行态                          |
| 销售流程域 | `AcceptanceRecord`           | 确认验收        | 已形成可验收成果                           | 相关责任人 / 业务确认角色            | `Project.stage = acceptance`                         | 是         | 留痕验收结论、时间、附件         | 无有效验收记录不得进入完成态                      |

说明：

- 这不是销售流程域的全量矩阵，而是当前已稳定、足以驱动后续设计的第一批对象动作。
- `Project` 的“编辑基础信息”和 `BidProcess` 的“补充附件/澄清材料”这类普通维护动作，后续仍需按字段敏感度与责任归属继续细化。
- 本表的价值在于先把主对象推进动作与 `BidProcess` 子流程动作分开，避免后续再次混回页面级权限。

### 5.4 合同资金域第一批稳定对象动作

基于 `contract-finance-design.md`，合同资金域可先固定以下第一批稳定对象动作，用于后续接口、审批流、提成联动和经营看板对齐。

| 业务域     | 对象                | 动作            | 触发条件                       | 授权主体                | 状态约束                                   | 是否需审批 | 审计要求                         | 备注                                      |
| ---------- | ------------------- | --------------- | ------------------------------ | ----------------------- | ------------------------------------------ | ---------- | -------------------------------- | ----------------------------------------- |
| 合同资金域 | `Contract`          | 登记            | 项目已进入签约阶段             | 销售人员 / 商务行政     | `Project.stage = contracting`              | 否         | 留痕登记人、项目、客户、合同草稿 | 建立合同草稿，不等于合同已生效            |
| 合同资金域 | `Contract`          | 提交审核        | 合同草稿已完善关键条款         | 商务行政                | `Contract.status = 草稿`                   | 是         | 留痕提交版本、附件、审批链路     | 审核通过后才可生效                        |
| 合同资金域 | `Contract`          | 生效            | 审核通过                       | 商务行政 / 必要审批角色 | `Contract.status = 待审核`                 | 是         | 留痕生效时间、审批结论、快照版本 | 生效后生成当前有效 `ContractTermSnapshot` |
| 合同资金域 | `ContractAmendment` | 发起变更        | 已存在生效合同                 | 销售负责人 / 商务行政   | `Contract.status in (已生效, 履约中)`      | 是         | 留痕变更原因、条款差异、审批结论 | 不得直接覆盖原合同关键字段                |
| 合同资金域 | `ReceivablePlan`    | 初始化 / 生效   | 已存在当前有效合同条款快照     | 商务行政 / 财务配合     | 已存在当前有效 `ContractTermSnapshot`      | 否         | 留痕生成依据、计划节点、版本时间 | 正式计划必须基于有效快照生成              |
| 合同资金域 | `ReceiptRecord`     | 登记到账        | 已存在合同或项目归属           | 销售助理 / 财务人员     | `ReceiptRecord.status = 草稿`              | 否         | 留痕来源、到账时间、金额、凭证   | 第一阶段允许人工录入                      |
| 合同资金域 | `ReceiptRecord`     | 财务确认        | 已登记到账记录                 | 财务人员                | `ReceiptRecord.status = 待确认`            | 是         | 留痕确认人、确认时间、结论       | 已确认后才纳入生效回款口径                |
| 合同资金域 | `ReceiptRecord`     | 冲销 / 作废     | 已确认错误或到账需撤回         | 财务人员 / 财务负责人   | `ReceiptRecord.status in (待确认, 已确认)` | 是         | 留痕冲销原因、关联原记录、处理人 | 已确认记录不得直接删除                    |
| 合同资金域 | `PayableRecord`     | 登记应付        | 已识别供应商成本或项目成本事实 | 商务行政 / 项目责任人   | `PayableRecord.status = 草稿`              | 否         | 留痕供应商、成本类型、金额       | 第一阶段为成本台账，不等于完整付款审批    |
| 合同资金域 | `PaymentRecord`     | 登记付款        | 已存在应付台账或明确成本支出   | 财务人员 / 商务行政     | `PaymentRecord.status = 草稿`              | 否         | 留痕支付时间、金额、凭证         | 第一阶段为付款跟踪记录                    |
| 合同资金域 | `PaymentRecord`     | 确认生效        | 已登记付款记录                 | 财务人员                | `PaymentRecord.status = 已登记`            | 是         | 留痕确认人、确认时间、处理结论   | 已确认后进入生效成本口径                  |
| 合同资金域 | `InvoiceRecord`     | 登记 / 更新状态 | 已存在合同或项目发票事项       | 财务人员 / 商务行政     | `InvoiceRecord.status in (草稿, 待开票)`   | 否         | 留痕发票状态、金额、抬头、时间   | 第一阶段以台账与状态跟踪为主              |
| 合同资金域 | `InvoiceRecord`     | 标记异常 / 解除 | 发票存在异常或异常已处理       | 财务人员                | `InvoiceRecord.status in (已开票, 异常)`   | 是         | 留痕异常说明、解除结论、责任人   | 异常解除不得直接覆盖历史                  |

说明：

- `ReceiptRecord` 的“财务确认”与 `PaymentRecord` 的“确认生效”属于进入生效口径的关键敏感动作，应与普通登记动作分开建模。
- 第一阶段不把 `PayableRecord` / `PaymentRecord` 扩展成完整付款审批流，但仍需要明确对象动作与审计留痕。

### 5.5 提成治理域第一批稳定对象动作

基于 `commission-settlement-design.md`，提成治理域可先固定以下第一批稳定对象动作，用于后续审批流、待办、发放限制和异常调整设计对齐。

| 业务域     | 对象                       | 动作            | 触发条件                           | 授权主体                | 状态约束                                            | 是否需审批 | 审计要求                         | 备注                                       |
| ---------- | -------------------------- | --------------- | ---------------------------------- | ----------------------- | --------------------------------------------------- | ---------- | -------------------------------- | ------------------------------------------ |
| 提成治理域 | `CommissionRoleAssignment` | 录入 / 编辑     | 项目已创建并进入可分配阶段         | 销售负责人 / 管理层     | `CommissionRoleAssignment.status = 草稿`            | 否         | 留痕参与角色、比例、责任说明     | 冻结前允许维护                             |
| 提成治理域 | `CommissionRoleAssignment` | 提交冻结        | 项目已完成移交                     | 销售负责人 / 商务行政   | `CommissionRoleAssignment.status in (草稿, 待冻结)` | 是         | 留痕提交人、冻结版本、确认链路   | 冻结后不得直接改写原版本                   |

| 提成治理域 | `CommissionCalculation`    | 触发计算        | 已有冻结角色版本与生效输入         | 系统 / 财务人员         | `CommissionCalculation.status = 待计算`             | 否         | 留痕规则版本、输入快照、触发原因 | 计算动作可系统触发                         |
| 提成治理域 | `CommissionCalculation`    | 复核生效        | 已完成计算                         | 财务人员 / 管理层       | `CommissionCalculation.status in (已计算, 待复核)`  | 是         | 留痕复核结论、版本号、责任人     | 生效后才可作为发放依据                     |
| 提成治理域 | `CommissionCalculation`    | 触发重算        | 合同变更、回款冲销、成本变化或违规 | 财务人员 / 管理层       | `CommissionCalculation.status in (已生效, 已冻结)`  | 是         | 留痕重算原因、替代链路、输入变化 | 不得直接覆盖旧结果                         |
| 提成治理域 | `CommissionPayout`         | 发起发放        | 满足发放启动条件                   | 财务人员 / 管理层       | `CommissionPayout.status = 草稿`                    | 否         | 留痕发放阶段、档位建议、测算金额 | 系统校验阶段上限                           |
| 提成治理域 | `CommissionPayout`         | 提交审批 / 批准 | 已形成有效发放草稿                 | 管理层 / 预算审批角色   | `CommissionPayout.status in (草稿, 待审批)`         | 是         | 留痕审批档位、批准结论、限制说明 | 审批负责放行档位，不负责重算               |
| 提成治理域 | `CommissionPayout`         | 登记发放        | 发放审批已通过                     | 财务人员 / 业务登记角色 | `CommissionPayout.status = 已批准`                  | 否         | 留痕发放时间、金额、登记人       | 第一阶段是业务发放记录，不等于真实财务付款 |
| 提成治理域 | `CommissionPayout`         | 暂停 / 冲销     | 出现异常、违规、退款或坏账         | 管理层 / 财务人员       | `CommissionPayout.status in (已批准, 已发放)`       | 是         | 留痕暂停或冲销原因、关联调整单   | 不得直接删除已发放记录                     |
| 提成治理域 | `CommissionAdjustment`     | 发起调整        | 已确认存在退款、坏账、违规等异常   | 财务人员 / 管理层       | `CommissionAdjustment.status = 草稿`                | 否         | 留痕异常类型、依据、影响范围     | 是重算、扣回、补发的统一入口               |
| 提成治理域 | `CommissionAdjustment`     | 提交审批 / 执行 | 调整草稿已完整                     | 管理层                  | `CommissionAdjustment.status in (草稿, 待审批)`     | 是         | 留痕审批结论、执行动作、关联对象 | 已执行调整不得直接删除                     |
| 提成治理域 | `CommissionRuleVersion`    | 提交生效 / 启用 | 规则草稿已完成                     | 管理层 / 制度维护角色   | `CommissionRuleVersion.status in (草稿, 待生效)`    | 是         | 留痕规则版本、生效时间、批准意见 | 已生效规则不得静默改写                     |

说明：

- 提成治理域中“系统计算”和“管理审批”必须分层，避免把审批结论误当成计算逻辑的一部分。
- `CommissionPayout` 的“登记发放”与“批准发放”是两个独立动作，前者是业务登记，后者是审批放行。

### 5.6 横切审批域第一批稳定对象动作

基于 `workflow-and-approval-design.md`，横切审批域可先固定以下统一公共动作，用于各模块在统一待办、通知和审计留痕上的实现对齐。

| 业务域     | 对象                 | 动作            | 触发条件               | 授权主体                   | 状态约束                                           | 是否需审批 | 审计要求                         | 备注                                   |
| ---------- | -------------------- | --------------- | ---------------------- | -------------------------- | -------------------------------------------------- | ---------- | -------------------------------- | -------------------------------------- |
| 横切审批域 | `ApprovalRecord`     | 提交审批        | 业务对象触发审批节点   | 业务发起人                 | `ApprovalRecord.currentStatus = draft`             | 否         | 留痕发起人、目标对象、提交时间   | 创建统一审批实例并派生待办             |
| 横切审批域 | `ApprovalRecord`     | 审批通过        | 当前处理人收到待办     | 当前审批人                 | `ApprovalRecord.currentStatus = pending`           | 是         | 留痕审批人、结论、意见、时间     | 驱动业务对象推进                       |
| 横切审批域 | `ApprovalRecord`     | 驳回            | 当前处理人收到待办     | 当前审批人                 | `ApprovalRecord.currentStatus = pending`           | 是         | 留痕驳回结论、意见、时间         | 驱动业务对象回退或关闭                 |
| 横切审批域 | `ApprovalRecord`     | 取消 / 关闭     | 审批终止或业务对象关闭 | 发起人 / 业务管理员 / 系统 | `ApprovalRecord.currentStatus in (draft, pending)` | 否         | 留痕终止原因、处理人、时间       | 审批关闭后对应待办必须同步关闭         |
| 横切审批域 | `ApprovalRecord`     | 转派            | 当前处理人无法继续处理 | 当前审批人 / 审批管理员    | `ApprovalRecord.currentStatus = pending`           | 否         | 留痕原处理人、新处理人、原因     | 第一阶段可先做受控转派                 |
| 横切审批域 | `ConfirmationRecord` | 提交确认        | 业务对象触发确认节点   | 业务发起人 / 系统          | `ConfirmationRecord.currentStatus = draft`         | 否         | 留痕确认类型、目标对象、发起时间 | 生成统一确认待办                       |
| 横切审批域 | `ConfirmationRecord` | 完成确认        | 当前确认人处理待办     | 当前确认人                 | `ConfirmationRecord.currentStatus = pending`       | 否         | 留痕确认人、意见、时间           | 多方确认必须累计满足后才放行           |
| 横切审批域 | `TodoItem`           | 认领 / 处理完成 | 已生成待办             | 当前待办处理人             | `TodoItem.status in (open, processing)`            | 否         | 留痕处理状态、处理时间、关联实例 | 不替代业务审批结论，仅表达待办处理进度 |
| 横切审批域 | `NotificationRecord` | 发送 / 已读     | 审批或确认状态发生变化 | 系统 / 接收人              | `NotificationRecord.status in (待发送, 已发送)`    | 否         | 留痕接收人、触发事件、已读时间   | 第一期优先站内通知                     |

说明：

- 横切审批域中的公共对象不替代业务对象动作授权，它们只承接统一派单、通知和留痕。
- `ApprovalRecord`、`ConfirmationRecord`、`TodoItem`、`NotificationRecord` 的动作应作为公共能力接口统一抽象，而不是由每个业务模块重新定义。

## 6. 与平台治理域的边界结论

基于当前已接受的 ADR 与平台治理域设计，第一阶段正式固定以下结论：

1. `platform:*` 权限只用于平台管理能力和平台接口访问边界。
2. `nav:*` 权限只用于导航可见性和页面入口体验。
3. 菜单可见不等于具体业务对象可操作。
4. 平台角色是权限集合容器，不直接等价于业务对象责任角色。
5. 组织上下文第一阶段不进入平台主授权链路，但可作为后续业务授权输入。
6. 审批通过/驳回是业务对象动作控制，不属于平台 RBAC 的直接表达范围。
