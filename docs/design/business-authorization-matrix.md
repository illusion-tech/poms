## 1. 文档目标

本文档用于把 `POMS` 中“平台访问控制”和“业务对象动作授权”这两层能力正式拆开，避免后续在接口设计、页面守卫、审批流和角色配置中继续混用。

本文档当前阶段的目标不是一次性写完所有业务域的完整对象-动作-角色矩阵，而是先形成第一阶段可评审的**首版基线**，统一回答以下问题：

- 哪些能力属于平台权限，哪些不属于
- 业务对象动作授权应按什么粒度建模
- 平台治理域应向业务授权层输出什么稳定输入
- 后续销售流程域、合同资金域、提成治理域和审批流设计应如何在本基线上继续下钻

## 2. 当前阶段定位

当前文档处于：

**“业务授权边界已经固定，销售、合同资金、提成治理和横切审批的首批稳定对象动作已可统一回填。”**

这意味着：

- 本文档可作为后续详细设计的上游约束
- 本文档当前仍未覆盖全量对象动作矩阵
- 本文档已经可以作为接口、按钮、待办和审批动作设计的统一上游输入

补充当前阶段判断：

- 平台治理域与提成治理域已重新纳回第一阶段必做范围
- 因此本矩阵不再只服务于“后续可选扩展”，而必须直接服务于第一阶段补齐实施
- 当前首要目标是把平台治理域动作矩阵与提成治理域动作矩阵推进到可直接指导接口切分与命令设计的程度

## 3. 核心原则

- **平台权限不替代业务授权**: `platform:*` 与 `nav:*` 只能回答“能否进入平台、看到菜单、访问平台接口”，不能直接回答“能否操作某个业务对象”
- **业务授权面向对象动作**: 业务授权应围绕“对象 + 动作 + 条件 + 责任角色”建模，而不是只围绕页面或菜单建模
- **后端是授权事实源**: 页面可见性、按钮展示和接口调用最终都以后端当前有效授权结果为准
- **审批不等于授权**: 审批流用于控制特定状态迁移和敏感动作，不等于一般业务读写权限
- **主对象与子流程分层授权**: `Project` 与 `BidProcess` 应分别建模授权动作，不能把投标动作继续折叠成 `Project` 的普通字段编辑
- **先边界、后细化**: 在销售流程域、合同资金域、提成治理域详细设计完成前，先固定授权分层和矩阵模板，避免后续口径漂移

## 4. 分层边界

### 4.1 平台权限层

平台权限层负责：

- 登录后是否允许进入系统
- 是否允许进入某类平台管理模块
- 是否允许看到某些导航入口
- 是否允许访问平台治理相关接口

平台权限层不负责：

- 某个 `Project` 是否允许当前用户编辑
- 某个 `BidProcess` 是否允许当前用户提交投标决策或登记定标结果
- 某个合同是否允许当前用户提交审批
- 某个提成角色分配是否允许当前用户冻结或变更
- 某个回款记录是否允许当前用户确认生效

### 4.2 业务对象动作授权层

业务对象动作授权层负责：

- 哪个角色或责任人可对哪个业务对象执行何种动作
- 是否受对象状态、审批节点、责任归属或组织归口限制
- 同一对象在不同阶段是否允许不同动作
- 敏感动作是否需要二次审批、复核或审计留痕

### 4.3 两层之间的衔接

平台治理域向业务授权层至少输出以下稳定输入：

- 当前用户的有效平台身份与 `roleKey[]`
- 当前用户的有效平台权限集合
- 当前用户的主责组织与附属组织上下文
- 当前用户可进入的模块与页面边界

业务授权层在此基础上继续判断：

- 当前用户是否是对象责任人或参与角色
- 当前用户是否满足对象状态和审批节点要求
- 当前用户是否满足归属组织、归口或其他业务条件

### 4.4 矩阵统一表达约定

为避免“审批”和“确认”在同一列中混用，本文档后续统一采用以下列口径：

- `组织范围`: 表达动作是否受主责组织、归口组织、项目归属组织或财务归口限制
- `字段敏感度`: 表达动作对应字段或数据的敏感级别，至少区分 `低`、`中`、`高`
- `放行方式`: 统一表达动作是否需要 `无`、`审批`、`确认`、`复核`、`多方确认` 或组合放行

其中：

- “审批”用于例外裁量、高风险放行与敏感状态迁移
- “确认”用于事实成立、生效认定与责任留痕
- “复核”用于计算结果、生效口径等需要第二责任人确认的场景

## 5. 第一阶段基线矩阵

### 5.1 分层矩阵

| 场景                     | 是否属于平台权限 | 是否属于业务对象动作授权 | 说明                             |
| ------------------------ | ---------------- | ------------------------ | -------------------------------- |
| 登录后看到平台配置菜单   | 是               | 否                       | 属于导航与平台入口控制           |
| 进入用户管理页面         | 是               | 否                       | 属于平台治理域访问控制           |
| 维护角色权限关系         | 是               | 否                       | 属于平台治理域能力               |
| 查看项目列表页面入口     | 是               | 否                       | 页面入口属于导航与平台访问控制   |
| 查看某个项目详情         | 否               | 是                       | 是否可见具体对象应由业务授权判定 |
| 编辑某个项目信息         | 否               | 是                       | 属于业务对象写动作               |
| 提交项目进入下一阶段审批 | 否               | 是                       | 属于对象状态迁移动作             |
| 创建或推进投标流程       | 否               | 是                       | 属于 `BidProcess` 对象动作       |
| 登记投标结果             | 否               | 是                       | 属于投标结果敏感动作             |
| 查看合同台账列表页面入口 | 是               | 否                       | 入口控制与对象访问控制分层       |
| 编辑某份合同条款         | 否               | 是                       | 属于合同对象动作授权             |
| 确认回款生效             | 否               | 是                       | 属于资金对象敏感动作             |
| 冻结提成角色分配         | 否               | 是                       | 属于提成治理域对象动作           |
| 查看统一待办入口         | 是               | 否                       | 待办入口本身是平台/导航能力      |
| 执行某条待办对应审批动作 | 否               | 是                       | 待办背后仍回到具体业务对象动作   |

### 5.2 第一阶段统一列模板

第一阶段各业务域对象动作已按以下统一列展开，并在后续分域章节中给出首批稳定基线：

| 业务域     | 对象                        | 动作                   | 触发条件 | 授权主体               | 组织范围             | 字段敏感度 | 状态约束             | 放行方式         | 审计要求 | 备注                       |
| ---------- | --------------------------- | ---------------------- | -------- | ---------------------- | -------------------- | ---------- | -------------------- | ---------------- | -------- | -------------------------- |
| 销售流程域 | `Project` / 子流程对象      | 创建、推进、关闭       | 见 `5.3` | 销售 / 售前 / 管理层   | `Project` 或项目组织 | 中 / 高    | 见对象与阶段约束     | 无 / 审批 / 确认 | 必须留痕 | 以生命周期与子流程映射为主 |
| 合同资金域 | `Contract` / 资金对象       | 登记、生效、确认       | 见 `5.4` | 商务 / 财务            | 项目 / 商务 / 财务   | 中 / 高    | 见合同与资金状态约束 | 无 / 审批 / 确认 | 必须留痕 | 以生效事实与快照口径为主   |
| 提成治理域 | 提成对象                    | 冻结、计算、发放       | 见 `5.5` | 财务 / 管理层          | 项目 / 财务 / 制度   | 高         | 见提成状态约束       | 无 / 审批 / 复核 | 必须留痕 | 以版本替代与异常留痕为主   |
| 横切审批域 | `ApprovalRecord` 等公共对象 | 提交、通过、确认、关闭 | 见 `5.6` | 发起人 / 处理人 / 系统 | 不适用               | 中 / 高    | 见审批实例状态约束   | 无 / 审批 / 确认 | 必须留痕 | 仅承接公共派单与审计能力   |

说明：

- `5.3` 到 `5.6` 已给出第一阶段首批稳定对象动作，不再属于“待补”状态。
- 后续仍需细化的，是字段级全量矩阵、少量复杂场景和实现级接口切分，而不是基础列口径本身。

结合 `platform-governance-design.md` 及其子设计，平台治理域在第一阶段补齐时，至少应固定以下对象动作：

| 业务域     | 对象                       | 动作                                  | 触发条件                        | 授权主体                  | 组织范围 | 字段敏感度 | 状态约束                                     | 放行方式 | 审计要求                         | 备注                                       |
| ---------- | -------------------------- | ------------------------------------- | ------------------------------- | ------------------------- | -------- | ---------- | -------------------------------------------- | -------- | -------------------------------- | ------------------------------------------ |
| 平台治理域 | `User`                     | 创建                                  | 具备平台用户管理权限            | 平台管理员                | 平台级   | 高         | 用户名唯一；必须指定主责组织                 | 无       | 留痕创建人、初始角色、初始组织   | 不等于自动激活完整访问能力                 |

| 平台治理域 | `User`                     | 启用 / 停用                           | 用户已存在                      | 平台管理员                | 平台级   | 高         | 停用后下一次请求必须失效                     | 无       | 留痕启停结论、原因、操作者       | 不允许物理删除替代停用                     |
| 平台治理域 | `UserRoleAssignment`       | 分配 / 撤销角色                       | 用户、角色均存在且状态允许      | 平台管理员                | 平台级   | 高         | 同一有效关系唯一；停用角色不得新增分配       | 无       | 留痕新旧角色关系、原因、操作者   | 角色分配不得混入普通用户更新               |
| 平台治理域 | `UserOrgMembership`        | 绑定 / 调整组织                       | 用户、组织均存在且组织可用      | 平台管理员                | 平台级   | 高         | 必须且仅有一个主责组织                       | 无       | 留痕主责/附属变更、原因、操作者  | 组织分配不得混入普通用户更新               |
| 平台治理域 | `Role`                     | 创建 / 更新                           | 具备角色管理权限                | 平台管理员                | 平台级   | 高         | 角色键稳定；系统角色受保护                   | 无       | 留痕角色定义变更前后值           | 自定义角色不得绕过权限字典                 |
| 平台治理域 | `Role`                     | 启用 / 停用                           | 角色已存在                      | 平台管理员                | 平台级   | 高         | 停用后不再参与新增分配与授权计算             | 无       | 留痕启停结论、原因、操作者       | 历史分配记录保留                           |
| 平台治理域 | `RolePermissionAssignment` | 分配 / 撤销权限                       | 角色存在；权限 key 合法且可分配 | 平台管理员                | 平台级   | 高         | 同一有效关系唯一；废弃权限不得新增绑定       | 无       | 留痕权限集合变更、原因、操作者   | 权限字典第一阶段不开放任意新增 key         |
| 平台治理域 | `OrgUnit`                  | 创建 / 更新                           | 具备组织管理权限                | 平台管理员                | 平台级   | 中 / 高    | 编码唯一；父节点必须合法且启用               | 无       | 留痕节点定义、父子关系、排序变更 | 普通更新不应绕过树结构约束                 |
| 平台治理域 | `OrgUnit`                  | 启用 / 停用                           | 组织已存在                      | 平台管理员                | 平台级   | 高         | 停用父节点需级联停用子树                     | 无       | 留痕启停范围、原因、操作者       | 停用后不得再作为新建用户可选组织           |
| 平台治理域 | `OrgUnit`                  | 移动 / 排序调整                       | 目标父节点合法且启用            | 平台管理员                | 平台级   | 高         | 不得形成环；不得移动到停用节点或后代节点之下 | 无       | 留痕原位置、新位置、原因         | 建议使用命令接口                           |
| 平台治理域 | `NavigationItem`           | 调整标题/图标/排序/显隐/禁用/权限要求 | 导航项存在且维护入口受控        | 平台管理员 / 导航维护角色 | 平台级   | 高         | `link` 必须对应真实路由；权限 key 必须有效   | 无       | 第一阶段必须写入统一 `audit_log`；受控代码/seed 发布需记录前后值摘要、操作者与发布批次 | 第一阶段允许受控代码/seed 维护，不要求完整可视化编排器 |

说明：

- 平台治理域动作虽然属于平台管理能力，但同样应按“对象 + 动作”建模，不能退化为页面入口权限。
- 平台治理域高敏动作第一阶段原则上仍应走命令接口；`NavigationItem` 作为例外，可通过受控代码/seed 维护，但必须同步 `navigation-route-mapping.md`、相关构建 / smoke 校验，并在同步到运行时后写入统一 `audit_log`。
- 统一运行时审计落库与安全事件结构化留痕已纳入第一阶段硬门槛，当前由 `P1-T26` ~ `P1-T29` 跟踪。

| 销售流程域 | `Project`                    | 提交签约登记    | 商务收口完成；若为招投标项目则已中标       | 销售人员 / 商务行政                  | `Project` 主责组织 | 高         | `Project.stage = contracting` 或待进入 `contracting` | 审批/确认 | 留痕签约条件、合同版本与审批结论 | 未完成合同审核不得形成有效签约                    |
| 销售流程域 | `ProjectHandover`            | 完成移交确认    | 合同台账已建立                             | 销售、技术支持、商务行政、相关责任人 | 项目归属组织       | 高         | `Project.stage = handover`                           | 多方确认  | 留痕交接清单、责任人、冻结版本   | 未完成移交不得进入执行态                          |
| 销售流程域 | `AcceptanceRecord`           | 确认验收        | 已形成可验收成果                           | 相关责任人 / 业务确认角色            | 项目归属组织       | 高         | `Project.stage = acceptance`                         | 确认      | 留痕验收结论、时间、附件         | 无有效验收记录不得进入完成态                      |

说明：

- 这不是销售流程域的全量矩阵，而是当前已稳定、足以驱动后续设计的第一批对象动作。
- `Project` 的“编辑基础信息”和 `BidProcess` 的“补充附件/澄清材料”这类普通维护动作，后续仍需按字段敏感度与责任归属继续细化。
- 本表的价值在于先把主对象推进动作与 `BidProcess` 子流程动作分开，避免后续再次混回页面级权限。

### 5.7 字段敏感度基线

第一阶段建议把字段敏感度至少收敛为以下三级：

- `低`: 普通说明性字段、非关键展示字段
- `中`: 一般业务维护字段，会影响跟踪、统计或待办展示，但不直接改变生效口径
- `高`: 会改变合同生效、回款生效、提成计算、审批结论、关闭结果等关键业务事实的字段

| 对象                       | 字段示例                                 | 敏感级别 | 控制原则                                   |
| -------------------------- | ---------------------------------------- | -------- | ------------------------------------------ |
| `Project`                  | 名称、备注、预期时间                     | 中       | 普通维护动作可由责任角色执行               |
| `Project`                  | `commercialMode`、关闭原因、阶段推进引用 | 高       | 不得由普通编辑直接覆盖，应受状态与审批约束 |
| `BidProcess`               | 标书附件、澄清说明                       | 中       | 可由投标相关责任角色维护                   |
| `BidProcess`               | 投标结果、决策结论                       | 高       | 必须经过审批或确认留痕                     |
| `Contract`                 | 合同编号、归属项目、普通说明             | 中       | 可在草稿态维护                             |
| `Contract`                 | 金额、税率、付款条款、首付款比例         | 高       | 只允许在受控审核链路中生效                 |
| `ReceiptRecord`            | 来源备注、附件说明                       | 中       | 可由登记角色补录                           |
| `ReceiptRecord`            | 到账金额、到账日期、确认结论             | 高       | 必须受财务归口和确认动作约束               |
| `CommissionRoleAssignment` | 角色说明                                 | 中       | 草稿态可维护                               |
| `CommissionRoleAssignment` | 参与角色、比例、冻结结果                 | 高       | 冻结后不得直接改写                         |
| `CommissionPayout`         | 备注、说明                               | 中       | 允许业务补充                               |
| `CommissionPayout`         | 档位、批准金额、暂停/冲销结果            | 高       | 必须经过审批留痕                           |

### 5.8 组织范围基线

第一阶段建议先固定以下组织范围约束：

| 对象 / 动作                  | 组织范围基线                       | 说明                                           |
| ---------------------------- | ---------------------------------- | ---------------------------------------------- |
| 销售流程域普通维护动作       | `Project` 主责组织                 | 先以项目主责组织为主，不扩展复杂跨组织数据范围 |
| 招投标与高层介入动作         | `Project` 主责组织 + 战略/审批归口 | 涉及例外审批时允许跨组织审批角色介入           |
| 合同登记与变更               | 商务归口组织 + `Project` 主责组织  | 商务行政负责文档完整性，项目归属负责业务归口   |

| 提成计算、发放、调整         | 财务归口组织 + 项目归属组织        | 财务负责生效口径，项目组织负责业务归属         |
| 平台治理域公共对象           | 不适用                             | 公共对象不直接承接组织范围授权                 |

### 5.9 关闭、作废与冲销类动作基线

第一阶段应把“关闭 / 作废 / 冲销”视为高敏感动作，不能归入普通编辑。

| 业务域     | 对象                   | 动作        | 触发条件                           | 授权主体              | 组织范围           | 字段敏感度 | 状态约束                                                                | 放行方式  | 审计要求                     | 备注                     |
| ---------- | ---------------------- | ----------- | ---------------------------------- | --------------------- | ------------------ | ---------- | ----------------------------------------------------------------------- | --------- | ---------------------------- | ------------------------ |
| 销售流程域 | `Project`              | 关闭丢单    | 商务收口失败、投标未中标或明确失单 | 销售负责人 / 管理层   | `Project` 主责组织 | 高         | `Project.stage in (assessment, scope-confirmation, commercial-closure)` | 审批      | 留痕关闭原因、时间、责任人   | 形成 `closed-lost`       |
| 销售流程域 | `Project`              | 终止关闭    | 已进入执行或验收后发生重大终止     | 管理层                | `Project` 主责组织 | 高         | `Project.stage in (execution, acceptance)`                              | 审批      | 留痕终止原因、审批结论       | 形成 `closed-terminated` |
| 合同资金域 | `ReceiptRecord`        | 作废 / 冲销 | 已登记错误或已确认后需撤回         | 财务人员 / 财务负责人 | 财务归口组织       | 高         | `ReceiptRecord.status in (待确认, 已确认)`                              | 审批/确认 | 留痕原记录、原因、处理链     | 不得直接删除             |
| 合同资金域 | `PaymentRecord`        | 作废        | 付款登记有误或需撤销               | 财务人员              | 财务归口组织       | 高         | `PaymentRecord.status in (已登记, 已确认)`                              | 确认      | 留痕作废原因、时间、处理人   | 已确认记录应保留冲回关系 |
| 合同资金域 | `InvoiceRecord`        | 关闭        | 发票流程完成或异常处理完结         | 财务人员              | 财务归口组织       | 中         | `InvoiceRecord.status in (已验票, 异常)`                                | 确认      | 留痕关闭结论、责任人、时间   | 异常关闭不得掩盖历史异常 |
| 提成治理域 | `CommissionPayout`     | 暂停 / 冲销 | 退款、坏账、违规、回款回退         | 管理层 / 财务人员     | 财务归口组织       | 高         | `CommissionPayout.status in (已批准, 已发放)`                           | 审批      | 留痕原因、关联调整单、时间   | 不得直接删除已发放记录   |
| 提成治理域 | `CommissionAdjustment` | 关闭        | 调整处理已完成或驳回后归档         | 管理层                | 财务归口组织       | 高         | `CommissionAdjustment.status in (已执行, 已驳回)`                       | 审批      | 留痕关闭原因、结论与归档时间 | 关闭不等于删除           |
| 横切审批域 | `ApprovalRecord`       | 关闭        | 审批终止、业务对象关闭或失效       | 发起人 / 业务管理员   | 不适用             | 高         | `ApprovalRecord.currentStatus in (draft, pending)`                      | 无        | 留痕终止原因、操作人、时间   | 必须同步关闭待办         |
| 横切审批域 | `ConfirmationRecord`   | 关闭 / 取消 | 确认不再有效或业务对象已终止       | 业务管理员 / 系统     | 不适用             | 高         | `ConfirmationRecord.currentStatus in (draft, pending)`                  | 无        | 留痕取消原因、关联对象、时间 | 不得静默消失             |

说明：

- 横切审批域中的公共对象不替代业务对象动作授权，它们只承接统一派单、通知和留痕。
- `ApprovalRecord`、`ConfirmationRecord`、`TodoItem`、`NotificationRecord` 的动作应作为公共能力接口统一抽象，而不是由每个业务模块重新定义。

### 5.13 前端按钮守卫与后端校验统一规则

第一阶段建议固定以下统一规则：

1. 按钮显示不直接等于可执行，前端按钮守卫只是体验优化。
2. 任何高敏感动作必须同时满足：平台入口可见、对象动作授权通过、状态前提满足。
3. 后端必须重复校验 `对象状态 + 责任角色 + 组织范围 + 放行方式`，不能信任前端显隐结果。
4. 对于“审批 / 确认 / 复核 / 多方确认”类动作，前端应展示明确动作按钮，而不是复用普通“保存”按钮。
5. 关闭、作废、冲销、重算、冻结、生效类动作必须单独建接口或明确命令型动作入口，避免与普通更新接口混用。

### 5.14 建议接口守卫分类

为便于后续接口设计，第一阶段建议把对象动作接口划分为三类：

| 接口类型       | 典型动作                                   | 守卫要求                                  | 说明                     |
| -------------- | ------------------------------------------ | ----------------------------------------- | ------------------------ |
| 普通更新接口   | 编辑草稿、补录说明、上传附件               | 平台入口 + 对象动作 + 状态前提            | 不允许修改高敏字段包     |
| 命令型动作接口 | 提交审批、确认生效、冻结、关闭、冲销、重算 | 平台入口 + 对象动作 + 状态前提 + 放行方式 | 必须独立留痕             |
| 系统派生接口   | 触发计算、生成快照、生成待办               | 系统内部校验 + 上游事实前提               | 由系统或受控后台流程触发 |

## 6. 与平台治理域的边界结论

基于当前已接受的 ADR 与平台治理域设计，第一阶段正式固定以下结论：

1. `platform:*` 权限只用于平台管理能力和平台接口访问边界。
2. `nav:*` 权限只用于导航可见性和页面入口体验。
3. 菜单可见不等于具体业务对象可操作。
4. 平台角色是权限集合容器，不直接等价于业务对象责任角色。
5. 组织上下文第一阶段不进入平台主授权链路，但可作为后续业务授权输入。
6. 审批通过/驳回是业务对象动作控制，不属于平台 RBAC 的直接表达范围。
