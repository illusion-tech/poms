## 1. 文档目标

本文档用于细化签约前统一工作区中的 `技术与成本`。

这个工作区的核心不是“多加一个成本录入页面”，而是把签约前的三类关键判断收成同一工作区：

- 技术可行性判断
- 范围边界确认
- 前期成本估算

同时，前期成本估算中应显式体现 `税务成本 / 税负影响`，而不是只在后续报价或财务评审时再隐含处理。

---

## 2. 工作区定位

### 2.1 面向用户

- 技术支持 / 售前
- 销售
- 财务
- 副总经理 / 流程负责人（以摘要视角进入）

### 2.2 使用时机

- 立项结论形成后，进入正式技术判断
- 项目准备投标或报价前，需要形成受控成本估算
- 销售需要理解售前结论并向后续协作角色传递时
- 财务和审批人需要理解成本基础时

### 2.3 工作区目标

- 让售前有正式、结构化的位置输出专业判断
- 让范围、风险、成本不再散在附件和备注里
- 让销售、财务和审批人能在同一上下文里理解售前结论
- 为投标、报价、经营核算提供可信输入

---

## 3. 工作区顶层结构

建议 `技术与成本` 工作区采用“左判断、右依据、下版本”的结构。

### 3.1 主区块建议

1. 需求与背景摘要
2. 技术可行性判断
3. 范围边界确认
4. 风险与保留意见
5. 前期成本清单
6. 税务成本说明
7. 成本估算版本记录

### 3.2 顶部固定摘要条

建议顶部固定显示：

- 当前技术判断状态
- 当前成本估算版本
- 当前税务成本 / 税负影响判断
- 当前是否存在阻断性风险
- 当前是否可进入投标 / 报价评审

---

## 4. 关键区块详细口径

### 4.1 需求与背景摘要

**作用**:

为售前判断提供统一输入上下文，减少从聊天、附件和口头转述中拼事实。

**建议包含**:

- 客户需求摘要
- 项目目标
- 关键交付场景
- 当前业务约束
- 外部依赖或第三方条件

**关键规则**:

- 这里不重复维护完整需求文档
- 只保留支撑技术判断和成本判断的必要摘要

### 4.2 技术可行性判断

**作用**:

形成“能不能做、做到什么程度”的正式结论。

**建议结论类型**:

- 可行
- 条件可行
- 暂不建议推进
- 不可行

**建议包含**:

- 判断结论
- 主要理由
- 当前前置条件
- 是否允许进入后续阶段

### 4.3 范围边界确认

**作用**:

把范围内、范围外、前提条件说清楚，并形成后续合同、投标、移交可复用的快照。

**建议包含 3 个结构区**:

1. 范围内事项
2. 范围外事项
3. 前提条件 / 依赖条件

**关键规则**:

- 不允许只写一段长备注替代结构化边界
- 每次重大变更应形成新快照，而不是覆盖历史

### 4.4 风险与保留意见

**作用**:

让售前的专业判断真实影响后续流程，而不是停留在“看过了”的备注。

**风险分类建议**:

- 技术能力风险
- 交付边界风险
- 集成依赖风险
- 时间排期风险
- 不确定成本风险

**保留意见建议表达**:

- 当前结论是否带保留意见
- 若带保留意见，需说明：
  - 风险点
  - 影响范围
  - 是否阻断后续阶段

### 4.5 前期成本清单

**作用**:

形成签约前成本估算的正式主能力。

**建议至少包含以下成本类别**:

1. 人力实施成本
2. 第三方采购 / 集成成本
3. 外包 / 专项支持成本
4. 差旅 / 部署 / 云资源 / 杂项费用
5. 税务成本 / 税负影响
6. 不确定成本项

**每个成本项建议字段**:

- 成本类型
- 简要说明
- 估算金额
- 估算依据
- 是否高不确定项

**关键规则**:

- 成本不能只有总金额，必须有成本项清单
- 税务成本不能只留在报价备注里，应在成本侧显式表达
- 高不确定项必须显式标记
- 成本项应支持后续与实际成本对比映射

### 4.6 税务成本说明

**作用**:

把签约前会影响毛利和报价判断的税务成本单独显式化，避免其被混入“其他费用”后失去判断价值。

**建议至少表达**:

- 当前税务口径或税负假设
- 税务成本是否已计入成本估算
- 若未计入，原因是什么
- 当前税务成本是否存在高不确定性
- 对报价 / 毛利判断的主要影响

**关键规则**:

- 税务成本不等于回款条件，它应作为经营判断中的单独输入被展示
- 若税务成本仍待财务确认，系统应显式标记为待确认项
- 税务成本变化应能进入成本估算版本差异说明

### 4.7 成本估算版本记录

**作用**:

应对投标多轮变化和前期判断演进，避免“当前这个数字是怎么来的”无法追溯。

**建议每个版本至少显示**:

- 版本号
- 更新时间
- 更新人
- 变更原因
- 总成本变化摘要
- 税务成本变化摘要

**关键规则**:

- 新版本不覆盖旧版本
- 报价评审引用时应明确引用哪个成本估算版本

## 5. 角色视角差异

### 5.1 技术支持 / 售前

默认重点：

- 技术可行性判断
- 范围边界确认
- 风险与保留意见
- 成本估算版本

### 5.2 销售

默认重点：

- 售前当前结论
- 是否已可进入投标 / 报价
- 高风险项
- 当前成本估算摘要
- 当前税务成本对报价的主要影响

### 5.3 财务

默认重点：

- 成本清单结构
- 税务成本口径
- 不确定成本项
- 后续毛利判断所需的成本依据

### 5.4 副总经理 / 流程负责人

默认重点：

- 技术结论摘要
- 保留意见摘要
- 成本总览与主要风险
- 税务成本对毛利的影响

---

## 6. 关键体验规则

### 6.1 售前结论必须结构化

系统不能继续依赖附件堆积和自由备注。

### 6.2 成本估算必须可解释

任何成本总额都应能拆回成本清单和估算依据。

### 6.3 税务成本必须显式可见

如果税务成本会影响报价、毛利和可推进性，它就不能继续藏在自由文本或后置审批备注里。

### 6.4 风险必须影响后续推进

如果存在阻断性风险，系统应在后续投标 / 报价环节中显式体现。

### 6.5 版本必须可回看

投标和报价过程中，多轮变化是常态，系统必须支持版本链，不可静默覆盖。

---

## 7. 与其他工作区的承接关系

`技术与成本` 必须至少承接并输出以下关系：

1. 输入来自：
   - 立项结论
   - 项目背景摘要
2. 输出给：
   - `招投标 / 商务竞标`
   - `报价与毛利评审`
   - `项目总览` 摘要区
   - 后续经营核算的估算成本基线
   - 税务成本对报价和毛利判断的摘要

如果技术判断和成本估算不能自然输出到这些工作区，第二阶段最关键的经营主线仍会断。
