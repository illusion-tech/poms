# ADR-001: 平台权限模型与授权边界

**状态**: 已接受 (Accepted)
**日期**: 2026-03-07

---

## 1. 背景 (Context)

`POMS` 当前已经具备基础认证与权限骨架：

- 后端存在 JWT、认证守卫与权限守卫
- 共享权限字典已集中定义在 `libs/shared/contracts/src/lib/shared-contracts.ts`
- 导航项已携带 `requiredPermissions`

但平台管理功能刚开始设计，关键边界尚未正式固定：

- 菜单可见权限与业务操作权限是否共用一套语义
- 角色是否只是权限集合，还是还要额外承载数据范围
- 前端是否需要重复做权限裁剪
- JWT 中是否直接携带完整角色/组织信息

如果这些问题不先定下来，后续用户管理、角色与权限、组织单元、导航菜单四个模块会在字段命名、接口权限、路由守卫和菜单展示上持续漂移。

---

## 2. 决策驱动因素 (Decision Drivers)

- **一致性 (Consistency)**: 权限定义、菜单显示、接口访问、页面守卫必须共用同一套权限语义
- **安全性 (Security)**: 菜单可见不等于接口可调用，后端必须保持授权主导权
- **可扩展性 (Extensibility)**: 第一版支持平台管理 RBAC，后续仍可扩展到组织或项目作用域
- **实现成本 (Implementation Cost)**: 当前代码已具备权限字典与守卫骨架，应尽量沿用而不是推倒重来
- **可维护性 (Maintainability)**: 避免把过多运行时上下文塞进 JWT，降低失效与同步复杂度

---

## 3. 候选方案 (Considered Options)

### 方案 1: 前后端分别维护权限规则
- **描述**: 前端自行判断菜单和页面权限，后端只做部分接口校验。
- **优点**: 前端看起来更灵活，局部页面开发更快。
- **缺点**: 极易出现权限漂移；菜单、页面和接口的授权结果不一致；长期维护成本高。

### 方案 2: 统一平台级 RBAC，后端为授权单一可信源
- **描述**: 权限字典统一管理；角色聚合权限；用户通过角色获得权限；后端负责接口授权与导航过滤；前端只做辅助路由守卫与体验优化。
- **优点**: 模型清晰、安全边界明确、最适合当前 `POMS` 的成熟度。
- **缺点**: 第一版暂不覆盖更复杂的数据范围权限，需要后续扩展。

### 方案 3: 一步到位引入平台权限 + 组织权限 + 数据范围权限
- **描述**: 在第一版同时建模平台级、组织级、项目级和数据级授权体系。
- **优点**: 理论上能力最完整。
- **缺点**: 复杂度明显过高，当前 `POMS` 还没有足够的业务实现基础支撑，会拖慢整个平台管理模块落地。

---

## 4. 决策 (Decision)

**最终选择：方案 2 - 统一平台级 RBAC，后端为授权单一可信源。**

**决策内容**:

- 第一版以平台级 RBAC 为主：`Permission -> Role -> User`
- `Permission` 继续作为共享契约中的单一事实源
- 菜单可见权限与业务操作权限同属一套权限字典，但语义分层：
  - `nav:*` 仅用于导航展示
  - `platform:*`、`project:*` 等用于真实业务操作
- 后端负责接口权限校验与导航过滤，前端不重复做业务权限裁剪
- 前端允许基于路由 `data.requiredPermissions` 做辅助守卫，防止用户直接输入 URL 越权访问
- JWT 中只保留轻量身份信息与必要权限上下文，不把完整角色对象、组织树等高变化数据直接写入 token

**决策理由**:

该方案最符合 `POMS` 当前阶段。现有权限字典、守卫、动态导航已经具备雏形，如果此时继续扩展为“后端主导、前端辅助”的统一平台级 RBAC，可以尽快把用户管理、角色与权限、组织单元、导航菜单四块能力接到同一条主链路上，同时保留未来向组织作用域授权演进的空间。

---

## 5. 后续影响 (Consequences)

- **共享契约**:
  - `Role` 需要从当前简单的 `id/name` 扩展为可支撑平台管理的正式模型，如 `roleKey`、`description`、`isActive`、`isSystemRole`、`permissionKeys`
  - 权限字典继续集中维护，避免前后端分别硬编码

- **后端**:
  - 所有平台管理接口都必须显式声明所需权限
  - 导航接口按用户权限过滤后返回，避免前端自行推断
  - 如果未来引入组织作用域授权，应在现有 RBAC 之上扩展，而不是推翻当前模型

- **前端**:
  - 需要补页面级权限守卫
  - 菜单渲染应逐步收敛到后端返回结果，减少本地静态 fallback
  - 页面按钮、操作入口可用用户权限做显隐或禁用，但不代替后端校验

- **文档**:
  - 后续用户管理、角色与权限、组织单元、导航菜单的详细设计，都应以本 ADR 为前提

- **测试**:
  - 需要覆盖“看得见菜单但访问无权限页面被拦截”“无菜单且接口拒绝访问”等关键授权场景
