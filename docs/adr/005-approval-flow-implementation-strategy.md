# ADR-005: 一期审批流实现策略

**状态**: 已接受 (Accepted)
**日期**: 2026-03-09

---

## 1. 背景 (Context)

`POMS` 第一阶段已经明确需要承接多类审批与确认场景，包括但不限于：

- 立项评估审批
- 报价与毛利评审
- 高层介入申请
- 项目移交确认
- 提成角色冻结与受控变更
- 提成发放审批
- 异常调整、冲销与违规处理

同时，需求说明已经明确：

- 第一阶段采用显式业务状态机 + 固定审批流
- 不以完整工作流引擎为前置条件
- 审批、审计、附件和通知属于第一阶段最小闭环能力

如果不尽快固定一期审批实现策略，后续详细设计会在两个方向之间摇摆：

- 先建设完整独立审批中心
- 先在各业务模块内嵌审批，再聚合统一待办

---

## 2. 决策驱动因素 (Decision Drivers)

- **交付节奏 (Delivery Speed)**: 一期需要优先形成主链路闭环，不能被通用审批平台建设拖慢
- **业务贴合度 (Business Fit)**: 当前审批流仍高度依赖具体业务状态机，不适合过早抽象成完全通用引擎
- **一致留痕 (Traceability)**: 审批记录、待办、通知和审计仍需统一沉淀
- **演进空间 (Evolution Path)**: 当前方案不能阻断未来演进到更独立的审批中心
- **实现复杂度 (Implementation Complexity)**: 需要避免“业务状态机 + 独立审批状态机 + 通用引擎配置”三套模型同时上马

---

## 3. 候选方案 (Considered Options)

### 方案 1: 一期直接建设独立审批中心
- **描述**: 所有审批流程先抽象为统一审批引擎或独立审批中心，各业务对象通过通用审批框架接入。
- **优点**: 理论上复用度高，长期能力完整。
- **缺点**: 当前阶段过重，需求边界尚未完全稳定，容易把业务问题提前抽象成平台问题。

### 方案 2: 模块内审批流 + 统一待办聚合
- **描述**: 审批状态机内嵌在各业务模块内，由公共能力统一沉淀待办、通知、审批记录和审计记录。
- **优点**: 与当前显式业务状态机最匹配，落地快，边界清晰，也便于后续统一升级。
- **缺点**: 初期不同模块的审批实现需要靠设计约束保持风格一致。

### 方案 3: 各模块各自实现审批，且不做统一待办
- **描述**: 每个模块自行维护审批流程、审批记录和待办，不做公共聚合。
- **优点**: 短期实现最直接。
- **缺点**: 很快会产生审批体验割裂、通知口径不一和审计链分散的问题。

---

## 4. 决策 (Decision)

**最终选择：方案 2 - 模块内审批流 + 统一待办聚合。**

**决策内容**:

- 一期审批流以内嵌在业务模块中的固定审批流为主
- 各业务对象通过显式状态机驱动审批，不先引入完整独立审批中心
- `common` 域提供统一的：
  - 待办聚合
  - 审批记录
  - 通知
  - 审计留痕
- 后续如审批流复用程度、配置需求或跨模块协作复杂度显著提升，再通过新 ADR 决定是否演进为更独立的审批中心或工作流平台

**决策理由**:

该方案与当前需求说明最一致。`POMS` 第一阶段的核心难点是把业务规则落成系统主链路，而不是先抽象一个通用审批平台。模块内审批流可以直接跟随业务对象和状态机落地，同时通过统一待办聚合和公共审批记录避免体验割裂，是当前阶段最稳妥、成本最低且后续仍可演进的方案。

---

## 5. 后续影响 (Consequences)

- **业务模块**:
  - 每个需要审批的对象都要显式定义审批状态、允许动作、审批人角色和回退规则
  - 审批结论必须与主对象状态迁移保持一致

- **公共能力**:
  - 需要提供统一待办模型
  - 需要提供统一审批记录模型
  - 需要把审批事件接入通知与审计

- **前端**:
  - 审批入口可分散在各模块页面中
  - 用户待办视图应统一聚合，不要求一期先做独立审批中心页面

- **后续文档**:
  - `workflow-and-approval-design.md` 需要以本 ADR 为前提展开
  - 业务权限矩阵设计需要与审批动作一并展开

- **测试**:
  - 需要覆盖“业务状态迁移与审批结论一致”“统一待办聚合正确”“通知与审计留痕完整”这三类关键场景
