# ADR-004: 合同资金域后端模块边界

**状态**: 已接受 (Accepted)
**日期**: 2026-03-09

---

## 1. 背景 (Context)

`POMS` 最新需求说明已将“合同资金域”明确为第一阶段正式范围，包含：

- 合同台账
- 应收计划与回款
- 应付与付款基础台账
- 发票管理
- 合同、资金、发票联动查询

当前 `POMS` 的逻辑架构和前端领域划分已经把“合同资金”作为独立一级业务域表达，但早期 HLD 的后端模块边界仍将 `contracts` 放在 `sales` 下。这会造成以下不一致：

- 逻辑域、前端域、后端域三者边界不统一
- 合同、回款、发票与提成之间的数据责任归属不清
- 后续接口命名、权限归口和聚合边界容易漂移

由于当前后端业务模块尚处于早期阶段，模块边界仍有较低调整成本，因此需要在详细设计前先固定一期后端的合同资金域边界。

---

## 2. 决策驱动因素 (Decision Drivers)

- **领域一致性 (Domain Consistency)**: 逻辑架构、前端信息架构与后端模块边界应尽量同构
- **数据责任清晰 (Data Ownership)**: 合同、回款、成本、发票需要有稳定的事实归属
- **提成依赖稳定 (Commission Dependency Stability)**: 提成治理强依赖合同资金数据，边界不清会直接影响重算与审计
- **演进成本 (Evolution Cost)**: 当前业务模块尚未大规模落地，尽早拆清边界成本最低
- **实现可控 (Implementation Practicality)**: 第一阶段仍需控制复杂度，避免过度分层

---

## 3. 候选方案 (Considered Options)

### 方案 A: 逻辑域独立，但一期实现层暂时并入 `sales`
- **描述**: 逻辑上保留“合同资金域”独立，但后端一期仍放在 `sales` 模块内部实现，例如 `sales/contracts`、`sales/receivables`、`sales/invoices`。
- **优点**: 一期实现更快，初始模块数量更少。
- **缺点**: 逻辑域、前端域、后端域不完全同构；后续如果合同资金域继续扩张，还需要再次拆分后端模块。

### 方案 B: 后端一期也独立拆出合同资金域
- **描述**: 后端与逻辑域保持同构，单独建立 `contract-finance` 模块，承载 `contracts`、`receivables`、`payables`、`invoices`。
- **优点**: 边界清晰、数据归属稳定、接口命名更一致，也更利于提成治理域依赖关系收敛。
- **缺点**: 一期模块初始化和公共依赖管理会比方案 A 略重。

---

## 4. 决策 (Decision)

**最终选择：方案 B - 后端一期独立拆出合同资金域。**

**决策内容**:

- 第一阶段后端模块边界与逻辑领域保持同构
- 单独建立 `contract-finance` 后端模块
- 该模块至少包括以下子域：
  - `contracts`
  - `receivables`
  - `payables`
  - `invoices`
- `sales` 模块不再直接持有合同资金事实源，仅负责销售流程与阶段推进
- `commission` 模块不直接维护合同、回款、发票、成本事实数据，而是消费 `sales` 与 `contract-finance` 的生效数据

**决策理由**:

在当前阶段，`POMS` 的后端业务模块还没有形成大量存量实现，及早按领域拆清后端边界，成本远低于后期二次拆分。需求说明又已经明确合同资金域是一阶段正式范围，并不是销售流程的附属字段。因此，选择方案 B 更有利于保持全局一致性，并为提成计算、重算、审计和联动查询提供稳定的数据边界。

---

## 5. 后续影响 (Consequences)

- **后端**:
  - 后端模块建议至少形成 `platform`、`sales`、`contract-finance`、`commission`、`common`
  - 合同资金相关接口命名与权限归属应收敛到 `contract-finance` 域

- **前端**:
  - “合同资金”菜单和页面域与后端域保持一致
  - 前端数据访问层应避免把合同资金接口继续归到销售域客户端中

- **提成治理**:
  - 提成计算、发放、重算需要显式依赖 `contract-finance` 域的生效数据
  - 重算触发条件可更清楚地挂到合同变更、回款确认、成本变化和发票状态变化上

- **文档**:
  - HLD、合同资金详细设计、提成结算设计都应以本 ADR 为前提

- **测试**:
  - 需要覆盖跨域协作场景，例如合同生效初始化应收计划、回款确认触发提成重算候选等
