# ADR-008: 当前用户资料输出契约

**状态**: 已接受 (Accepted)
**日期**: 2026-03-10

---

## 1. 背景 (Context)

平台治理域详细设计已经明确：

- 当前用户资料输出应作为前端初始化、顶部身份展示和基础会话上下文的正式输入
- 当前用户资料必须返回当前有效 `roleKey[]`
- 当前用户资料必须返回当前有效 `PermissionKey[]`
- 当前用户资料中的组织信息必须明确主责组织与附属组织关系，而不是让前端自行推断

但当前设计与契约之间仍存在未闭合点：

- `user-management-design.md` 要求 `orgUnits` 中至少应能表达 `membershipType`，用于区分 `primary / secondary`
- 当前共享契约中的 `UnitOrg` 仅包含 `id/name/code/description`
- 当前 `auth/profile` 的开发期实现仍返回 `roles=[]` 与 `orgUnits=[]`

如果不先固定当前用户资料输出契约，后续会同时影响：

- 用户管理设计评审
- 共享契约扩展方式
- 前端当前用户上下文与资料展示建模
- 认证与会话初始化流程

---

## 2. 决策驱动因素 (Decision Drivers)

- **关系语义清晰 (Relationship Clarity)**: 前端必须能直接识别主责组织与附属组织，不能靠位置或数量猜测
- **契约稳定性 (Contract Stability)**: 当前用户资料输出契约需要尽早稳定，避免认证、前端和用户管理三处反复改口径
- **实现复杂度 (Implementation Complexity)**: 方案不能让前端或后端维护两套含义接近但长期并存的组织输出模型
- **可扩展性 (Extensibility)**: 当前设计应能平滑支持后续组织展示或附属组织扩展
- **最小暴露原则 (Minimal Exposure)**: 当前用户资料只应暴露轻量视图，不应直接暴露完整组织树或关系实体原文

---

## 3. 候选方案 (Considered Options)

### 方案 A: 扩展 `UnitOrg`，继续使用 `orgUnits[]`
- **描述**: 保留 `SanitizedUserWithOrgUnits.orgUnits` 结构，但为组织轻量视图增加 `membershipType` 字段，必要时后续再补少量展示字段。
- **优点**: 延续当前命名和返回结构，对现有接口与前端心智模型改动较小；可以在一个数组内完整表达主责与附属关系。
- **缺点**: `UnitOrg` 语义从“纯组织轻量视图”升级为“带用户关系语义的组织轻量视图”，命名会略显泛化。

### 方案 B: 拆成 `primaryOrgUnit + secondaryOrgUnits`
- **描述**: 不扩展 `UnitOrg`，而是把当前用户资料输出拆为一个主责组织字段和一个附属组织数组字段。
- **优点**: 主责与附属关系语义最直接，不需要通过字段值区分。
- **缺点**: 契约结构更分裂，未来如果增加更多关系类型会继续膨胀；前端也需要分别处理两个字段。

### 方案 C: 保持当前 `UnitOrg[]` 不变，由前端推断主责/附属
- **描述**: 不调整共享契约，继续返回纯 `UnitOrg[]`，通过约定顺序或附加接口由前端区分主责和附属。
- **优点**: 短期改动最少。
- **缺点**: 语义不稳定，违反“前端不得猜测关系事实”的设计原则，长期风险最高。

---

## 4. 决策 (Decision)

**最终选择：保留 `orgUnits[]` 的单数组结构，但不直接扩展通用 `UnitOrg`，而是新增用户上下文专用轻量类型。**

**决策内容**:

- `SanitizedUserWithOrgUnits` 继续保留 `orgUnits[]` 结构
- `orgUnits[]` 的元素类型不再直接复用通用 `UnitOrg`
- 新增用户上下文专用轻量类型，建议命名为 `UserOrgUnitSummary`
- `UserOrgUnitSummary` 至少包含：
	- `id`
	- `name`
	- `code`
	- `description`
	- `membershipType`
- 第一阶段 `membershipType` 只允许：
	- `primary`
	- `secondary`
- 当前用户资料接口必须基于真实关系实体聚合返回有效角色、有效权限和带关系语义的组织轻量视图
- 开发期 `roles=[]`、`orgUnits=[]` 只能作为临时占位实现，不再作为正式接口口径

**决策理由**:

该方案同时满足“前端不猜测关系事实”“结构对消费层友好”“关系语义显式化”和“避免污染通用组织轻量视图”四个目标。

与拆成 `primaryOrgUnit + secondaryOrgUnits` 相比，单数组结构对前端展示和遍历更自然，也更适合后续扩展更多关系类型。

与直接扩展 `UnitOrg` 相比，新增 `UserOrgUnitSummary` 能保持 `UnitOrg` 继续作为通用组织轻量视图使用，避免把用户关系语义混入所有组织展示场景。

---

## 5. 后续影响 (Consequences)

- **共享契约**:
	- 需要新增 `UserOrgUnitSummary` 或等价 schema
	- `SanitizedUserWithOrgUnits.orgUnits` 应切换到该专用类型数组
	- `UnitOrg` 保持通用组织轻量视图定位，不直接承载用户关系语义

- **用户管理设计**:
	- 需要把“若当前共享契约中的 `UnitOrg` 尚不足以表达该口径，应补齐契约扩展任务”升级为明确结论
	- 当前用户资料输出的正式结构以本 ADR 为准

- **组织单元设计**:
	- 可继续保留 `UnitOrg` 作为纯组织轻量视图
	- 用户上下文场景下的组织输出应显式使用专用类型，而不是复用纯组织视图

- **后端实现**:
	- `auth/profile` 必须从开发期占位实现升级为基于真实关系的聚合输出

- **前端实现**:
	- 当前用户上下文、顶部资料和用户详情页应统一消费 `orgUnits[]`
	- 前端不得通过顺序或数量推断主责/附属组织

- **后续文档**:
	- 该 ADR 的决策结果必须回写到用户管理设计、评审清单和进度跟踪文档
