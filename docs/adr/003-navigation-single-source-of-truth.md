# ADR-003: 导航菜单单一可信源策略

**状态**: 已接受 (Accepted)
**日期**: 2026-03-07

---

## 1. 背景 (Context)

`POMS` 当前已经有一套后端定义的内置导航树，并支持按权限过滤后返回给前端。但前端路由仍存在较强的 Demo 色彩，且本地菜单仍保留静态 fallback，这带来了几个明显风险：

- 后端导航链接与前端真实路由不完全一致
- 菜单可能显示了，但页面实际上不存在或不可访问
- 菜单权限、页面守卫和接口权限之间可能逐渐漂移
- 后续如果做导航配置管理，前后端容易同时维护两份菜单事实源

在平台管理功能启动前，需要先明确导航由谁主导、前端是否保留本地兜底、以及信息架构如何统一。

---

## 2. 决策驱动因素 (Decision Drivers)

- **单一事实源 (Single Source of Truth)**: 菜单结构和权限规则需要有唯一可信来源
- **一致性 (Consistency)**: 路由、导航、权限和页面实现要能相互对齐
- **安全性 (Security)**: 菜单显示不代表接口可访问，但菜单本身也应按权限正确过滤
- **可运营性 (Operability)**: 后续如果引入导航管理页面，需要当前设计可平滑过渡
- **迁移成本 (Migration Cost)**: 需要兼顾当前已存在的静态前端路由和样例页面

---

## 3. 候选方案 (Considered Options)

### 方案 1: 前端静态菜单为主，后端导航只做参考
- **描述**: 菜单结构主要由前端本地维护，后端导航接口可有可无。
- **优点**: 前端改动看起来更直接。
- **缺点**: 权限与菜单极易漂移，后续无法自然演进到后台可配置导航。

### 方案 2: 后端导航为单一可信源，前端忠实渲染
- **描述**: 菜单结构、权限过滤、层级和排序由后端主导；前端按返回结果渲染，并通过路由守卫做辅助防护。
- **优点**: 权限边界清晰，可自然过渡到导航管理能力。
- **缺点**: 需要先统一路由信息架构，并逐步清理现有前端 fallback。

### 方案 3: 前后端双轨长期并存
- **描述**: 后端和前端都各自维护菜单，并通过约定尽量保持同步。
- **优点**: 短期内迁移阻力较小。
- **缺点**: 长期一定产生偏差，是最难维护的方案。

---

## 4. 决策 (Decision)

**最终选择：方案 2 - 后端导航为单一可信源，前端忠实渲染。**

**决策内容**:

- 后端返回的导航树是平台菜单结构的唯一可信来源
- 后端按用户权限过滤导航项，前端不重复进行业务权限过滤
- 前端仅负责：
  - 将后端导航映射为 UI 组件
  - 处理禁用、隐藏、展开、激活等展示行为
  - 通过页面级守卫阻止用户直接访问无权限页面
- 前端现有静态 fallback 菜单只作为迁移阶段的临时机制，不作为长期架构
- 平台管理路由统一收敛到 `/platform/users`、`/platform/roles`、`/platform/org-units`、`/platform/navigation`
- 后续若实现导航管理，应直接维护后端导航数据，而不是新增一套前端本地配置体系

**决策理由**:

导航天然是“权限 + 信息架构 + UI 入口”的交汇点，最怕出现多套事实源。对 `POMS` 来说，后端已经具备导航树与权限过滤雏形，因此最合理的方向是继续强化后端主导，让前端专注于渲染与交互，并同步完成路由收敛。这样既能降低权限漂移风险，也为将来做导航管理和审计打下基础。

---

## 5. 后续影响 (Consequences)

- **后端**:
  - 导航模型需要继续作为共享契约的一部分维护
  - 导航接口返回的数据应与前端真实路由保持一致
  - 后续导航管理模块应直接读写后端导航数据源

- **前端**:
  - 需要逐步移除长期静态 fallback 菜单
  - 需要补齐与后端导航一致的真实路由
  - 路由守卫应支持读取 `requiredPermissions` 做辅助拦截

- **信息架构**:
  - 平台管理功能入口统一挂在 `/platform/*`
  - 页面命名、菜单 key、link 和面包屑命名需要同步收敛

- **实施节奏**:
  - 第一阶段先统一路由与后端导航定义
  - 第二阶段再考虑导航管理 CRUD、排序和审计能力

- **测试**:
  - 需要覆盖“不同权限用户看到不同菜单树”“直接访问未授权路由被拦截”“菜单 link 与真实路由一致”这三类关键场景
