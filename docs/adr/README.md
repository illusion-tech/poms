# POMS ADR

本目录用于记录 `POMS` 的架构决策记录（Architecture Decision Record, ADR）。

这些文档用于在功能设计和实现之前，先固定高影响、难回退的设计决策，避免后续在接口、数据模型、权限判断和前端信息架构上反复返工。

本目录与 `docs/design/` 的职责不同：

- `docs/adr/` 负责记录“为什么这样定”以及关键结论的演进历史
- `docs/design/` 负责记录“系统整体如何设计”和“各业务域如何落地”

因此，`ADR` 采用编号治理，而设计文档采用分类治理。

## ADR 适用范围

以下类型的内容优先进入 `ADR`：

- 高影响、难回退、会影响多个模块或多个文档的设计决策
- 需要明确记录“为什么选这个方案，而不是另一个方案”的议题
- 后续可能发生演进，且需要保留历史决策脉络的议题

以下类型的内容通常不应直接写成 `ADR`：

- 某个业务域内部的常规字段展开
- 页面级交互、按钮布局、一般性接口草案
- 评审检查单、评审摘要、进度跟踪

这些内容更适合进入 `docs/design/` 或 `docs/modules/`。

## 约定

- 文件命名使用 `NNN-短标题.md`
- 文件顺序号用于表达决策时间顺序与引用关系，不用于表达设计层级
- 状态建议使用：`提议中 (Proposed)`、`已接受 (Accepted)`、`已废弃 (Deprecated)`、`已被替代 (Superseded)`
- ADR 关注“为什么这样定”，不替代详细功能设计文档
- 若决策已变更，应新增后续 ADR，而不是静默改写旧结论

## 建议阅读方式

建议先阅读 `docs/design/poms-requirements-spec.md` 和 `docs/design/poms-hld.md`，再回看本目录下已接受的 ADR。

如果读者正在参与某一专项设计评审，则建议：

1. 先读对应设计主文档
2. 再回看关联 ADR
3. 最后结合评审清单确认该 ADR 是否已经被充分落到详细设计中

## 与设计文档的关系

- `ADR` 固定高层决策边界
- `docs/design/` 把 ADR 结论翻译成系统设计和业务域设计
- `docs/modules/` 在需要时继续下沉为更细粒度的实现层设计

如果某项内容只是“沿着已接受 ADR 继续细化”，通常不需要再新增 ADR，而应补充设计文档。
