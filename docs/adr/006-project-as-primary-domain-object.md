# ADR-006: 第一阶段主对象正式命名为 Project

**状态**: 已接受 (Accepted)
**日期**: 2026-03-09

---

## 1. 背景 (Context)

`POMS` 当前已经明确以下对象分层：

- `Lead`：客户接触后的最初线索对象
- 主推进对象：承接立项评估、技术确认、报价评审、高层介入、签约、项目移交、验收以及后续与合同资金、提成治理的联动
- `Contract`：签约结果与履约基础
- `ProjectHandover`：签约后进入正式执行前的独立里程碑

在前期讨论中，主推进对象一度以 `Opportunity / Project` 双命名表达，目的是保留“签约前偏商机、签约后偏正式项目”的语义弹性。

但随着需求说明和 HLD 持续细化，这个对象已经被明确要求承载：

- 销售推进主链路
- 责任归属与审批引用
- 项目移交与执行态切换
- 合同资金联动
- 验收记录关联
- 提成治理上游主对象语义

如果继续沿用 `Opportunity`，或长期保留 `Opportunity / Project` 双命名，会在详细设计、接口命名、权限矩阵、数据库模型和测试用例中带来持续语义漂移。

---

## 2. 决策驱动因素 (Decision Drivers)

- **领域贴合度 (Domain Fit)**: `POMS` 是项目型销售管理平台，而不是通用 CRM
- **模型稳定性 (Model Stability)**: 主对象命名将扩散到 API、实体、表名、权限和审计模型
- **概念分层清晰 (Concept Clarity)**: 已有 `Lead` 承担最初线索，不需要再用 `Opportunity` 承担第二层近似概念
- **生命周期覆盖能力 (Lifecycle Coverage)**: 主对象需要覆盖签约前推进阶段与签约后正式项目阶段
- **实施收敛性 (Implementation Convergence)**: 详细设计和代码实现应避免长期双命名

---

## 3. 候选方案 (Considered Options)

### 方案 A: 正式命名为 Opportunity
- **描述**: 主对象在第一阶段正式命名为 `Opportunity`，后续签约后继续沿用该名称。
- **优点**: 对签约前销售推进阶段语义较直观。
- **缺点**: 难以自然覆盖移交后、执行中、验收中、提成联动等后半生命周期语义。

### 方案 B: 长期保留 Opportunity / Project 双命名
- **描述**: 文档中继续使用 `Opportunity / Project` 过渡表达，不尽快收敛为单一正式命名。
- **优点**: 讨论期灵活。
- **缺点**: 不适合进入详细设计和代码实现，会导致产品、研发、测试口径持续不一致。

### 方案 C: 正式命名为 Project，以阶段语义表达商机阶段
- **描述**: 第一阶段统一以 `Project` 作为主对象正式命名；签约前的“商机”语义通过生命周期阶段、状态机或页面文案表达。
- **优点**: 能覆盖完整生命周期，术语最稳定，也更符合 `POMS` 的平台定位。
- **缺点**: 需要在文档和 UI 中额外说明“商机阶段项目”的业务语义。

---

## 4. 决策 (Decision)

**最终选择：方案 C - 第一阶段主对象正式命名为 `Project`。**

**决策内容**:

- 第一阶段主对象的正式名称统一为 `Project`
- `Lead` 继续作为最初线索对象独立存在
- “项目机会”或“商机阶段”仅作为 `Project` 的业务阶段语义存在，不再作为正式主对象命名
- 详细设计、接口命名、实体命名、表命名、权限矩阵、审计分类和测试用例均统一使用 `Project`
- `Project` 通过生命周期阶段与状态机表达签约前后的差异，例如：
  - 商机推进阶段
  - 待签约阶段
  - 移交中阶段
  - 正式执行阶段
  - 验收阶段
  - 关闭阶段

**决策理由**:

`POMS` 的主对象已经远超 CRM 意义上的“商机”，它需要贯穿销售推进、签约、移交、验收、合同资金联动和提成治理。既然系统已经有 `Lead` 承担最初线索层，主对象再命名为 `Opportunity` 的收益已经很小，反而会持续制造“商机”和“项目”之间的概念重叠。直接统一为 `Project`，再通过阶段语义表达签约前的商机状态，是更稳定、更贴近业务、也更利于实施收敛的选择。

---

## 5. 后续影响 (Consequences)

- **需求与设计文档**:
  - `poms-requirements-spec.md` 和 `poms-hld.md` 中的 `Opportunity / Project` 需要统一改为 `Project`
  - 相关待确认问题中不再保留主对象命名分歧

- **后端**:
  - 领域实体、DTO、服务命名、接口路径统一采用 `Project`
  - 项目阶段差异通过状态机和字段表达，而不是拆成两个主对象

- **前端**:
  - 页面、菜单、列表、详情统一使用“项目”作为正式对象名
  - 若业务上需要强调“商机阶段”，可通过标签、筛选项或页面分组表达

- **测试与审计**:
  - 测试用例、日志分类和审计归档统一使用 `Project`
  - 阶段差异应围绕状态和里程碑验证，而不是围绕对象命名验证
