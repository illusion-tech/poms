# ADR-002: 组织单元建模与授权关系

**状态**: 已接受 (Accepted)
**日期**: 2026-03-07

---

## 1. 背景 (Context)

`POMS` 计划开始设计组织单元能力，并与用户管理、角色与权限联动。当前共享契约中的组织单元模型较轻，仅包含基础字段，尚未回答几个关键问题：

- 组织单元是否只是用户归属信息，还是平台管理与数据归属的核心边界
- 组织单元是否需要树结构
- 用户与组织之间是单一归属，还是允许多组织关联
- 组织内权限是否要在第一版直接建成“用户-组织-角色”三元关系

如果这些问题不先定清楚，后续会同时影响：

- 用户详情与列表字段设计
- 角色分配方式
- 组织单元管理页面形态
- 未来项目、客户、合同等业务数据的归属策略

---

## 2. 决策驱动因素 (Decision Drivers)

- **业务表达能力 (Domain Fit)**: `POMS` 需要支撑平台管理和后续业务归属，组织单元不能只是展示字段
- **渐进演进 (Incremental Evolution)**: 第一版要先保证能落地，不能一次把复杂作用域权限全部做完
- **结构稳定性 (Structural Stability)**: 树结构一旦后补，数据迁移和前端交互改造成本会更高
- **操作体验 (Usability)**: 管理端需要清晰地展示层级、启停、排序和父子关系
- **扩展边界 (Future-Proofing)**: 未来可能会出现基于组织的数据访问范围或审批范围

---

## 3. 候选方案 (Considered Options)

### 方案 1: 组织单元只做平铺字典
- **描述**: 组织单元仅保留 `id/name/code/description`，用户只能挂一个组织。
- **优点**: 模型最简单，短期开发快。
- **缺点**: 难以表达上级/下级关系，不利于后续权限范围和业务归属扩展。

### 方案 2: 第一版即采用树结构，但授权仍以平台级 RBAC 为主
- **描述**: 组织单元从第一版开始建成树结构，支持 `parentId`、`children`、`isActive`、`displayOrder` 等基础字段；用户可关联一个或多个组织；组织作用域授权暂不直接进入主授权链路。
- **优点**: 先把高成本的数据结构建对，同时控制实现复杂度。
- **缺点**: 第一版的“组织与权限”关系不会一步到位，需要后续补强。

### 方案 3: 第一版直接引入完整的用户-组织-角色三元授权模型
- **描述**: 用户在不同组织下拥有不同角色，权限计算直接依赖组织上下文。
- **优点**: 模型最完整，适合复杂多组织权限体系。
- **缺点**: 对 `POMS` 当前阶段过重，会大幅增加认证、授权、页面交互和 API 设计复杂度。

---

## 4. 决策 (Decision)

**最终选择：方案 2 - 第一版采用树结构组织单元，但授权仍以平台级 RBAC 为主。**

**决策内容**:

- 组织单元在第一版就采用树结构
- 组织单元至少包含以下核心能力：
  - `id`
  - `name`
  - `code`
  - `description`
  - `parentId`
  - `isActive`
  - `displayOrder`
- 用户与组织单元允许建立关联，但第一版不强制把“组织上下文权限计算”接入主授权链路
- 第一版平台管理页面中的组织单元，主要承担：
  - 用户归属与展示
  - 数据归属的预留边界
  - 平台配置中的组织层级维护
- 若未来出现明确的组织范围授权需求，再通过新的 ADR 决定是否引入“用户-组织-角色”三元关系

**决策理由**:

树结构属于高成本基础设施，应该尽早定型；而组织作用域授权属于高复杂度能力，不适合在平台管理第一版同时推进。先把组织单元作为稳定的层级实体建好，再在后续基于真实业务需求扩展授权关系，是对 `POMS` 当前阶段最稳妥的路径。

---

## 5. 后续影响 (Consequences)

- **数据模型**:
  - 共享契约中的 `OrgUnit` 需要扩展为可表达层级和状态的正式模型
  - 需要考虑同级名称或编码唯一性规则

- **后端**:
  - 组织单元管理 API 应支持树形查询、创建、编辑、启停、调整父子关系或排序
  - 删除组织单元时需要明确策略：禁止删除有子节点或已被引用的节点

- **前端**:
  - 组织单元管理页建议以树表或树形列表为核心视图
  - 用户管理页中的组织信息不再只用字符串字段表示，而应与正式组织单元实体关联

- **权限**:
  - 第一版只要求“谁可以管理组织单元”
  - 暂不在第一版处理“某个用户只能管理某棵子树”的复杂作用域授权

- **文档**:
  - 后续组织单元详细设计应补充树结构操作规则、唯一性约束、删除限制和与用户关联的具体方式

- **测试**:
  - 需要覆盖父子层级、启停状态、排序调整、删除限制等核心场景
